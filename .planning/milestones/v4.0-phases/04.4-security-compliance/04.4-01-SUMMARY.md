## Decisions Made

- **Non-blocking insert:** `insertAuditEvent` uses try/catch logging pattern so a Supabase write failure never surfaces to the parent as an error. This is the most important design choice for the audit system.
- **Typed action_type:** Enforced at both the SQL level (CHECK constraint) and TypeScript level (union type). Any new action type must be added in both places.
- **Three-state consent_given:** NULL means parental consent was never solicited (pre-COPPA flow), TRUE means given, FALSE means denied or withdrawn. This allows the consent gate (Plan 05) to distinguish "not yet asked" from "actively refused".
- **Import pattern:** Used `import { supabase } from '@/lib/supabase'` singleton (matches chat.repo, categories.repo, expenses.repo) rather than `createClient()` from plan template — consistent with project convention.
- **Executor note:** SQL migration must be run manually in Supabase SQL Editor — it is not auto-applied by the app.

## Deviations from Plan
