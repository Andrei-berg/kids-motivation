# Phase 4.4: Security & Compliance - Context

**Gathered:** 2026-05-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Families can trust their data is protected and parents have full visibility into account changes. Delivers: account deletion with full data cascade, downloadable data export, COPPA/GDPR consent gate for children under 13, and a parent audit log of all significant actions.

Creating/editing child profiles, shop items, or reward rules are out of scope — this phase is data protection and accountability only.

</domain>

<decisions>
## Implementation Decisions

### Account Deletion UX
- Type-to-confirm friction: parent must type a confirmation word (DELETE or their email) before the action proceeds
- Show a data summary before confirming: counts of what will be removed (e.g., "2 children, 847 coin history entries, 12 weeks of grades, 34 transactions")
- Deletion modal offers "Download your data first" link/button — export is not required but prominently surfaced
- Lives in a red-bordered "Danger Zone" section at the bottom of an Account tab in Settings (GitHub/Vercel pattern)

### Data Export Format & Delivery
- Export package: ZIP containing both a `family-data.json` (complete structured data) and per-table CSV files
- Delivered as an instant browser download — no email, no server-side async job
- Scope: everything — children profiles, all grades, all coin transactions, wallet history, expenses, schedules, chat history, badges

### Audit Log
- Logged actions: shop purchase approvals/rejections, manual coin adjustments, badge awards, settings changes, account deletion request, data exports
- Displayed as a dedicated screen in Parent Center (new route alongside Dashboard, Analytics, Settings)
- Retention: keep all history with no time limit; filter by child (Adam / Alim / All)
- Each entry format: action description + child name + coin value if applicable + timestamp
  - Example: "Approved shop purchase: Minecraft (50💰) for Adam — May 17, 14:32"
  - Example: "Adjusted coins: +20💰 for Alim — May 15, 09:10"

### Consent Gate (COPPA/GDPR)
- Triggers during child registration when parent enters a birth year indicating age < 13
- Consent screen contents: brief list of data collected (grades, coins, behavior tracking) + required checkbox "I consent as the parent/guardian" + confirm button
- If consent is skipped or denied: child account is not activated — appears in Settings Family tab as "Pending consent" with a link to complete it
- Consent status shown per-child in Settings Family tab; parent can withdraw consent at any time, which routes to the account deletion flow for that child

### Claude's Discretion
- Exact wording of data deletion warning text and consent screen copy
- Pagination or infinite scroll for audit log (based on expected data volume)
- Specific visual treatment of "Pending consent" state in the Family tab
- How to handle the case where a parent enters no birth year (treat as adult / no consent gate)

</decisions>

<specifics>
## Specific Ideas

- Danger Zone pattern: red-bordered section, visually separated from normal Settings cards (matches GitHub, Vercel, Linear conventions)
- Audit log entry format modeled on clear, human-readable event descriptions — not raw database fields
- Export-before-delete nudge: surface the export link inside the deletion confirmation modal so parents don't lose data accidentally

</specifics>

## Existing Code Insights

### Reusable Assets
- `components/parent-center/screens/Settings.tsx` — existing Settings screen with tabs, Card/Btn/Pill/Toggle/Icon from `ui.tsx`. Account deletion "Danger Zone" and consent status go here (new Account tab or extend Family tab)
- `components/parent-center/ParentCenter.tsx` — route system via `route` state + `setRoute`. Adding an 'audit' route follows the same pattern as Dashboard/Analytics
- `components/parent-center/ui.tsx` — Card, Btn, Pill, Toggle, Icon, Tabs — use these for audit log entries, consent screen, and deletion modal
- `components/parent-center/tokens.ts` — T.cardBorder, T.indigo, T.cyan, T.muted etc. — all UI must use these tokens for visual consistency

### Established Patterns
- Repositories (`lib/repositories/`) — 7 repos (children, grades, wallet, schedule, expenses, chat, categories). Account deletion must call all of them in sequence or a single cascading Supabase RPC
- Toast notifications via `notify()` in ParentCenter — use for export success, deletion errors
- `lib/i18n.tsx` + `useT()` hook — all new UI strings must be i18n-wrapped

### Integration Points
- ParentCenter route system: add `'audit'` to the `Route` type in `types.ts` and render a new `AuditScreen` component
- Child registration flow (wherever `getChildren()` + child creation lives) — consent gate logic hooks in here
- Settings screen — needs new "Account" tab (or extended Family tab) for Danger Zone + consent status display
- Audit log writes: every existing parent action (shop approval, coin adjust, badge award) needs an `insertAuditEvent()` call added post-action
