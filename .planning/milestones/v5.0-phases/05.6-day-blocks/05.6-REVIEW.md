## Summary

Re-review of the day-blocks feature after gap-closure plans 05.6-06/07/08, against diff
base `0d2845d8^`. Verification performed this pass: `npx tsc --noEmit` clean,
`tests/day-blocks.test.ts` 15/15 pass, full read of both day-fill forms, the award route,
migrations, repo, settings manager, and the verify script.

**Both prior Critical findings are verifiably closed:**

- **CR-01 (Room checklist missing from kid flag-off branch) — CLOSED.** The flag-off
  branch of `KidDayFillForm` now renders the Room `FillSection` with `roomBody`
  (photo-proof UI included) at `components/kid/KidDayFillForm.tsx:1080-1084`, before the
  activities/grades/exercises/sections/reading sections, restoring pre-5.6 parity.
- **CR-02 (client-rotatable idempotency key on custom_block awards) — CLOSED, with
  defense-in-depth.** The award route keys `custom_block` intents on the natural
  `` `${date}:${entry.block_id}` `` string (`app/api/wallet/award/route.ts:575`), which
  survives any entry-row UUID rotation; the entries read is now child- **and**
  family-scoped (`route.ts:550-555`); migration `05.6-02` drops the `day_block_entries`
  DELETE policy (no client role can delete entry rows at all), ties INSERT/UPDATE
  `child_id` to the row's `family_id` via WITH CHECK, adds a parent/extended membership
  guard to the SECURITY DEFINER seed function, and backfills old UUID-keyed
  `wallet_transactions.source_id` values to the natural key with a collision guard. The
  backfill's intra-statement collision risk is provably nil given the
  `(child_id, date, block_id)` uniqueness on entries and the
  `(child_id, source_type, source_id)` uniqueness on transactions. The
  `verify-day-blocks-award.mjs` rotation probe exercises the exact exploit path.

No new Critical-severity issues found. The money path remains fully server-side
(service-role behind `requireFamilyMember`/`assertChildInFamily`; flag read server-side,
never from the request body; block-driven amounts clamped to ±100000). Six warnings and
seven info items follow — the most important being a migration re-run footgun that would
silently revert the 05.6-02 hardening, and a kid-form fallback branch that diverges from
the server when a flag-on family has zero visible blocks.

## Warnings

### WR-01: Re-running migration 05.6-01 silently reverts the 05.6-02 hardening

**File:** `supabase/migrations/05.6-01-day-blocks.sql:220-228, 238-267` (interaction with `05.6-02`)
**Issue:** `05.6-01` advertises itself as "Idempotent: safe to re-run" (header, line 6),
and this project applies SQL manually via the Supabase SQL editor, where re-runs are
routine. But re-running `05.6-01` after `05.6-02` has been applied would:
1. **Recreate the `day_block_entries_delete_own_family` DELETE policy** (lines 220-228)
   that `05.6-02` deliberately dropped as CR-02 defense-in-depth — the file even warns
   "do not recreate a DELETE policy on this table without re-opening CR-02".
2. **Replace the guarded `seed_default_day_blocks` with the original unguarded
   SECURITY DEFINER version** (lines 238-267), re-opening WR-07 (any authenticated user
   can seed/write another family's money-adjacent block config).
3. The backfill `SELECT seed_default_day_blocks(id) FROM families` (line 278) would then
   run unguarded again.
This is a one-command, silent security regression path.
**Fix:** Update `05.6-01` in-place so a re-run converges on the final state: remove the
`day_block_entries` DELETE policy creation (or follow it with `DROP POLICY IF EXISTS`),
and paste the guarded function body from `05.6-02`. At minimum, change the header of
`05.6-01` to "superseded by 05.6-02 — do NOT re-run after 05.6-02" and add the same
warning above the DELETE policy and the function.

### WR-02: Kid form flag-on with zero visible blocks falls back to the full legacy form — preview promises coins the server will not credit

**File:** `components/kid/KidDayFillForm.tsx:1074-1078` (contrast `components/DailyModal.tsx:1282-1288`)
**Issue:** The render branch is
`dayBlocksEnabled && visibleBlocks.length > 0 ? (assembled blocks) : (full legacy form)`.
For a flag-ON family where `assembleDayBlocks` returns an empty list for today (all
blocks deactivated, or every block's `day_types`/`days_of_week` filters exclude today —
e.g. every block set to school-only and the kid opens the form on a weekend), the kid
gets the complete legacy form: room checklist, grades, activities, reading — and
`coinsPreview` (line 302-359) advertises room/grade/book coins on the hero card and the
save button. But the server's flag-ON path (`award/route.ts:298-579`) credits **nothing**
for hidden blocks, so the kid is promised coins that never arrive. The parent
`DailyModal` handles the identical state differently (renders no block sections at all,
lines 1282-1288), so the two forms diverge in exactly the case D-06 says they must not.
The same divergence occurs transiently when the kid page's flag fetch fails
(`app/kid/day/page.tsx:125-130` catch → flag treated as off) while the server flag is on.
**Fix:** In the kid form, treat `dayBlocksEnabled === true` as authoritative:
`{dayBlocksEnabled ? visibleBlocks.map(...) : (legacy form)}` and render an explicit
"nothing to fill today" empty state when `visibleBlocks.length === 0`, matching
DailyModal. Zero out the legacy contributions in `coinsPreview` when flag-on.

### WR-03: `loadFeatureFlag` swallows query errors, silently mispricing awards on transient failures

**File:** `app/api/wallet/_lib.ts:87-94`
**Issue:** `loadFeatureFlag` destructures only `{ data }` and ignores `error`. A missing
row/column legitimately means flag-off (documented D-09 behavior), but a **transient
query error** (network blip, pool exhaustion) is indistinguishable and also yields
`false`. For a flag-ON family, the award route then runs the flag-OFF branch and credits
that day's sources at legacy prices — same `source_type`/`source_id` keys as the flag-on
branch (`day.id`, grade row ids, etc.) — so when the flag reads correctly on the next
save, the idempotency index **permanently locks in the wrong amounts** for that day (no
multiplier, no price override; also skips custom/exercise blocks for that save until
retried). A silent error on a branch decision that changes money amounts should fail
loudly, not quietly pick a branch.
**Fix:**
```ts
const { data, error } = await admin.from('families')
  .select('day_blocks_enabled').eq('id', familyId).maybeSingle()
if (error) throw error  // missing row (data === null) still → false
return data?.day_blocks_enabled ?? false
```

### WR-04: Clearing the vacation-multiplier input does not clear the stored multiplier

**File:** `components/settings/DayBlocksManager.tsx:107-120`
**Issue:** In `saveVacationMultiplier`, when the parent clears the field
(`trimmed === ''` → `nextValue = undefined`), the update payload is
`{ multipliers: { ...(block.multipliers ?? {}), ...({}) } }` — the spread of the
**existing** `multipliers` still carries the old `vacation` key, so the DB value is
rewritten unchanged. The input is uncontrolled (`defaultValue`, line 363), so after blur
it displays empty while the multiplier silently remains active in every award
computation. A parent who "removes" a ×2 vacation multiplier keeps paying ×2.
**Fix:** Build the map explicitly and delete the key when clearing:
```ts
const next = { ...(block.multipliers ?? {}) }
if (nextValue === undefined || !Number.isFinite(nextValue)) delete next.vacation
else next.vacation = nextValue
await updateDayBlock(block.id, { multipliers: next })
```

### WR-05: Award route dedupes per-child overrides differently from the client repo — inactive override hides a block server-side that the UI still shows

**File:** `app/api/wallet/award/route.ts:72-86, 319-325` (contrast `lib/repositories/day-blocks.repo.ts:22-61`)
**Issue:** The client repo (`getDayBlocks` with `activeOnly: true`, used by both forms)
filters `is_active` **in SQL first**, then dedupes — so an *inactive* per-child override
row is never fetched and the active family-default row wins (block visible). The award
route fetches **all** rows (no `is_active` filter, line 319-323), dedupes with
child-override-wins (line 81), and only then lets `assembleDayBlocks` drop inactive rows
— so the same inactive child override shadows the active family default and the block is
**invisible to the server** (never credited) while both forms render it and preview its
coins. Per-child overrides are reserved for Phase 5.8 and no UI creates them today, so
this is latent — but it will surface as an "earned in UI, never credited" bug the moment
5.8 ships, and the code comment claims the two implementations mirror each other.
**Fix:** Make `dedupeBlocksForChild` consider only active rows when picking the winner
(filter `rows.filter(r => r.is_active)` before deduping in the route), or add the same
`.eq('is_active', true)` to the route's query — either aligns it with repo semantics.

### WR-06: verify-day-blocks-award.mjs failure paths leave an active, earnable synthetic block in a real family's config

**File:** `scripts/verify-day-blocks-award.mjs:125-214`
**Issue:** The rotation probe inserts a real `day_blocks` row
(`name: 'verify-day-blocks rotation probe'`, `price: 1`, `is_active: true`) into an
arbitrary real family picked from prod (line 64-69). Several failure paths call
`process.exit(1)` **without** deleting it: entry insert failure (lines 147-150), first
natural-key award failure (lines 171-174), entry delete/re-insert failures (lines
183-197), and the non-23505 second-award path (lines 211-214) cleans only the
transaction. `process.exit` inside `try` bypasses the `catch` cleanup (it only handles
thrown exceptions). If that family has the flag on, the probe block appears in both
day-fill forms as an earnable 1-coin block until manually removed. The step-2 probe also
briefly inserts a visible `wallet_transactions` row into a real child's history.
**Fix:** Hoist cleanup into a `finally`-style helper: register `testBlockId`/
`testEntryId`/transaction keys as soon as created and run a single `cleanup()` before
every `process.exit(1)` (or `throw` instead of exiting so the existing `catch` cleanup
runs, then exit from `main().catch`). Consider `is_active: false` on the probe block so
it can never render even mid-run.

### IN-02: `fillMode` prop is declared and passed but never used

**File:** `components/kid/KidDayFillForm.tsx:37-46, 82-90`; `app/kid/day/page.tsx:308`
**Issue:** `KidDayFillFormProps.fillMode` (with the "mode 3 only" comment for grades at
line 128-129) is not destructured in the component — grades render for every school day
regardless of `child.kid_fill_mode`. Either the mode gating was intentionally dropped
(then the prop and comments are dead) or it was lost in the rewrite.
**Fix:** Remove the prop and stale comments, or restore the `fillMode === 3` gate on the
grades section — decide explicitly.

### IN-03: `childId` interpolated into PostgREST `.or()` filter strings

**File:** `app/api/wallet/award/route.ts:323`; `lib/repositories/day-blocks.repo.ts:28`
**Issue:** `` .or(`child_id.is.null,child_id.eq.${childId}`) `` splices `childId` into a
PostgREST filter expression. `children.id` is TEXT with no format constraint; an id
containing `,`/`(` would alter the filter. Exploitability is negligible today (the id
must pass `assertChildInFamily` first, family scoping is a separate `.eq`, and generated
ids match `[a-z0-9_]`), but it is a latent injection-shaped pattern.
**Fix:** Validate `childId` against `/^[a-z0-9][a-z0-9_-]{0,30}$/` at the route boundary
(and/or add a CHECK constraint on `children.id`).

### IN-04: Flag-on coin previews drift from server-credited amounts

**File:** `components/DailyModal.tsx:656-698`; `components/kid/KidDayFillForm.tsx:302-359`
**Issue:** Under flag-on, the previews ignore block visibility (e.g. `behaviorCoins`
counts even when the behavior block is hidden today, DailyModal:666-668), ignore
`resolveBlockPrice` overrides on built-ins, and ignore day-type multipliers for both
built-in and custom blocks — while the server applies all three. The WR-05 comment
("previews mirror exactly what /api/wallet/award credits") is no longer accurate flag-on.
Custom blocks with a negative price also preview as a deduction while the server skips
them (`base <= 0` guard, route.ts:562).
**Fix:** Reuse `resolveBlockPrice`/`resolveMultiplier` in the previews when
`dayBlocksEnabled`, or soften the comments to say the flag-on preview is approximate.

### IN-05: DayBlocksManager numeric-input edge cases and misleading delete error

**File:** `components/settings/DayBlocksManager.tsx:62, 95-105, 170-181`
**Issue:** (a) `Number(price)` yields `NaN` for junk and fractional values for "2.5" —
both bubble up as raw Postgres errors ("invalid input syntax for type integer") in the
error banner instead of being validated; the `nextPrice === block.price` short-circuit
also never matches `NaN`. (b) `handleDelete`'s catch shows the "built-in blocks can't be
deleted" message for **every** failure (RLS denial, network error), even though the
function already returns early for built-ins — the real error `e` is discarded.
**Fix:** Validate with `Number.isInteger(n)` before calling the repo; show `e.message`
for non-trigger errors and reserve `cannotDeleteBuiltin` for the trigger's
"legacy day block cannot be deleted" message.

### IN-06: Grade block visibility hardcodes school-day gating on top of configurable day_types

**File:** `components/kid/KidDayFillForm.tsx:952-957`; `components/DailyModal.tsx:1170-1175`
**Issue:** Flag-on, both forms additionally gate the assembled `grade` block on
`dayType === 'school'` / `!nonSchool`, while the server gates only on the block's own
`day_types` (default `{school}`). If a parent edits the grade block's `day_types` to
include weekends, the server would credit weekend grades but neither form ever shows the
grades UI — the block config is silently overridden by hardcoded logic.
**Fix:** Drop the extra `dayType === 'school'` condition flag-on and rely on the block's
`day_types` (the seeded default already restricts to school).

### IN-07: Dead locale key

**File:** `public/locales/en.json` / `public/locales/ru.json` (`settings.dayBlocksManager.addPlaceholder`)
**Issue:** `addPlaceholder` duplicates `namePlaceholder` and is referenced nowhere
(`DayBlocksManager` uses `namePlaceholder`/`pricePlaceholder` only).
**Fix:** Remove `addPlaceholder` from both locale files.
