# Phase 05.6 Plan 07: custom_block idempotency + family-scoped entries read Summary

**Closed CR-02 (coin-minting via day_block_entries UUID rotation) and WR-01 (cross-family entries read) in `app/api/wallet/award/route.ts` by switching the custom_block idempotency key to a natural `${date}:${entry.block_id}` string and adding a `family_id` filter to the entries query.**

## Accomplishments
- `custom_block` award intents now key idempotency off `${date}:${entry.block_id}` (the natural key), so deleting and re-inserting a `day_block_entries` row (permitted by family-wide entries RLS) no longer rotates the idempotency key and can no longer bypass the `(child_id, source_type, source_id)` unique index to mint duplicate coins.
- The `day_block_entries` select in the award route now filters `.eq('family_id', member.familyId)` in addition to the existing `child_id`/`date` filters, so a cross-family entry row inserted under the same `child_id` cannot be read into another family's award computation.
- No other source type's `sourceId`/`sourceType`, loop guards, `who_fills` check, or coin math was touched — diff is scoped to the two lines identified in the plan plus a documentation comment.
