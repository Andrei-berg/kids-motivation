# Phase 05.6 Plan 03: Award Route Block-Driven Coin Computation Summary

**`/api/wallet/award` now branches server-side on `families.day_blocks_enabled`: flag-off families get the byte-identical legacy 7-intent computation, flag-on families price built-in blocks via `resolveBlockPrice`/`resolveMultiplier` and credit custom blocks (including the 3 seeded "previously-free" defaults) through a new `custom_block` source_type — closing SC3 and the D-03 false-coin-promise gap with zero regression, verified by 4/4 green money verify scripts against prod.**

## Accomplishments

- Modified `app/api/wallet/award/route.ts` to read `loadFeatureFlag(admin, member.familyId)` right after the settings/streak prologue and branch the entire 5-intent computation (room/behavior, grade, sport, activity, book) on it. The flag-off branch is the original code, only re-indented — confirmed via `git diff --ignore-all-space` showing just 4 real line changes (the extended import + header comment), zero logic drift (D-09).
- Built the flag-on branch: resolves the server-side day type (mirrors `lib/services/streaks.service.ts`'s `family_calendar` + `vacation_periods` reads), loads and dedupes the family's `day_blocks` rows for the child, calls `assembleDayBlocks` to get the exact visible/earnable set for the date, then indexes visible built-ins by `legacy_key` in a `Map` — a source type earns nothing at all if its block is hidden today (D-08).
- Built-ins (room/behavior/book/exercise) now price via `resolveBlockPrice(block, settings) ?? <the matching wallet_settings key>` and scale by `resolveMultiplier(block, dayType)`; per-row types (grade/sport/activity) keep their existing per-row scale (`gradeCoins`/`coachCoins`/`activity.coins`) unmodified and only apply the multiplier — `resolveBlockPrice` correctly returns `null` for these so the flat-price path is never engaged for them.
- Added a genuinely new paid source, `exercise` (`sourceType: 'exercise'`, keyed per `home_exercises` row id) — fires only when flag-on and the block is visible, closing a previously-unpaid category (D-03).
- Added the `custom_block` source type: every visible custom block (legacy_key null) with a `done` `day_block_entries` row for the date is priced by its own `price` field × the day-type multiplier and credited generically — no special-casing for the 3 seeded "previously-free" defaults (home_help/weekend_homework/reading_extra), they pay out through this exact loop (D-03/WR-05 closed). `who_fills === 'parent'` blocks are rejected when `member.role === 'child'` (T-056-09, the server is the authoritative who_fills boundary since entries RLS is family-wide).
- Every block-driven coin amount (built-in and custom) passes through `clampBlockCoins` (`abs <= 100000`, rounded) before being pushed as an intent — defends against a crafted/malformed price or multiplier minting an unbounded award (T-056-07), on top of the existing parent-only RLS write protection on `day_blocks`.
- The streak bonus (block 6) stays completely outside/after the flag branch, identical in both paths — a streak is a bonus, not a day_block.
- Created `scripts/verify-day-blocks-award.mjs`: confirms `day_blocks`/`day_block_entries` tables and `families.day_blocks_enabled` exist, then proves a duplicate `(child_id, 'custom_block', source_id)` `wallet_transactions` insert is rejected with `23505` (T-056-08), using `coins_change: 0` so no balance is ever touched, and cleans up its synthetic row (re-run stays green — verified by running it twice).
- Ran the full money verify suite against prod: `verify-day-blocks-award.mjs`, `verify-award-idempotency.mjs`, `verify-award-reads.mjs`, `verify-wallet-rls.mjs` — all four exit 0 with no regression (SC3).
- `npx tsc --noEmit` and `npm run lint` are clean (no new errors/warnings in the touched files); `npm test` — 62 passed, 26 skipped (integration suites needing extra fixtures), same baseline as prior plans in this phase.
