# Phase 05.6 Plan 05: Day Blocks Settings Editor Summary

**Minimal per-family day-blocks management surface — list/toggle/price/vacation-multiplier for built-ins, full add/rename/price/who_fills/day_types/delete for custom blocks (incl. the 3 seeded "previously-free" blocks) — mounted as a new Parent Center → Settings → Schedule → "Day Blocks" sub-tab**

## Performance

- **Duration:** ~25 min
- **Tasks:** 2 completed
- **Files modified:** 4 (1 created, 3 modified)

## Accomplishments
- Built `DayBlocksManager.tsx`, a direct structural mirror of `RoomTasksManager.tsx`: loads all family blocks (`activeOnly: false` so disabled ones remain visible for re-enabling), reorder via up/down buttons, toggle active, add-custom form (icon picker + name + price), rename-on-click for custom rows, delete-with-confirm for custom rows.
- Built-in (legacy_key non-null) rows show a 🔒 "system block" badge, allow editing price and a single "vacation ×" multiplier input (writes into the `multipliers` JSON field, preserving other keys), and have no delete affordance in the UI — `deleteDayBlock` is still wrapped in try/catch surfacing `cannotDeleteBuiltin` as a second line of defense against the DB guard.
- Custom rows (legacy_key null) — including the 3 seeded "previously-free" blocks (home_help, weekend_homework, reading_extra) from Plan 01 — get full editing: rename, price, a `who_fills` select (kid/parent/both), and `day_types` checkboxes (school/weekend/vacation; empty = applies always), satisfying D-03's requirement that these seeded blocks be ordinary editable/deletable rows.
- Mounted the component as a new `blocks` sub-tab in `Settings.tsx`'s `ScheduleTab`, alongside the existing `room`/`activities`/`calendar` sub-tabs, using the exact same render-branch pattern.
- Added `settings.dayBlocksManager.*` (12 keys, including nested `whoFills.*` and `dayTypes.*` label groups) and `settings.tabs.blocks` to both `en.json` and `ru.json` — parity verified via the plan's node script (`match true`).

## Decisions Made
- `addDayBlock` is called without a `legacyKey` field — the plan's cited interface comment (`addDayBlock({familyId, legacyKey:null, ...})`) does not match the actual Plan 01 repo function signature, which has no `legacyKey` parameter at all and always inserts `legacy_key: null` for any row it creates. Verified by reading `lib/repositories/day-blocks.repo.ts` directly rather than trusting the plan's interface summary; `npx tsc --noEmit` confirms the call matches the real signature.
- Price and vacation-multiplier inputs are uncontrolled (`defaultValue` + `onBlur` commit) rather than fully controlled — avoids re-render/refetch churn per keystroke while still saving on blur, matching the existing rename-on-click/blur pattern in `RoomTasksManager`.
- Multiplier write merges into the existing `multipliers` object (`{ ...block.multipliers, vacation: value }`) rather than replacing it wholesale, so any future Phase 5.8 keys in that JSON aren't clobbered.

## Deviations from Plan

None - plan executed exactly as written. The `addDayBlock` call-shape adjustment (omitting `legacyKey`) is a correction to match the plan's own already-committed Plan 01 code, not a deviation from this plan's intent — the plan's action text describes calling `addDayBlock({..., legacyKey: null, ...})`, and the correct interpretation (confirmed against the source) is that `addDayBlock` never accepts a `legacyKey` argument and always creates custom (legacy_key null) rows, which is exactly the behavior the plan calls for.
