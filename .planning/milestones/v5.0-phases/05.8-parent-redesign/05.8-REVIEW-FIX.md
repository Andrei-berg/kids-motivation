## Fixed Issues

### CR-01: D-02 schedule-link picker is unreachable — dead code, feature never renders

**Files modified:** `components/settings/DayBlocksManager.tsx`
**Commit:** 1806bcf
**Applied fix:** Per operator guidance, wired the schedule-link picker into the D-03
per-child override view (`renderChildResolvedList`, the `byChild` layout) instead of
`renderFamilyDefaultList`, whose rows always have `child_id === null` and could never
satisfy the picker's `childMemberId` gate. Added `pickChildScheduleLink`/
`resyncChildScheduleLink` handlers that go through the existing `writeChildOverride`
copy-on-write path (same as the sibling price/who-fills handlers), reusing the existing
`openScheduleLinkFor` toggle state and `scheduleItemsByMember` data — so picking a
schedule item creates the per-child override on first use and updates it thereafter.

### CR-02: Weekly Summary "Tasks done" fact is a mislabeled, structurally-capped metric

**Files modified:** `lib/services/coins.service.ts`, `components/parent-center/ParentCenter.tsx`
**Commit:** 8830ee8
**Applied fix:** Per operator's chosen option (genuine weekly calc), extended
`getWeekScore` to also return `behaviorDays` (already computed internally, previously
discarded) alongside the existing `roomOkDays`/`filledDays`. `ParentCenter.tsx` now
aggregates `roomOkDays + behaviorDays` over `filledDays * 2` across all children for the
D-08 "This week → Tasks done" `StatusChip`, replacing the same-day
`todayDone`/hardcoded-`todayTotal=5` placeholder that structurally capped the rate at
40%. The pre-existing `todayDone`/`todayTotal`/`todayPct` fields are untouched — they
still drive the unrelated per-child "Today" cards (Dashboard/ChildProfile/Analytics
today-progress UI), which were out of scope for this finding.

### WR-01: `weeklyError` flag is effectively dead code

**Files modified:** `lib/repositories/wallet.repo.ts`, `components/parent-center/ParentCenter.tsx`
**Commit:** 2bba50b
**Applied fix:** Added `getTransactionsStrict` (same query as `getTransactions`, but
rethrows on a Supabase error instead of swallowing it and resolving `[]`). The D-08
week-scoped transaction fetch in `ParentCenter.tsx` now uses it, so `.catch(() => {
weeklyError = true; ... })` is reachable and the `weeklySummary.loadError` banner can
actually surface on a genuine fetch failure.

### WR-02: ChatPanel day-delta indicator colors carry banned neon values as decimal RGB

**Files modified:** `components/parent-center/screens/ChatPanel.tsx`
**Commit:** 842f077
**Applied fix:** Added a local `hexAlpha` helper (copied verbatim, matching the existing
`CalendarGrid.tsx` precedent for the private `design/atoms.tsx` helper) and replaced the
hardcoded `rgba(0,230,118,*)` / `rgba(255,107,107,*)` literals (decimal-RGB equivalents
of the banned `#00E676`/`#FF6B6B` neon hex) with `hexAlpha(T.success/T.danger, alpha)`
derivations.

### WR-03: New Tabs usages don't get the UI-SPEC's mandated indigo active-tab accent

**Files modified:** `components/settings/CalendarGrid.tsx`
**Commit:** 92ea1fb
**Applied fix:** Investigation found `DayBlocksManager.tsx` already imports `Tabs` only
from `@/components/design/atoms` (indigo-active) for both its usages — `CalendarGrid.tsx`
was the sole remaining consumer of `parent-center/ui.tsx`'s neutral-highlight `Tabs`.
Switched its child-filter `Tabs` import to `design/atoms.tsx` and passed `theme="ink"`,
matching the convention already established elsewhere in this phase.

### WR-04: UI-SPEC's D-05 "tap cell to add/edit vacation period" interaction was never implemented

**Files modified:** `components/settings/CalendarGrid.tsx`, `components/settings/PeriodsManager.tsx`, `components/parent-center/screens/Settings.tsx`
**Commit:** c81ffe7
**Applied fix:** Implemented the locked UI-SPEC interaction. `CalendarGrid` now accepts
an optional `onCellClick(dateStr, period)` callback, fired for cells with no sick dots
(sick-dot cells keep their existing popover). `PeriodsManager` accepts a consumable
`openRequest` prop: opens the edit form for a matched period, or the add form pre-filled
with the tapped date for none, then calls `onOpenRequestHandled` to clear it.
`Settings.tsx`'s `ScheduleTab` wires the two together and made `AccordionSection`
optionally controlled (backward-compatible, uncontrolled by default for every other
consumer) so the vacations accordion auto-expands to reveal the opened form.

### WR-05: No test coverage for `addDayBlock` `legacyKey` passthrough / `getDayBlocks` dedup

**Files modified:** `tests/day-blocks-repo.test.ts` (new)
**Commit:** 8522fdf
**Applied fix:** Added a `vi.mock('@/lib/supabase')`-backed suite (matching the
`tests/calendar-range.test.ts` pattern) asserting: `addDayBlock({legacyKey})` inserts
`legacy_key` correctly (and `null` when omitted); `getDayBlocks(familyId, {childId})`'s
merge returns exactly one row per shared `legacy_key` (the child-specific one), in both
row orderings; and `getDayBlocks` without a `childId` never dedupes.

### WR-06: Per-child override price field silently accepts non-numeric input

**Files modified:** `components/settings/DayBlocksManager.tsx`, `public/locales/en.json`, `public/locales/ru.json`
**Commit:** 566cb56
**Applied fix:** `handleChildPriceEdit` now rejects a parsed price that is non-null and
not `Number.isFinite` with an inline error (`settings.dayBlocksManager.invalidPrice`,
added to both locales) instead of silently writing `NaN` → `null` and clearing the
override's price. Matches the `Number.isFinite` convention already used by
`saveVacationMultiplier` in the same file.

### WR-07: Partial-failure in `writeChildOverride`'s create-then-carry-over path

**Files modified:** `lib/repositories/day-blocks.repo.ts`, `components/settings/DayBlocksManager.tsx`
**Commit:** 74ea547
**Applied fix:** Extended `addDayBlock` to accept `daysOfWeek`/`multipliers`/
`scheduleLink` directly (the DB columns already default safely — `addDayBlock` simply
never passed them through). `writeChildOverride`'s first-edit path now folds the family
default's carry-over fields plus the actual `fieldEdit` into a single `addDayBlock`
insert, eliminating the two-write window entirely rather than adding rollback/retry
logic around it.
