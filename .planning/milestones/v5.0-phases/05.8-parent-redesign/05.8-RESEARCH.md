# Phase 5.8: parent-redesign - Research

**Researched:** 2026-07-20
**Domain:** React/Next.js parent-facing UI redesign on an existing Supabase-backed
config model (day_blocks, family_calendar, vacation_periods) — no new external
libraries, no schema changes, no money-logic changes.
**Confidence:** HIGH (all findings verified by direct code/schema reads against
this repo; no external framework research was required — this phase is 100%
internal-codebase work)

## Summary

This phase closes out the ink-theme migration for Parent Center and upgrades two
functional-but-unstyled settings screens into full designed surfaces. Direct
reads of every file CONTEXT.md flagged as open/risky produced five concrete,
load-bearing findings that materially change how the planner should scope
tasks versus CONTEXT.md's framing:

1. **D-09 recolor scope is much narrower than CONTEXT.md implies.** Of the 8
   `components/parent-center/screens/*` + `ui.tsx`/`ParentGoals.tsx` files CONTEXT
   listed, six (`WalletsScreen`, `ChildProfile`, `ChildrenTasksShop`, `AuditScreen`,
   `ParentGoals`, and `Settings.tsx` itself incl. its inline `CoinsRulesTab`) **already
   import and use `T` tokens** from `components/parent-center/tokens.ts` — they are
   not on the old palette. The one screen with genuine hardcoded neon remnants is
   **`ChatPanel.tsx`** (`#00E676`, `#FFD93D`, `#FFC107`, `#F59E0B` — a `DARK_ACT`
   activity-color map). `ui.tsx` has one residual non-token literal: the `cyan`
   `Btn` variant's hover color `#4DDCFF`, used exactly once (`ChildProfile.tsx`'s
   Freeze button). The real, large recolor surface is the **7 `components/settings/*`
   sub-managers actually mounted** in `Settings.tsx`'s Schedule tab (`SubjectsManager`,
   `SectionsManager`, `CalendarSettingsManager`, `PeriodsManager`, `ActivitiesManager`,
   `RoomTasksManager`, `DayBlocksManager`) — none of these import `T` tokens at all.
2. **CONTEXT.md's "streak/coins-rules managers, family manager" are dead code, not
   mounted anywhere.** `components/settings/{FamilyManager,StreakSettings,
   CoinRulesEditor,NotificationSettings,CategoryManager,TaskManager,ScheduleEditor}.tsx`
   have zero import references anywhere in `app/` or `components/` — Settings.tsx
   implements coin-rules and family editing **inline** (already on `T` tokens).
   Recoloring these 7 orphaned files would be wasted effort; excluding them from
   scope should be confirmed at plan time (candidates for 5.11 deletion, not 5.8 recolor).
3. **`schedule_link` (D-02) only makes sense on per-child rows** — `schedule_items`
   are keyed by `family_members.id` (`child_member_id`), while `day_blocks.child_id`
   is `children.id`. A family-default block (`child_id IS NULL`) cannot meaningfully
   link to one child's schedule item. The simplest, award-route-safe implementation
   is to **denormalize on selection**: write `days_of_week` (0=Mon..6=Sun) computed
   from the picked `schedule_items.day_of_week` (1=Mon..7=Sun, so `dbDow = itemDow - 1`)
   directly into the `day_blocks` row alongside `schedule_link` (stored purely for
   UI display/re-sync). This requires **zero changes** to `assembleDayBlocks` or
   `/api/wallet/award` — both already read `days_of_week` for visibility — confirming
   CONTEXT.md's claim that the award route stays untouched, but only if this
   denormalize-on-write approach is used (not a live join at render time).
4. **No existing "family_members.id ↔ children.id" lookup helper exists** as a
   reusable function — every screen that needs both ids (D-02/D-03/D-06 all do)
   currently inlines its own `supabase.from('family_members').select('id, child_id,
   display_name').eq('family_id', fid).eq('role','child')` query (established,
   working precedent: `components/settings/PeriodsManager.tsx` lines 71-83). The
   planner should have the new screens reuse this exact inline pattern (or factor
   it into one shared helper) rather than `getFamilyChildren`/`useFamilyMembers`,
   both of which are wrong for this purpose (see Pitfalls).
5. **No "days in an arbitrary date range" repo function exists** for the D-05/D-06
   calendar grid — the closest precedent is `getWeekData`'s inline 7-day query
   pattern. A new function (e.g. `getDaysInRange(childId, start, end)`) mirroring
   that exact `.gte('date',...).lte('date',...)` shape is needed; day-type coloring
   itself is computed client-side via existing `getDayType()` + `getFamilyCalendar()`
   + `getVacationPeriods()` (no DB day-type column to query), while the D-06 sick
   overlay needs the new range query for `days.is_sick`.

**Primary recommendation:** Scope D-09 to the 7 mounted `components/settings/*`
managers + `ChatPanel.tsx`'s `DARK_ACT` map + `ui.tsx`'s stray `#4DDCFF`; treat
the other parent-center screens CONTEXT listed as already-compliant (spot-check
only). Implement D-02 as denormalize-on-pick (`days_of_week` copy), not a live
join. Add one shared child-id-resolution helper and one shared date-range days
query before building the D-02/D-03/D-05/D-06 screens on top of them.

## Architecture Patterns

### Day Constructor data flow (D-02/D-03/D-04)

```
DayBlocksManager (client)
   │
   ├─ family_members query (id, child_id, display_name; role='child')  ──► child picker (D-03 option a)
   │      pattern: PeriodsManager.tsx:71-83 (reuse verbatim)
   │
   ├─ getScheduleItems(familyId, childMemberId)  ──► schedule picker options (D-02)
   │      (schedule_items keyed by family_members.id, NOT children.id)
   │
   ├─ user picks a schedule_item
   │      │
   │      ▼
   │  updateDayBlock(blockId, {
   │    schedule_link: item.id,                         // display/re-sync only
   │    days_of_week: item.day_of_week.map(d => d - 1),  // 1..7 → 0..6 conversion
   │  })
   │
   └─ getDayBlocks(familyId, { childId })  ──► block list render
          (childId here is children.id — already resolves child-override-over-
           family-default via legacy_key matching; per-child CUSTOM schedule-

Kid day-fill / award route (UNCHANGED):
   assembleDayBlocks(blocks, dayType, date)  — filters on days_of_week (0=Mon..6=Sun)

```

### Anti-Patterns to Avoid
- **Reintroducing Tailwind's stock color palette** (`gray-*`, `amber-*`, `yellow-*`,
  `red-*`, `green-*`) in any recolored file — `tailwind.config.js` has no custom
  theme colors registered (`theme: { extend: {} }`), so these classes are
  disconnected from the ink token system by construction; the app's established
  convention for themed color is inline `style` objects referencing `T`.
- **Using `getFamilyChildren()` for the D-02/D-03/D-06 child pickers** — it
  filters to `user_id IS NULL` (unclaimed profiles only, per its onboarding-claim
  use case) and always returns `childId: null`. It will silently omit any child
  who has claimed a real account and can never supply the `children.id` these
  screens need.
- **A live-join implementation of `schedule_link`** (resolving the linked
  schedule item at render/award time) — unnecessary complexity; the denormalize-
  on-write approach keeps `/api/wallet/award` and `assembleDayBlocks` untouched,
  matching CONTEXT.md's explicit "not touched" scope boundary.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Day-of-week index conversion (schedule_items 1-7 vs day_blocks 0-6) | Ad-hoc per-screen math | One small shared converter (`scheduleDowToBlockDow = (d) => d - 1`) documented next to both type definitions | Off-by-one here silently breaks D-02 visibility for every family that uses it; a single tested helper prevents drift between the picker and any future consumer |
| Money-amount coloring in new UI (Weekly Summary card, ChatPanel recolor) | Custom gold hex literal | `Amount` / `LedgerRow` from `components/design/atoms.tsx` (`money=true`) | These atoms are the single place `ink.goldText` is wired up; `components/parent-center/tokens.ts`'s own `T` object has **no gold/goldText key** — hand-rolling a gold literal in a parent-center screen would introduce a second gold value out of sync with the atom's |
| Month/year date-range math | Custom loops incrementing `Date` day-by-day with manual month-length tables | `date-fns` (`startOfMonth`, `endOfMonth`, `eachDayOfInterval`, `getDay`) | Already a dependency; avoids reintroducing the DST/month-length bugs the codebase has specifically guarded against elsewhere (`localDateString()`, noon-anchored dates in `day-type.ts`/`day-blocks.ts`) |
| children.id ↔ family_members.id resolution | A new abstraction/hook | Copy the inline `supabase.from('family_members').select('id, child_id, display_name')` query (PeriodsManager.tsx:71-83) | Working, RLS-compatible precedent already in the codebase; a new hook is unnecessary indirection for what's used in 3-4 places |

**Key insight:** Every "don't hand-roll" item above already has a working
precedent somewhere in this codebase — the risk in this phase is not "no
solution exists" but "the obvious-looking existing helper (`getFamilyChildren`,
a fresh gold hex, a fresh Date loop) is the WRONG one to reach for."

## Common Pitfalls

### Pitfall 1: children.id vs family_members.id confusion
**What goes wrong:** A screen calls `getScheduleItems(familyId, childId)` using
`children.id` where `family_members.id` (`child_member_id`) is required, or vice
versa passes `family_members.id` into `getDayBlocks(familyId, { childId })`,
which expects `children.id`.
**Why it happens:** Both are opaque string ids from Supabase; nothing in the
type system distinguishes them (`ScheduleItem.child_member_id: string` vs
`DayBlock.child_id: string | null`).
**How to avoid:** Always resolve both ids together from one
`family_members` query (id + child_id) at the top of any D-02/D-03/D-06 screen,
never pass a single id across the two systems without the explicit mapping.
**Warning signs:** Schedule picker returns empty for a child who has schedule
items, or a per-child day-block override silently doesn't show for the right child.

### Pitfall 2: `addDayBlock`/`updateDayBlock` never pass `child_id` today
**What goes wrong:** D-03's per-child override UI calls the existing
`addDayBlock`/`updateDayBlock` functions expecting them to write a per-child row,
but every current call site in `DayBlocksManager.tsx` omits `childId` (defaults
to `null` = family-wide row), so a "per-child edit" silently mutates the
family-default block instead of creating an override.
**Why it happens:** The repo functions already accept `childId?: string | null`
(confirmed in `day-blocks.repo.ts`) — the gap is purely in the UI never passing it.
**How to avoid:** Any new "edit for this child" code path must explicitly pass
`childId` to `addDayBlock`, and `updateDayBlock` needs no signature change (it
takes an `id`, not `childId` — the *row itself* is already child-scoped by
whichever row was fetched/created) but the caller must ensure it's updating the
correct row (family-default vs. this child's override row), which may require
create-then-update (insert an override copy first) rather than blindly calling
`updateDayBlock` on the family-default row's id.
**Warning signs:** Editing a "per-child" price change affects every child's view.

### Pitfall 3: `getWeekScore` looks tempting for D-08 fact #1 but is the WRONG number
**What goes wrong:** `getWeekScore(childId, weekStart).total` is already computed
in `ParentCenter.tsx` for the sparkline and looks like a ready-made "coins this
week" figure — but it's re-derived from `days`/`subject_grades` with a **hardcoded**
`GRADE_COINS` map and flat `roomOkDays*3`/`behaviorDays*5` constants that predate
5.4 (streak-settings) and 5.6 (day_blocks configurable pricing). It does not
reflect a family's actual `wallet_settings` or `day_blocks.price` overrides.
**Why it happens:** It's the only "week aggregate" already sitting in
`ParentCenter.tsx`'s data-load effect, making it look reusable.
**How to avoid:** D-08 fact #1 explicitly says "SUM of positive
`wallet_transactions` this week" — use `getTransactions(undefined, <limit>,
weekStartISO)` (already supports a `since` param) and sum `coins_change > 0`
across children, not `getWeekScore.total`. This also matches CLAUDE.md's "do not
hardcode coin values in components" rule — `getWeekScore` is a hardcoded-math
component-facing function, not the money source of truth.
**Warning signs:** Weekly Summary's coin figure disagrees with the wallet balance deltas.

### Pitfall 4: `getTransactions` current call in `ParentCenter.tsx` won't cover a full week
**What goes wrong:** `ParentCenter.tsx` already calls
`getTransactions(undefined, 30)` (last 30 transactions total, no date filter) for
the activity feed — reusing this prop for D-08 fact #1 will undercount on any
family with >30 transactions/week (multiple kids, active family).
**How to avoid:** Make a dedicated call with `since = weekStart` and a higher/no
limit for the Weekly Summary aggregation, separate from the 30-row activity feed.
**Warning signs:** Weekly Summary total is suspiciously low or drops for active families.

### Pitfall 5: `days_of_week` conversion off-by-one
**What goes wrong:** `schedule_items.day_of_week` is `1=Mon..7=Sun`;
`day_blocks.days_of_week` / `assembleDayBlocks`'s internal `dayIndex` is
`0=Mon..6=Sun`. Copying the array verbatim silently shifts every block's visible
day by one (e.g. a Tuesday football link shows the block on Monday).
**How to avoid:** Convert with `itemDow - 1` before writing to `days_of_week` —
verified correct for both ends (Mon: 1→0, Sun: 7→6).
**Warning signs:** A schedule-linked block appears/disappears on the wrong day.

### Pitfall 6: Recoloring dead code
**What goes wrong:** Time spent recoloring
`components/settings/{FamilyManager,StreakSettings,CoinRulesEditor,
NotificationSettings,CategoryManager,TaskManager,ScheduleEditor}.tsx` — none are
imported anywhere in `app/` or `components/` (verified via repo-wide grep).
**How to avoid:** Confirm at plan time these are out of scope (likely 5.11
deletion candidates); do not schedule recolor tasks for them.

### Day-of-week conversion for D-02 schedule linking
```typescript
// schedule_items.day_of_week: 1=Mon..7=Sun (lib/models/schedule.types.ts)
// day_blocks.days_of_week / assembleDayBlocks dayIndex: 0=Mon..6=Sun (lib/day-blocks.ts)
function scheduleDowToBlockDow(days: number[]): number[] {
  return days.map(d => d - 1)
}

// On picking a schedule_item in the D-02 picker:
await updateDayBlock(block.id, {
  schedule_link: item.id,
  days_of_week: scheduleDowToBlockDow(item.day_of_week),
})
```

### New repo function needed for D-05/D-06 (mirrors getWeekData's existing shape)
```typescript
// Source: pattern lifted from lib/repositories/children.repo.ts getWeekData
// (lines 496-525) — generalize the date bound instead of a fixed week.
export async function getDaysInRange(childId: string, startDate: string, endDate: string) {
  const { data, error } = await supabase
    .from('days')
    .select('*')
    .eq('child_id', childId)
    .gte('date', startDate)
    .lte('date', endDate)
    .order('date')
  if (error) throw error
  return data ?? []
}
```

1. **Should ChatPanel.tsx be consolidated onto the shared `ChatThread` component,

   - What's unclear: CONTEXT.md's D-09 wording ("parent side of the theme-aware

   - Recommendation: Scope D-09's ChatPanel item as a pure recolor of the
     `DARK_ACT` map + any other hardcoded literals (matches the phase's stated
     "reskin, not rebuild" intent); treat full `ChatThread` consolidation as
     out of scope unless the user confirms otherwise at plan/discuss time.

2. **D-03 per-child override: does `updateDayBlock` need a signature change, or
   only new call sites?**
   - What we know: the repo function's `fields` parameter has no `child_id`
     field at all (by design — you don't re-parent an existing row via update).
   - What's unclear: whether "per-child override" means creating a NEW row
     (`addDayBlock` with `childId` set, copying the family-default's other
     fields) or some other mechanism.
   - Recommendation: creating a new override row via `addDayBlock({ ...familyDefaultFields, childId })`
     is the only mechanism consistent with the existing unique-index design
     (`day_blocks_family_child_legacy_key_override_idx`) — plan for "copy on
     write," not a magic re-scope of the existing row.

3. **Does the Weekly Summary card need a dedicated new repo function, or is
   inlining the three queries directly in `Analytics.tsx`/`ParentCenter.tsx`
   acceptable?**
   - What we know: `AnalyticsScreen` currently receives fully pre-computed

     no data-fetching itself.
   - What's unclear: whether the Weekly Summary's three facts should be computed
     in `ParentCenter.tsx`'s existing `loadAll()` effect and passed down as new
     props (consistent with the current architecture), or fetched independently
     inside `AnalyticsScreen`.
   - Recommendation: compute in `ParentCenter.tsx` alongside the existing
     `getWeekScore`/`getStreaks` calls (same `Promise.all` batch) and pass down
     as new props — matches the established data-flow pattern exactly.
