<objective>
Add the D-03 per-child override UI to DayBlocksManager, building BOTH candidate layouts behind a top-of-screen `Tabs` toggle (per the explicit "build both, decide at verification" user direction mirroring 5.7's D-11): (a) a child selector above the block list showing that child's resolved blocks with override rows tagged "Custom"; (b) an inline per-block "Override for…" expander listing all children on one screen. Both use copy-on-write against the real repo.

**Scope line (built-in blocks only, D-03 this phase):** per-child overrides work for BUILT-IN blocks — those whose family-default row has a non-null `legacy_key` (the 7 seeded grade/room/behavior/sport/activity/book/exercise blocks). Custom (family-added) blocks — family-default `legacy_key` NULL — do NOT get a working per-child override this phase, because dedupe requires a matching `legacy_key` and there is no dedup mechanism for null-legacy_key rows without a schema change (out of scope per CONTEXT.md's phase boundary). The override affordance is hidden on custom rows.

Purpose: The repo already resolves child-override-over-family-default (getDayBlocks with childId) BY MATCHING legacy_key — but the current `addDayBlock` hardcodes `legacy_key: null`, so a copy-on-write override would NEVER match its family default and would DUPLICATE the row (shown twice in settings AND double-listed as a fillable/awardable task in the kid day-fill form). This plan (1) teaches `addDayBlock` to carry the family default's `legacy_key`, and (2) builds both write UIs, gated to built-in blocks.
Output: A `legacyKey`-aware `addDayBlock`; two working override layouts gated by a layout toggle, both exercising real copy-on-write with no duplication, built-in-only. Layered on the Plan-05 recolored + schedule-linked DayBlocksManager.
</objective>
