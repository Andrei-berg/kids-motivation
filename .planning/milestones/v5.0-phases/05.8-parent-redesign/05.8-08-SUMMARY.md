# Phase 05.8 Plan 08: Day Constructor per-child override (D-03) Summary

**Built-in day blocks (grade/room/behavior/sport/activity/book/exercise) can now be overridden per child via two selectable UI layouts, both driven by a single copy-on-write write path that dedupes against the family default by `legacy_key` — no duplicate rows in settings or in the live kid day-fill form.**

- **Completed:** 2026-07-21T03:15:05Z
- **Tasks:** 2/2 completed
- **Files modified:** 2

## Accomplishments
- `addDayBlock` accepts an optional `legacyKey` param (was hardcoded `legacy_key: null` on every insert) — this was the root fix that makes per-child override rows dedupe instead of duplicate
- Candidate (a): "By child" layout — a child-selector `Tabs` (Family default + one tab per child) showing each child's `getDayBlocks(familyId, {childId})` resolved list, override rows tagged with a "Custom" `StatusChip`
- Candidate (b): "By block" layout — the existing family-default block list, with a new "Override for…" inline expander under each built-in row listing all children with Avatar + name + price + who-fills, `[Custom ✕]` chip on an active override
- Both candidates share one `writeChildOverride(childId, legacyKey, fieldEdit)` handler: create-once (with immediate follow-up `updateDayBlock` carrying `days_of_week`/`multipliers`/`schedule_link` from the family default), update-if-active, reactivate-if-inactive
- Override affordance strictly gated to built-in rows (family-default `legacy_key != null`); custom (family-added) blocks render read-only in the per-child view, no override control shown
- Remove-override uses `setDayBlockActive(false)` behind a `confirm()` dialog — never `deleteDayBlock`, which the 05.6-01 legacy-delete trigger rejects for `legacy_key`-carrying rows
