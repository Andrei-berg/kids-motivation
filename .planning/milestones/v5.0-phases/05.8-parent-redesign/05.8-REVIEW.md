## Critical Issues

### CR-01: D-02 schedule-link picker is unreachable — dead code, feature never renders

**File:** `components/settings/DayBlocksManager.tsx:88-100, 447-728, 665-720`
**Issue:**
The D-02 schedule-link picker UI (the `{childMemberId && (...)}` block at line 669) is gated on `childMemberId`, which is derived per-row as:
```tsx
const childMemberId = block.child_id
  ? children.find(c => c.id === block.child_id)?.memberId
  : undefined
```
`block` here comes exclusively from `renderFamilyDefaultList`'s iteration over the top-level `blocks` state (`{blocks.map((block, idx) => { ... })}`, line 448). `blocks` is populated **only** by:
```tsx
const data = await getDayBlocks(familyId, { activeOnly: false })   // line 92 — no childId passed
```
Per `lib/repositories/day-blocks.repo.ts`'s `getDayBlocks`, when `childId` is omitted the query applies `.is('child_id', null)` — i.e. it returns **only family-default rows**. Consequently every `block` rendered by `renderFamilyDefaultList` always has `block.child_id === null`, so `childMemberId` is **always `undefined`**, and the schedule-link picker (`🔗 Schedule link: …` trigger, dropdown, synced `StatusChip`, re-sync button) can never render in either the `byChild` or `byBlock` layout.

Neither of the two D-03 per-child override views wires the picker in either: `renderChildResolvedList` (candidate a) only renders a price input/static text, and the "Override for…" expander (candidate b) only renders price + who-fills fields — neither includes a schedule-link control. The two plans (05, which built the picker; 08, which built the per-child views) never connected the picker to a row that actually has a `child_id`.

This makes the entire D-02 requirement — "the block auto-appears only on its linked schedule item's weekdays," explicitly reserved from Phase 5.6 for this phase, and listed as Plan 05's own must-have truth ("A per-child block can be linked to one of the child's schedule_items via an in-row picker") — **completely non-functional**: no user can ever open the picker through any screen. The `05.8-09-SUMMARY.md` human-verify checkpoint records approval of the recolor bar, the D-03 layout decision, and the calendar composition, but does **not** record explicit confirmation of checklist item 2 ("DAY CONSTRUCTOR (D-02): … open the Schedule link picker …") — consistent with the operator being unable to find a per-child row to click on in the first place.

**Fix:** Wire a real per-child block into the picker's rendering path — e.g. render the schedule-link expander inside the D-03 per-child views (`renderChildResolvedList` for built-in rows, and/or inside the "Override for…" expander's per-child row) using the resolved override row's `id`/`schedule_link`, or add a dedicated per-child block list that actually loads `getDayBlocks(familyId, { childId })` rows with `child_id` intact into a renderable list. Add a regression test/manual-verify step that actually reaches and exercises the picker (not just greps for its presence in source).

### CR-02: Weekly Summary "Tasks done" fact is a mislabeled, structurally-capped metric (near-permanently red, not week-scoped)

**File:** `components/parent-center/ParentCenter.tsx:139-142, 192-194`
**Issue:** The new D-08 "This week → Tasks done" `StatusChip` is fed by:
```tsx
const todayDone = [roomOk, behaviorOk].filter(Boolean).length   // line 141 — max 2
const todayTotal = 5                                             // line 142 — hardcoded
...
const totalToday = parentChildren.reduce((s, c) => s + c.todayTotal, 0)
const doneToday   = parentChildren.reduce((s, c) => s + c.todayDone, 0)
const taskRate = totalToday > 0 ? Math.round((doneToday / totalToday) * 100) : 0
```
`todayTotal` is a hardcoded placeholder (5) while only 2 booleans (`room_ok`, `good_behavior`) are ever counted into `todayDone`. The family-wide rate is therefore mathematically capped at `2/5 = 40%` **no matter how well every child is doing**, and it is computed from **today's** two flags, not from the week. Fed into `completionTone` (thresholds 80/50), this guarantees the "Tasks done" `StatusChip` on the new "This week" card renders `danger` (red) in virtually every real-world case, and its value never reflects "this week" as the card's own heading and copy (`analytics.weeklySummary.tasksDone`) claim.

This is new-code exposure of a pre-existing placeholder (`todayTotal = 5` dates to `e3dcabde`, 2026-04-24, unrelated to 5.8) — but Plan 07 deliberately wired this exact broken metric into a new, prominently-labeled, always-visible card ("reuse whatever today/completion metric the per-child batch already produces," per `05.8-07-PLAN.md` Task 1). The result: every parent who opens Analytics will see a red "Tasks done" badge essentially regardless of actual family behavior, directly undermining the phase's own must-have truth ("Completion rate shows as a success/warning/danger StatusChip at 80/50 thresholds" — technically true syntactically, but the success/warning tiers are structurally unreachable).

**Fix:** Either (a) compute a genuinely week-scoped completion metric (e.g. aggregate `days`/`day_block_entries` rows across the week window, matching the "This week" framing), or (b) if a same-day proxy is intentionally kept for this phase, fix the underlying `todayTotal`/`todayDone` placeholder to count all of the day's actually-fillable items (not a hardcoded 5 against 2 real booleans) so the 80/50 thresholds are reachable, and adjust the card's copy to say "today" rather than "this week" if that's the true scope.

## Warnings

### WR-01: `weeklyError` flag is effectively dead code — the Weekly Summary card's error state can never be reached

**File:** `components/parent-center/ParentCenter.tsx:124`, `lib/repositories/wallet.repo.ts:889-911`
**Issue:** `getTransactions(undefined, 500, weekStartISO).catch(() => { weeklyError = true; return [] })` assumes `getTransactions` can reject. It cannot: `getTransactions` itself never throws — on a Supabase error it does `console.error(...); return []` and resolves normally (`lib/repositories/wallet.repo.ts:905-910`). The `.catch()` in `ParentCenter.tsx` is therefore unreachable in practice, so `weeklyError` never becomes `true`, and the dedicated `weeklySummary.loadError` error banner built in `Analytics.tsx`'s `WeeklySummaryCard` is unreachable UI. A genuine fetch failure silently degrades to `coinsThisWeek = 0`, which is indistinguishable from — and gets rendered as — the "empty" (first-week-of-use) state instead of the intended error state.
**Fix:** Have `getTransactions` (or a wrapper used only for this call) surface failures (e.g. return a sentinel/throw on `error`), or check the Supabase `error` object directly in `ParentCenter.tsx` instead of relying on `.catch()` after a function that already swallows its own errors.

### WR-02: ChatPanel day-delta indicator colors still carry the exact "banned" neon values, just as decimal RGB (missed by the remnant-sweep grep)

**File:** `components/parent-center/screens/ChatPanel.tsx:180, 298`
**Issue:** `rgba(0,230,118,.1)` / `rgba(0,230,118,.3)` and `rgba(255,107,107,.1)` / `rgba(255,107,107,.3)` remain hardcoded. `0,230,118` is the exact decimal-RGB equivalent of `#00E676` — the very neon-green hex Plan 04 and the Plan 09 "remnant sweep" gate were chartered to eliminate from this file. Because the sweep's grep only matches hex-string literals (`#00E676`, `#4DDCFF`, etc.), this decimal-RGB occurrence slipped through undetected, and the `05.8-04-SUMMARY.md` explicitly (and reasonably, in isolation) treats it as "out of scope" since it wasn't in the plan's locked table — but it is nonetheless a hardcoded non-token color left in a file whose explicit purpose this phase was to make token-driven.
**Fix:** Replace with `hexAlpha`-style token derivations (e.g. `T.success`/`T.danger` at the appropriate alpha) so the day-delta indicators are also token-driven, and extend the remnant-sweep grep to also match decimal-RGB equivalents of the banned hex values, not just hex-string literals.

### WR-05: No test coverage for the phase's highest-risk new logic — `addDayBlock` `legacyKey` passthrough and `getDayBlocks`'s dedup merge

**File:** `lib/repositories/day-blocks.repo.ts:18-62, 133-167`, `tests/day-blocks.test.ts`
**Issue:** Plan 08's own threat model (T-058-19) and objective narrative identify the `legacy_key`-matched dedup as *the* mechanism preventing a per-child override from being duplicated in both the settings screen and the live kid day-fill form (`getDayBlocks` is also read by `app/kid/(app)/day/page.tsx` and `DailyModal.tsx`) — i.e. a regression here would double-count/double-list an awardable task. Despite this being called out as the "root fix," there is no unit test for `addDayBlock`'s new `legacyKey` param, nor for `getDayBlocks`'s legacy-key-based merge/preference logic (which row wins when both a family default and a child override share a `legacy_key`, `activeOnly` interaction, etc.). `tests/day-blocks.test.ts` only covers the pre-existing pure functions (`assembleDayBlocks`, `resolveBlockPrice`, `resolveMultiplier`); no repo-layer or dedup test was added.
**Fix:** Add a `vi.mock('@/lib/supabase')`-backed unit test (matching the `tests/calendar-range.test.ts` pattern already established this phase) asserting: (a) `addDayBlock({legacyKey})` inserts `legacy_key: legacyKey`; (b) `getDayBlocks(familyId, {childId})`'s merge returns exactly one row per shared `legacy_key` (the child-specific one), regardless of row order.

### WR-06: Per-child override price field silently accepts non-numeric input and nulls the price

**File:** `components/settings/DayBlocksManager.tsx:410-416`
**Issue:**
```tsx
async function handleChildPriceEdit(childId: string, row: DayBlock, value: string) {
  if (!row.legacy_key) return
  const trimmed = value.trim()
  const nextPrice = trimmed === '' ? null : Number(trimmed)
  if (nextPrice === row.price) return
  await writeChildOverride(childId, row.legacy_key, { price: nextPrice })
}
```
If a parent types a non-numeric value (e.g. "abc"), `Number(trimmed)` is `NaN`. `NaN === row.price` is always `false` (NaN never equals anything, including itself), so the guard never short-circuits, and `{ price: NaN }` is sent to `writeChildOverride` → `updateDayBlock`/`addDayBlock`. Supabase-js JSON-serializes the request body; `JSON.stringify(NaN)` produces `null`, so the write silently succeeds and **clears** the override's price instead of surfacing a validation error to the parent.
**Fix:** Validate with `Number.isFinite(nextPrice)` before writing (matching the `WR-04`/`Number.isFinite` convention already used elsewhere in this codebase per the Plan 08 threat model's own note); show an inline error instead of silently nulling the price.

### WR-07: Partial-failure in `writeChildOverride`'s create-then-carry-over path can permanently drop `days_of_week`/`multipliers`/`schedule_link`

**File:** `components/settings/DayBlocksManager.tsx:372-408`
**Issue:** First-edit override creation is two sequential writes:
```tsx
const created = await addDayBlock({ ...familyDefaultFields, familyId, childId, legacyKey })
await updateDayBlock(created.id, {
  days_of_week: familyDefault.days_of_week,
  multipliers: familyDefault.multipliers,
  schedule_link: familyDefault.schedule_link,
  ...fieldEdit,
})
```
`addDayBlock` cannot persist `days_of_week`/`multipliers`/`schedule_link` (DB defaults `'{}'`/`'{}'`/`NULL` apply on insert), so the follow-up `updateDayBlock` is what actually carries those fields over. If that second call fails (network blip, RLS hiccup) after the first succeeds, the newly-created override row is left permanently active with empty `days_of_week`/`multipliers` and a `null` `schedule_link` — silently discarding a schedule-link or vacation multiplier the family default had. A later retry of the same edit finds this row via the `existing` branch (`else if (!existing.is_active) …` / `else { updateDayBlock(existing.id, fieldEdit) }`) and only patches `fieldEdit`, never re-attempting the carry-over — the loss is permanent until a parent notices and manually re-links/re-multiplies.
**Fix:** Wrap both writes so a failure in the second either rolls back (delete/deactivate the just-created row) or is retried with the full carry-over payload, not just `fieldEdit`, on the next edit attempt.
