<domain>
## Phase Boundary

Make the room checklist family-configurable end to end: rename/add/remove/reorder items with
nothing about the checklist living in code or fixed DB columns. First de-hardcoding phase of v5.0.

In scope: `room_tasks` + `room_checks` tables (+ RLS), dual-write from both day forms, award
fallback logic, a settings editor, seeding defaults for existing and new families.
Out of scope: per-item coin pricing and day-blocks assembly (phase 5.6), design retheme (5.3/5.7),
the withdrawal-approval defect (separate gap-closure plan), streak settings (5.4).
</domain>

<decisions>
## Implementation Decisions

### Current state (verified in code this session)
- Schema: `days` has 5 fixed columns `room_bed, room_floor, room_desk, room_closet, room_trash`;
  a DB trigger (`room_score_trigger` / `update_room_score()`) derives `days.room_ok` from them
  (≥3 of 5) — `room_ok` cannot be set directly (proven by 05.1-05 test work).
- Award: `/api/wallet/award` pays a FLAT `settings.coins_per_room_task` once per day when
  `day.room_ok` is true (sourceType 'room', sourceId = day PK). NOT per-item.
- Forms: `components/kid/KidDayFillForm.tsx` (ROOM_ITEMS array of 5, labelKey i18n) and
  `components/DailyModal.tsx` (5 useState + hardcoded array) both write the 5 columns via
  `children.repo.ts` upsert.

### Locked decisions
1. **New tables.** `room_tasks`: id, family_id, name (user-facing text, family edits freely),
   icon (emoji), sort_order, is_active, created_at. `room_checks`: child_id, date, task_id,
   done — one row per checked task per day (PK/unique on child_id+date+task_id). Both
   family-isolated with RLS matching existing family tables; room_checks written by clients
   (it is NOT a money table — day data, like `days` itself).
2. **Seeding.** Migration seeds the default 5 tasks (кровать/пол/стол/шкаф/мусор with current
   emoji) for every EXISTING family and on new-family creation (same path that seeds other
   defaults — find it; if none exists, seed lazily on first read). Legacy column ↔ default task
   mapping is stable via a `legacy_key` column (bed|floor|desk|closet|trash|null) on room_tasks.
3. **Dual-write, read-new-first.** Both forms render the checklist from `room_tasks` (active,
   ordered) and write `room_checks` rows; for tasks having `legacy_key` they ALSO write the
   legacy `days.room_*` columns (keeps trigger/`room_ok`, analytics, wallboard working).
   Custom tasks (no legacy_key) exist only in room_checks.
4. **Award unchanged in economics, new in source.** `/api/wallet/award` computes room
   completeness from `room_checks` (count done among active tasks) with FALLBACK to
   `day.room_ok` when the family has no room_checks rows for that day. Behavior for a default
   5-task family must be IDENTICAL to today: flat `coins_per_room_task` once when done_count ≥
   threshold. Threshold generalization for N≠5 active tasks: Claude's discretion (fixed default
   3 stored per family, or ceil(0.6·N) — pick one, document, keep 5→3 exact).
5. **Settings editor.** In Parent Center Settings (children/schedule area — planner picks the
   natural tab), a Room checklist manager: add (name+emoji), rename, toggle active, reorder,
   delete (custom tasks hard-delete; the 5 legacy-mapped tasks can only be deactivated, not
   deleted, while dual-write era lasts). Follow the existing manager-component patterns
   (`components/settings/*Manager.tsx`).
6. **Kid form** renders task names/icons from the table (no i18n labelKey for custom names —
   names are user data; the seeded 5 may keep localized names created per family language at
   seed time — Claude's discretion, simplest wins).
7. **Migrations** are additive only; applied via SUPABASE_DB_URL pg path per CLAUDE.md; a
   verify script or integration test proves seeding + RLS.
8. **Tests.** Extend the 05.1 integration suite: award-from-room_checks parity test (5-task
   family: 3/5 done → coins_per_room_task awarded once, idempotent; 2/5 → nothing) and
   fallback test (no room_checks rows → legacy room_ok path still awards).

### Claude's Discretion
- Table DDL details, index names, exact RLS policy shapes (mirror existing family-scoped policies).
- Repo layer placement (`children.repo.ts` vs a new `room.repo.ts`).
- Threshold generalization rule (see decision 4).
- Whether wallboard/analytics legacy readers need touching (they keep working via legacy columns —
  prefer NOT touching them this phase).
</decisions>
