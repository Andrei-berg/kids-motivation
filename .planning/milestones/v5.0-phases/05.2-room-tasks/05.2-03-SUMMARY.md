## Decisions Made
- Threshold `max(1, ceil(0.6 * activeCount))` per CONTEXT decision 4 — exact parity (3) for the default 5-task family, sensible scaling for custom task counts (1 task → 1, 2 → 2, 10 → 6).
- `roomComplete` additionally requires `activeCount > 0` on the checks path — a family that deactivated every task cannot be awarded from stale check rows.
- Teardown does not direct-delete `room_tasks`: the 05.2-01 legacy-delete guard raises on direct deletes of legacy-mapped rows even for the service role (trigger fires regardless of RLS). The families FK cascade inside `destroyTestFamily` removes them at `pg_trigger_depth >= 2`; the teardown test asserts zero remaining `room_tasks`/`room_checks`/`wallet_transactions`.

### Auto-fixed Issues

**1. [Rule 1 - Bug] Test teardown direct-deleted guard-protected room_tasks (would silently fail)**
- **Found during:** Task 2, while writing afterAll/teardown per the plan's instruction to "delete the family's room_tasks/room_checks rows in afterAll before destroyTestFamily"
- **Issue:** The 05.2-01 legacy-delete guard trigger blocks DIRECT deletes of legacy-mapped room_tasks (raises `legacy room task cannot be deleted`) — a `db.from('room_tasks').delete().eq('family_id', ...)` from the test would error (silently, since supabase-js returns rather than throws), leaving the plan's literal teardown instruction unfulfillable.
- **Fix:** Teardown deletes `room_checks` directly (no guard) and relies on the `families` FK ON DELETE CASCADE inside `destroyTestFamily` to remove `room_tasks` (cascade-nested delete runs at `pg_trigger_depth >= 2` and passes the guard — the exact semantics 05.2-01 fixed). The teardown test asserts `room_tasks` is empty for the family post-cleanup.
