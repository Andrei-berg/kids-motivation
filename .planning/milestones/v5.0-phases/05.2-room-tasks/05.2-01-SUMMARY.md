## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Delete-guard trigger blocked FK-cascade deletes, not just direct deletes**
- **Found during:** Task 2, while running `scripts/verify-room-tasks.mjs` against the live DB
- **Issue:** The trigger checked `pg_trigger_depth() = 0` to distinguish a "direct" delete from a cascade-nested one, on the assumption that a top-level statement runs at depth 0. In fact, `pg_trigger_depth()` returns 1 for the currently-executing BEFORE DELETE trigger itself (the trigger's own invocation counts as depth 1), so the guard fired on *every* delete — including a `DELETE FROM families` cascading via `ON DELETE CASCADE` down to `room_tasks`. This would have permanently blocked family and child deletion (including CLAUDE.md's COPPA account-deletion cascade requirement) for every family, since the backfill guarantees every family has a legacy task. Verified empirically via a debug trigger logging `pg_trigger_depth()`: direct delete = depth 1, cascade-nested delete = depth 2.
- **Fix:** Changed the condition to `pg_trigger_depth() <= 1` (blocks only depth-1, i.e. the direct/top-level delete) and updated the trigger's comment to document the semantics. Verified both directions live: a direct `DELETE FROM room_tasks WHERE id = <legacy>` still raises and is blocked; `DELETE FROM families WHERE id = <family>` (cascading to the same legacy task) now succeeds.
- **Files modified:** `supabase/migrations/05.2-01-room-tasks.sql` (function `room_tasks_block_legacy_delete_fn`)
- **Verification:** Re-applied migration to live DB (idempotent `CREATE OR REPLACE FUNCTION`); ran the direct-delete and cascade-delete probes manually (both correct); ran `scripts/verify-room-tasks.mjs` — all 5 checks pass, exit 0.
- **Committed in:** `25e1479` (Task 2 commit, since the bug was only observable once the migration was applied live and the delete guard was exercised)

Worked around in `verify-room-tasks.mjs` with a small `parsePgArray()` helper that handles both a real array and the literal-string form. Not a migration bug — purely a client-side parsing detail in the verify script.
