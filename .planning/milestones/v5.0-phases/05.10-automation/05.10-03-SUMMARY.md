# Phase 05.10 Plan 03: Scheduled Allowance (SC1) Summary

**Folded per-child weekly/monthly allowance crediting into the existing daily cron route (from Plan 02), reusing `creditAwards()`'s `(child_id, source_type, source_id)` idempotency mechanism with no new idempotency code, plus a new `isoWeekKey()` helper and the `vercel.json` schedule entry that actually wires the route up.**

## Accomplishments
- New `isoWeekKey(date)` in `utils/helpers.ts` computes the ISO-8601 week key (`YYYY-Www`, Monday-first, Thursday-shift algorithm) used as the weekly allowance period key — unit-tested for a mid-year date and a Jan 1 ISO week-year boundary (2027-01-01 correctly resolves to `2026-W53`).
- `app/api/cron/daily/route.ts` gained a fourth check: queries `children` for `allowance_period IS NOT NULL AND allowance_amount > 0`, computes weekly (anchor === today's ISO day-of-week) or monthly (day-of-month anchor, clamped to the month's last day so anchor 31 still fires in short months) eligibility, and calls the existing `creditAwards()` with `sourceType='allowance'`/`sourceId=periodKey` — zero new idempotency code, fully delegated to `creditAwards()`'s pre-check + unique-index backstop.
- `vercel.json` now schedules `/api/cron/daily` daily at `0 13 * * *` (13:00 UTC / 16:00 MSK) alongside the existing `/api/cron/missed-tasks` entry — exactly two crons, staying on the Vercel Hobby tier (D-10).
- Allowance credit success also pushes "Пособие зачислено: +N 🪙" to the child via the route's existing `pushItems`/`Promise.allSettled` batch — no separate send pass.

## Decisions Made
- Weekly `periodKey` uses `isoWeekKey(now)` — the raw cron-invocation `Date`, same source the route's pre-existing `todayDow` (used for both the section-reminder check and the new weekly-allowance anchor match) already derives from via `now.getDay()`. This keeps the allowance eligibility check and its period key internally consistent with each other and with Plan 02's established pattern, rather than introducing a second (family-local-shifted) day-of-week source that could disagree with `todayDow` near a UTC/MSK midnight boundary.
- Monthly day-of-month and days-in-month are derived by parsing the already-family-local `today` string (`localDateString(now)` output, `YYYY-MM-DD`) rather than calling local-time `Date` getters on `now` directly — this makes the monthly path's date math independent of the executing server's OS timezone setting (relevant since this worktree's test environment happens to be `Europe/Moscow`, but a cron runtime is not guaranteed to be).
