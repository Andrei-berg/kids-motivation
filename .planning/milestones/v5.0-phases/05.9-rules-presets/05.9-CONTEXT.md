# Phase 05.9: rules-presets - Context

**Gathered:** 2026-07-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Three independent de-hardcoding features, continuing the v5.0 "nothing gets
restyled while it is hardcoded" principle:

1. **Rule philosophy presets** — Classic / No-penalties / Bonuses-only, one-tap
   application of predefined `wallet_settings` values, offered during onboarding
   and in Parent Center Settings.
2. **`grade_scale` per family** — 5-point / 12-point / A-F, driving grade input
   and the grade→coins award mapping.
3. **Behavior as a configurable tag set** — replaces the single `good_behavior`
   boolean + flat `coins_per_good_behavior` price with a family-defined list of
   tags, each with its own (possibly negative) price.

No new capability beyond these three; this is HOW to implement what ROADMAP.md
already scoped for 5.9.

</domain>

<decisions>
## Implementation Decisions

### Rule presets

- **D-01:** Three presets exist as distinct UI choices, but only two underlying
  value sets: **Classic** = today's `SETTINGS_DEFAULTS` unchanged (grade 5/4
  reward, grade 3/2/1 penalize: -3/-5/-10; coach 5/4 reward, coach 3 neutral,
  coach 2/1 penalize: -3/-10). **No-penalties** and **Bonuses-only** are
  numerically identical to each other: same positive values as Classic, all
  negative values floored to 0 (grade 3/2/1 → 0, coach 2/1 → 0). They stay two
  separate cards/labels in the UI for framing/discoverability, but no
  flat-tier-collapsing logic needs to be built — do not implement a third
  distinct numeric scheme.
- **D-02:** Applying a preset touches grade coins (`coins_per_grade_1..5`),
  room/behavior/exercise (`coins_per_room_task`, `coins_per_good_behavior` or
  its tag-set replacement — see Behavior tags below, `coins_per_exercise`),
  coach ratings (`coins_per_coach_1..5`), and `coins_per_book`. Streak
  bonus/day fields (`streak_*`) and `base_exchange_rate` / P2P limits are
  **not** touched by presets — those remain separately-edited settings
  sections, untouched by this feature.
- **D-03:** Applying a preset in Settings shows a **preview diff** (old → new
  values, per field) before the parent confirms — not a silent overwrite, not
  a blind confirm dialog with no visible numbers.
- **D-04:** Onboarding's existing `StepCoinRules` (manual g5/g4/g3/room/sport
  number entry in `app/onboarding/v2/page.tsx`) is **replaced entirely** by the
  preset picker. Step 3 becomes 3 preset cards, tap one, done — no number
  fields shown during onboarding. Fine-tuning individual values happens later
  in Parent Center → Settings → Coin Rules (`CoinsRulesTab`, already exists
  and already has this editor). `completeOnboarding`'s `coinRules?` parameter
  shape will need to change from manual g5/g4/etc to a preset selection (or be
  removed in favor of always writing full preset values).

### grade_scale

- **D-05:** Coin-per-grade mapping is stored as **one JSON map in
  `wallet_settings`**, keyed to the family's active scale — not new fixed
  columns per scale. New fields: `grade_scale` (enum: `five_point` |
  `twelve_point` | `a_f`) and `grade_coin_map` (JSONB, e.g.
  `{"5":10,"4":5,"3":-3,"2":-5,"1":-10}` for five-point,
  `{"12":10,...,"1":-10}` for twelve-point, `{"A":10,...,"F":-10}` for a-f).
  Settings UI renders one row per possible value of the active scale — same
  editing pattern as today's fixed grade rows in `CoinsRulesTab`, just
  data-driven length/labels instead of 5 fixed columns. The existing
  `coins_per_grade_1..5` columns become the five-point default/migration
  source, not deleted outright (families need a working default before they
  ever touch this).
- **D-06:** `subject_grades.grade` widens to store the **literal value** the
  parent/child entered, in whatever scale is active — `"5"`, `"11"`, `"B+"`.
  This requires a type change (the column is currently `INTEGER NOT NULL CHECK
  (grade >= 2 AND grade <= 5)` per `supabase/supabase-schema-v2.sql` — the
  live/current CHECK state must be verified against prod during research, no
  later migration touches it) and dropping the numeric-only CHECK. The award
  route looks up `settings.grade_coin_map[grade]` directly — no numeric
  conversion in either direction, no lossy A→number mapping.
- **D-07:** Badges/streaks (`lib/services/badges.service.ts`,
  `lib/services/streaks.service.ts`) hardcode grade semantics that only hold
  for the 5-point scale — e.g. `checkFullWeekGrades` treats `grade <= 2` as a
  penalty grade, `checkExcellentWeek`-style logic treats `grade === 5` as
  excellent. **This is explicitly OUT of scope for 5.9** — these stay
  5-point-only assumptions. For families on 12-point or A-F scales,
  grade-driven badges may misfire or never trigger. This is a **documented
  known gap**, not a silent one — flag it clearly in the phase's
  SUMMARY/verification and candidate for a future phase (generalizing to
  "coins > 0 = good, coins < 0 = penalty" via `grade_coin_map` instead of
  hardcoded thresholds would be the natural fix, but is not this phase's job).
- **D-08:** A family can switch `grade_scale` at any time, including with
  existing grade history. Because `subject_grades.grade` stores the literal
  value (D-06) and award idempotency (per `(child_id, source_type, source_id)`)
  already prevents re-crediting past days, switching scale is purely a
  forward-looking change — old grades keep displaying their original literal
  value untouched, nothing needs migrating or reinterpreting. No lock
  mechanism needed.

### Behavior tag set

- **D-09:** Behavior tags mirror the **Room Tasks pattern** exactly two
  tables: `behavior_tags` (family-level definitions: id, family_id, name,
  icon, price, active, order — price can be positive or negative) +
  `behavior_marks` (per child_id + date + tag_id). New/existing families get a
  seeded default tag ("Good behavior", +5) matching today's single flag, so
  families who never touch this see no behavior change. `days.good_behavior`
  stays as a legacy fallback column during the transition, same dual-write /
  fallback shape `room_tasks`/`room_checks` established in Phase 5.2 (see
  `05.2-01-PLAN.md`, `05.2-03-PLAN.md` for the exact precedent: legacy-delete
  guard, threshold/parity fallback, dual-write on both day-fill forms).
- **D-10:** Tag price can be **positive or negative** — e.g. "Helped a
  sibling" +5, "Rude to a parent" -10. Not bonus-only.
- **D-11:** **Multiple tags can apply per child per day**; the day's behavior
  coin total is the **sum of all applied tags' prices** — same computation
  shape as `extra_activities`/`activity_logs` today (each done activity
  contributes its own value, summed), not a threshold model like Room Tasks.
- **D-12:** Behavior tags support a **child-propose / parent-approve** flow,
  not parent-only-direct like today's `good_behavior` flag:
  - Child proposes a tag from their own day-fill form (`KidDayFillForm`);
    `behavior_marks` gets a `status` column (`pending` | `approved` |
    `rejected`) plus `proposed_by`/`decided_by`/`decided_at`.
  - Coins only credit on parent approval — same shape as `reward_purchases`
    (request → approve/reject), reusing that established pattern rather than
    inventing a new one. Parent approval happens in the DailyModal / Parent
    Center review surface (exact screen is a planning/research decision, not
    locked here).
  - A pending tag is **visible to the child** as "pending" on their day (no
    coins reflected yet); the parent sees it in their approval queue —
    mirrors the shop purchase-request mental model already established in the
    app ("things a parent must confirm show a pending state, not silence").
  - This flow applies to child-initiated tags. A parent applying a tag
    directly (their own assessment, including penalty tags) is not gated by
    this same approval step — it takes effect immediately, consistent with
    today's parent-only `good_behavior` assessment never needing self-approval.

### Claude's Discretion

- Exact screen/component where a parent reviews pending child-proposed
  behavior tags (new dedicated queue vs. folded into the existing DailyModal
  behavior section) — left to planning.
- Exact i18n copy/labels for preset cards, scale picker, and tag manager UI.
- Whether `grade_coin_map` migration backfills existing families' custom
  `coins_per_grade_1..5` values into the five-point default map, or those
  columns are simply read as fallback until a family's `grade_coin_map` is
  populated — a migration-strategy detail for planning/research.

</decisions>

  — direct template for `behavior_tags`/`behavior_marks` repo + migration

- `app/onboarding/v2/page.tsx` `StepCoinRules`, `completeOnboarding` (in
  `lib/onboarding-api.ts`) — being replaced/adapted for the preset picker.

### Established Patterns
- Per-family configurable list + per-entry marks table + legacy dual-write
  fallback (Room Tasks, Phase 5.2) is the established de-hardcoding template
  this codebase already uses twice (room, day-blocks) — behavior tags is the
  third application of the same shape.
- Award computation is always server-side, idempotent per
  `(child_id, source_type, source_id)`, recomputed fresh from saved rows on
  each `/api/wallet/award` POST — this is why grade_scale switching mid-year
  is safe (D-08) and why behavior_marks approval crediting can reuse the same
  idempotency guarantee reward_purchases already relies on.

### Integration Points
- `app/api/wallet/award/route.ts` — `gradeCoins()` function and the grades
  loop (flag-off path) need to read `grade_coin_map` instead of the switch
  statement; the day-level behavior intent (both flag-off and flag-on
  `behaviorBlock` paths) needs to become a loop over approved
  `behavior_marks` for that day instead of a single `day.good_behavior` check.
- `lib/day-blocks.ts` `behavior: 'coins_per_good_behavior'` legacy-key mapping
  — the flag-on day-blocks path also reads this key; behavior tags need to
  stay consistent with (or explicitly supersede) this mapping so Phase 5.6's
  day-blocks feature flag branch doesn't regress.

<specifics>
## Specific Ideas

- User's stated philosophy for why "Classic" keeps real penalties (grade 3
  → -3, grade 2 → -5, grade 1 → -10): a child should learn that actions have
  negative consequences too, not only positive reinforcement. This directly
  shaped keeping Classic's existing punitive defaults unchanged rather than
  softening them, and justifies offering No-penalties/Bonuses-only as
  opt-in alternatives rather than replacing Classic's philosophy.

</specifics>
