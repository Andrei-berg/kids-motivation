# Phase 05.5 Plan 05: Region Preset Picker for PeriodsManager Summary

**PeriodsManager's inline single-year `PRESETS` quick-add block is replaced with a real region/term-mode preset picker backed by `lib/vacation-presets.ts` (listPresets/applyPreset), with a Replace-vs-Add-missing confirm dialog on re-apply and a visual badge distinguishing preset-sourced rows from manual ones**

## Performance

- **Duration:** ~20 min active
- **Tasks:** 1
- **Files modified:** 3 (1 component, 2 locale files)

## Accomplishments
- Removed the old hardcoded `const PRESETS = [...]` (4 fixed 2025–2026 dates) and its "Templates 2025–2026" quick-add grid.
- Added a preset picker: a `<select>` sourced from `listPresets()` (4 bundled presets: `ru-quarters`, `ru-moscow-trimesters`, `kz`, `by`) showing each preset's bundled label, plus an "Apply" button, plus a hint line marking the schedule as a template (typовой график — actual dates vary, editing expected).
- `handleApplyPreset()`: checks whether any existing period already carries the selected `preset_id`. If none — applies directly as `'add-missing'`. If some exist — reuses the existing `confirm()` convention (same pattern as `handleDelete`) to ask Replace vs Add-missing; OK → `applyPreset(familyId, id, 'replace')`, Cancel → `'add-missing'`.
- After a successful apply, calls `upsertFamilyCalendar(familyId, { region_preset: selectedPresetId })` so `family_calendar.region_preset` is kept in sync with the chosen preset (WARNING-1 resolution) — the Settings→Calendar region select (Plan 04) and this picker now read/write the same field.
- Reloads the period list via the existing `loadData(familyId)` and shows a transient status line with the inserted-row count (`{{count}} periods added`).
- List render: preset-sourced rows (`preset_id != null`) get a small amber pill badge (📅 + "Preset"/"Шаблон") next to the period name; manual rows are unmarked. Manual add/edit/delete controls are untouched — fully first-class (D-01).
- Added `settings.periodsManager.presetPicker.{title,hint,apply,confirmReplaceOrAdd,appliedCount}` and `settings.periodsManager.presetBadge` to both `en.json` and `ru.json`.

## Orchestrator Handoff Resolution (WARNING-1 id parity)

Verified `components/settings/CalendarSettingsManager.tsx`'s static `REGION_OPTIONS` values (`ru-quarters`, `ru-moscow-trimesters`, `kz`, `by`) match the `id` field of every file in `data/vacation-presets/` (confirmed via `grep '"id"' data/vacation-presets/*.json`) and thus match `listPresets()`'s output exactly. Both region controls (this plan's picker and Plan 04's select) read/write the same `family_calendar.region_preset` values — WARNING-1 sync holds. **`CalendarSettingsManager.tsx` was left untouched**: it is outside this plan's `files_modified` scope (`components/settings/PeriodsManager.tsx`, `public/locales/en.json`, `public/locales/ru.json` only), and no code change was needed since the ids already match — only documentation/verification was required. Wiring `CalendarSettingsManager`'s select directly to `listPresets()` (removing the now-redundant static constant) is a follow-up.

## Decisions Made
- Preset select shows `preset.label` verbatim (bundled Russian strings like "Казахстан — четверти") rather than building the label from i18n region/term-mode keys — the bundled preset directory is static, non-localized data (consistent with D-03's "no runtime fetching" design), so the label is not routed through `t()`.
- Kept `listPresets()` called directly in the component body (no `useMemo`) since it returns a fixed module-level array reference — negligible cost, no correctness risk.
