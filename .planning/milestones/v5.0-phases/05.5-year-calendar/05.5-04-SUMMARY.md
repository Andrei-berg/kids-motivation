## Decisions Made
- **Static region options instead of `listPresets()` import (Rule 3 — blocking dependency absent):** the plan says to populate the region select from `listPresets()` in `lib/vacation-presets`, but that module is created by plan 05.5-03 — a sibling wave-2 plan executing in a parallel worktree — and does not exist in this worktree. Importing it would fail tsc and the build. Instead the select uses a `REGION_OPTIONS` constant whose values (`ru-quarters`, `ru-moscow-trimesters`, `kz`, `by`, plus an empty "none") mirror the preset ids plan 05.5-03's frontmatter defines (its `data/vacation-presets/*.json` file stems). Since both controls read/write the same `family_calendar.region_preset` column, WARNING-1 sync holds as long as the ids match. **Follow-up for Plan 05.5-05 (wave 3, depends on both 03 and 04):** switch this select to `listPresets()` (or verify the ids match 05.5-03's as-built output) when both modules coexist in one tree.
- Added a `datesRequiredError` guard (both dates must be set before save) beyond the plan's `year_start <= year_end` check — a half-set range would create a `family_calendar` row that misclassifies days against a single open bound.
- `region_preset` saved as `null` when the empty "none" option is selected (rather than empty string) to keep the column semantics clean.
