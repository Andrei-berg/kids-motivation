# Phase 05.5 Plan 02: Server-Side Streaks + CR-01 Fix Summary

**One-liner:** Closed CR-01 by moving updateStreaks onto the service-role client inside the guarded award route, gating the streak bonus on the server's real today (D-12.1/D-12.2), and making all four streak calculators day-type transparent (D-09/D-10) via an optional backward-compatible resolver.

## What Was Built

### Task 1 — updateStreaks server-side with day-type transparency (`c4c7265`)
- `updateStreaks(admin: Admin, childId, date)` — every `supabase.from()` read/write in `updateStreaks` and the private `updateStreak` helper now uses the passed-in admin client (days, subject_grades, children, streaks reads+writes, chat-post child lookup). This also fixed the latent bug where server-side calls read 0 rows through the sessionless anon singleton.
- Day-type context loaded inside `updateStreaks` via `admin.from('family_calendar')` + `admin.from('vacation_periods')`; each walked day's `is_sick` resolved from the loaded `days` rows; `dayTypeOf` resolver built on 05.5-01's `getDayType`.
- Transparency skip branches: study streak skips `weekend|vacation|sick`; room/sport/behavior skip only `sick`. The resolver is an optional trailing parameter — omitted, calculators behave exactly as before (existing 2-arg tests compile and pass unchanged).
- `getSportActiveDays` gained an optional trailing `client = supabase` param; `updateStreaks` passes admin.
- `getStreakBonuses` reader untouched (clients keep SELECT after the 05.5-07 RLS lock).
- `tests/streaks.test.ts`: added transparency cases (weekend/vacation transparent for study; sick transparent for all four; all-school resolver equals legacy behavior; weekend does NOT skip room). 20/20 pass.

### Task 2 — Hardened award route + repair action (`12329f3`)
- `award/route.ts` calls `updateStreaks(admin, childId, date)` after the auth guard and before the bonus computation, and returns `streakEvents` in the JSON (D-12.2).
- Streak-bonus intent gated behind `date === localDateString()` — bonus payable only for the real server-side today; every other intent (grades, room, behavior, sport, activities, book) remains backfillable (D-12.1).
- `repair-achievements.ts` creates an admin client once and calls `updateStreaks(admin, child.id, today)`.

### Task 3 — Day forms delegate to the award route; dead file deleted (`1f1a8f4`)
- `KidDayFillForm.tsx` and `DailyModal.tsx`: removed the `updateStreaks` import and browser call; both now parse `streakEvents` from the award response (guarded with `?? { broken: [], records: [] }`) and feed the existing fire-and-forget `notifyStreakEvents` push. Day-save logic, room checks, reading log, coin flyup untouched.
- Deleted `components/DailyModal-old.tsx` (imported by nothing; its stale 2-arg call would have broken tsc/build).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Award integration suites crashed at import without env keys**
- **Found during:** Task 2 verification (`npm test`)
- **Issue:** The route's new `streaks.service` import transitively pulls in `lib/supabase.ts`, whose `createBrowserClient` throws at module load when `NEXT_PUBLIC_SUPABASE_URL` is absent — so `tests/integration/award.test.ts` and `room-award.test.ts` failed to even load (their `describe.skipIf` design requires the module graph to import cleanly without env).
- **Fix:** `vi.mock('@/lib/supabase', () => ({ supabase: {} }))` in both suites. The stub is inert when tests do run: the server award path never touches the anon singleton.
- **Files modified:** tests/integration/award.test.ts, tests/integration/room-award.test.ts
- **Commit:** `2a0032a`

**2. [Rule 1 - Test asserting removed behavior] award.test.ts encoded pre-D-12.1 semantics**
- **Found during:** Task 2 verification
- **Issue:** The suite expected a streak bonus for past-date POSTs (2020 dates) and seeded `streaks` rows directly — both invalidated by design: the bonus is now today-only, and `updateStreaks` recomputes (overwrites) seeded counts.
- **Fix:** Past-date test now asserts NO `streak` tx (6 total) plus `streakEvents` present in the response; the settings-driven bonus test was rewritten around the ROOM streak computed from a real `days` row for `localDateString()` (room has no weekend transparency, so it is deterministic on any weekday), asserting the configured 250-coin bonus.
- **Files modified:** tests/integration/award.test.ts
- **Commit:** `2a0032a`
