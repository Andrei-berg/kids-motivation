# Phase 05.7 Plan 06: 3-Tab Awards Screen + Leaderboard Merge Summary

3-tab Awards screen (Бейджи/Стрики/Рейтинг) with accent-indigo XP header; the Rating tab ranks N children via rankChildren podium+list with a solo "Ты против себя" fallback, and /kid/leaderboard is deleted without redirect.

## What was built

### Task 1 — 3-tab Awards shell + XP header (commit 70fad15)
- `app/kid/achievements/page.tsx` rebuilt around `ScreenHeader` (title `kidHeader.awards`, gold coin balance from the child's wallet) — first mount of the shared header on a kid screen.
- XP bar + level card directly under the header, above the tabs (D-05): level via `levelForXp(child.xp)`, `xpInLevel = xp % 1000`, fill color `paper.accent` — XP/level are never gold.
- Shared `Tabs` atom (`theme="paper"`) with the 3 tabs from `t('achievements.tabBadges'|'tabStreaks'|'tabRating')`.
- Badges tab keeps the "Ближе всего" coach card (restyled to a paper card with accent ring/ProgressRing — the old gold-sun gradient violated the gold-only-on-money rule) plus the existing badge grid and bottom sheet, reskinned to 16px radii and 500/600/700 body-font weights.
- Streaks tab holds the four category streak cards.
- Local `useDesktop` duplicate replaced with the shared `lib/hooks/useDesktop` hook.

### Task 2 — Rating tab + route deletion (commit 3308d25)
- Rating data lazy-loads on first tab visit: reuses the old leaderboard's all-children query (`family_members` where `role='child'`, per-child `getChild/getWallet/getWeekScore/getChildBadges/getStreaks`), deduped child ids, each mapped to a `RatingEntry extends RankEntry`.
- ALL entries feed `rankChildren` (Pitfall 5): top-3 podium (avatars sized 76/60, non-gold rank medals from `RANK_TONES = [accent, teal, pink]`, "me" highlighted in accent with the ТЫ label) + list rows for rank 4+. Layout holds for 2 children (two side-by-side podium spots, no VS layout).
- Secondary stats (XP / badge count / streak record) render as neutral `StatChip`s (lineSoft bg, ink2 text) — deliberately not `StatusChip`, whose tones are status-semantic-only.
- Coin figures render exclusively through the gold `Amount` atom.
- D-08 solo (`entries.length === 1`): `soloCompareHeading` card with this-week vs last-week from two `api.getWeekScore(...).total` calls (`getWeekRange(today).start` and `getWeekRange(addDays(today, -7)).start`); `soloCompareEmpty` shown when last week's total is 0.
- `app/kid/leaderboard/page.tsx` deleted — route now 404s, no redirect (D-06). No `/kid/leaderboard` references remain anywhere under `app/kid/achievements`. KidNav's leaderboard tab is Plan 08's scope (untouched); Plan 12 sweeps the rest of the kid surface.
- New i18n keys `achievements.soloThisWeek`/`soloLastWeek` added to both `ru.json` and `en.json` in the same commit (Pitfall 6 parity — spot-checked 1/1 in both files for all 7 keys this plan relies on).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `getWeekScore` return type mismatch in the ported query**
- **Found during:** Task 2
- **Issue:** The plan's "reuse the leaderboard query" snippet assigns `getWeekScore(...)` directly to a `number` field, but the service returns an aggregate object (`{ total, coinsFromGrades, ... }`). The old leaderboard masked this with an unchecked `as Entry[]` cast — its "coins (week)" stat was actually rendering the object coerced at runtime.
- **Fix:** `.then(w => w?.total ?? 0)` on both call sites (rating entries + solo last-week), matching how `app/kid/wallet/page.tsx` consumes the same service (`weekData?.total ?? 0`). The aggregation itself was NOT modified (RESEARCH caveat honored).
- **Files modified:** app/kid/achievements/page.tsx
- **Commit:** 3308d25

No other deviations — plan executed as written.
