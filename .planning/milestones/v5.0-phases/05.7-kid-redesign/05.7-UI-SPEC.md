# Phase 5.7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Nature of this phase:** rollout, not invention. The palette/type/atom
system was locked in Phase 5.3 (design-tokens) against the approved design
contract artifact «семейный банк» v2.3
(https://claude.ai/code/artifact/ab9621cc-2f84-42ff-a873-d07f8b841715). This
UI-SPEC compiles those already-locked values into the contract shape 5.7's
planner/executor/checker need, and adds the phase-specific decisions
(nav, motion, copy) captured in `05.7-CONTEXT.md`. It does not re-litigate
anything Phase 5.3 already decided.

## Design System

| Property | Value |
|----------|-------|
| Tool | none — bespoke hand-built token/atom system, not shadcn |
| Preset | not applicable |
| Component library | none. Shared atoms live in `components/design/atoms.tsx` (`LedgerRow`, `Amount`, `StatusChip`, `useCountUp`, `Stamp`) and `components/kid/design/atoms.tsx` (kid-only: `AnimatedNum`, `Avatar`, `XPBar`, `ProgressRing`, `SectionHeader`, `CoinPill`, `KMButton`, `Confetti`) |
| Icon library | none — hand-drawn inline SVG icon set (stroke-based `<svg>` per icon name, see `KidNav.tsx`'s `TabIcon` and `parent-center/ui.tsx`'s `Icon`). Continue this pattern; do not add `lucide-react`/`react-icons` |
| Font | Bitter (display, weights 600–800) + Golos Text (body, weights 400–700) + JetBrains Mono (numbers, weights 500–700) — Cyrillic + Latin subsets, `next/font/google`, wired as CSS vars in `app/layout.tsx`. Referenced via `base.fontDisplay`/`base.fontBody`/`base.fontMono` in `lib/design/tokens.ts` |

**Why the shadcn gate was skipped:** `components.json` does not exist, and
this is a Next.js app, but a mature, three-phases-deep, explicitly-locked
design system already exists (`lib/design/tokens.ts` + `components/design/
atoms.tsx`, shipped Phase 5.3). Phase 5.3's own CONTEXT.md states "Do NOT
introduce a fourth theme" and every atom "pulls all of its colors from
`lib/design/tokens`." Initializing shadcn now would contradict that explicit
phase boundary and CLAUDE.md's "preserve original logic, change only what's
necessary" rule. Treated as: design system already detected, tool = none,
proceed straight to pre-populating the contract from existing tokens.

## Spacing Scale

`lib/design/tokens.ts` does not export a formal spacing-token object today
(components space via literal pixel values). This phase adopts the standard
8-point scale as the working discipline for every new/reskinned surface —
it is a convention this spec establishes for 5.7, not a pre-existing token
export:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, dotted-leader offsets, chip internal gaps |
| sm | 8px | LedgerRow internal gap, compact row spacing, StatusChip padding unit |
| md | 16px | Default card padding, gap between stacked cards, screen-edge padding |
| lg | 24px | Section-to-section spacing (e.g. header block → tab bar → content) |
| xl | 32px | Space above/below the wallet hero balance, major screen-region breaks |
| 2xl | 48px | Rare — large empty-state illustration spacing |
| 3xl | 64px | Not expected in kid screens (mobile-first, short viewports) |

Exceptions:
- **44×44px minimum touch target** on every interactive element (nav tabs,
  buttons, checkboxes, chips that are tappable) — carried over from the
  existing UX-02 requirement (Phase 4.2), still binding here since this
  phase rebuilds the nav and most toggles.
- **Nav bar height**: full-width ledger panel (D-02) should be tall enough
  to give each of the 5 tabs a ≥44px tap target within an 8pt-aligned
  total height (e.g. 64px or 72px, planner/executor picks one and holds it
  constant).
- **Dotted-leader row height** (`LedgerRow`) is intentionally compact
  (14px name + optional 11.5px sub) — do not force it onto the 8pt scale;
  it is an existing, already-shipped atom, not new spacing to design.

## Typography

`lib/design/tokens.ts` fonts (`base.fontDisplay`/`fontBody`/`fontMono`) are
locked. This phase declares the general UI type scale for anything **outside**
the `Amount`/`Stamp` atoms (which own their own internal sizing —
see Atom-Owned Type below). Existing kid pages currently scatter 9–56px
sizes and 600–900 weights across screens (`achievements`, `day` pages) —
5.7's reskin work should consolidate onto this scale, not add to the sprawl.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 500 (Golos Text) | 1.5 |
| Label | 12px | 600 (Golos Text) | 1.3 |
| Heading | 18px | 700 (Bitter) | 1.2 |
| Display | 24px | 700 (Bitter) | 1.15 |

Two weights only: **500** (default/body) and **700** (emphasis/headings).

**Atom-owned type (exception, already shipped — do not redesign):**
- `Amount` (`components/design/atoms.tsx`): sm 13px/700, md 15px/800,
  lg 22px/800, JetBrains Mono, tabular-nums. This is the only place font
  weight 800 appears — reserved for money figures.
- `LedgerRow` name: 14px/600 Golos Text (slightly bolder than general Body
  to read as a ledger entry title) — already shipped, reuse verbatim.
- `StatusChip` label: 11px/600 Golos Text — already shipped, reuse verbatim.
- Hero/XP big numbers (e.g. wallet hero balance display size, streak-count
  hero) may use Display-role sizing (24px) fed through `useCountUp`, or a
  screen-specific larger hero size if the "сберкнижка" hero (D-16) needs
  more visual weight than 24px — if so, treat it as a `size="xl"` addition
  to the `Amount` atom (extend, don't fork) rather than a bespoke inline
  style, and keep it inside the atom-owned-type exception, not the general
  scale.

## Color

Base palette and both themes are locked (`lib/design/tokens.ts`, Phase 5.3).
Kid screens use the **paper** theme exclusively (`theme='paper'` on every
atom); ink is the parent theme (Phase 5.8) and is only relevant here for
`ChatThread`'s theme-aware prop groundwork (D-11).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `paper.bg` `#FAF3E9` | Screen background on every kid page |
| Secondary (30%) | `paper.card` `#FFFFFF` | Cards, LedgerRow containers, nav ledger panel, bottom sheets, modal surfaces |
| Accent (10%) | `paper.accent` `#5B4BD4` (indigo, text-safe on paper) | **Active nav tab (bold label + icon, D-02), primary CTA buttons (e.g. "Сохранить день"), focus/selected states, tappable links/toggles.** Nothing else — indigo = interactive is the contract's core invariant |
| Destructive | `paper.dangerText` `#B33846` (text) / `paper.danger` `#D95563` (fills/borders) | Logout confirmation button, penalty `LedgerRow` tone, lockout/error messaging only |

**Gold is a separate, absolute invariant — not part of the 10% accent
budget:** `paper.goldText` `#9C6B10` (text-safe) is reserved **exclusively**
for money: wallet hero balance, `Amount` atom with `money={true}`,
`LedgerRow` `tone="earn"`, shop prices. Gold must never appear on a button,
icon, badge, or any non-money UI element — this was the explicit rule locked
in Phase 5.3 and every screen this phase touches must respect it (e.g. the
Awards XP bar, level number, and badge icons are **not** money and must not
be gold).

Success (`paper.success` fill / `paper.successText` text) and warning
(`paper.warning` fill / `paper.warningText` text) are available for
`StatusChip` tones (e.g. purchase request "approved"/"pending") — same
reservation discipline: status semantics only, never decorative.

Accent reserved for: active nav tab indicator + label, primary CTA button
fills, focus rings, the logout button's icon (interactive, not destructive
— the confirmation step is where danger color appears), tappable chips/links.
Never for: card backgrounds, decorative dividers, badge/achievement icon
fills (those use their own icon-native colors or `success`), body text.

## Copywriting Contract

Voice: plain verbs, sentence case, actions named by outcome — matches the
"family-design-writing" discipline in `05.7-CONTEXT.md`'s Claude's
Discretion note and the bank aesthetic's restraint (no leading emoji on new
copy, unlike some legacy strings such as `wallet.saveBtn: "💾 Сохранить"`).
All keys need both `en.json` and `ru.json` entries (existing dotted-key
convention, `public/locales/{en,ru}.json`).

| Element | Copy |
|---------|------|
| Primary CTA (day save, `kidDay.saveDay`) | ru: **«Сохранить день»** · en: **"Save day"** — fires the D-17 stamp+count-up gesture only after `/api/wallet/award` resolves |
| Empty state heading (wallet ledger, `wallet.emptyHeading`) | ru: **«Пока пусто»** · en: **"Nothing here yet"** |
| Empty state body (wallet ledger, `wallet.emptyBody`) | ru: **«Заполни день — здесь появится история операций.»** · en: **"Fill in a day — your transaction history will show up here."** |
| Error state (award/save failure, `kidDay.saveError`) | ru: **«Не удалось сохранить день. Проверь соединение и попробуй ещё раз.»** · en: **"Couldn't save today. Check your connection and try again."** — pairs with a retry action, never a dead-end message |
| Destructive confirmation — Logout (`kidNav.logoutConfirm`) | ru: **«Выйти из аккаунта?»** / body: **«Тебе нужно будет снова войти по коду и PIN.»**, buttons **«Отмена» / «Выйти»** · en: **"Log out?"** / body: **"You'll need to sign back in with your code and PIN."**, buttons **"Cancel" / "Log out"** — the button uses `paper.dangerText`, the trigger icon in the Day header stays indigo (D-03) |

No other destructive actions are in this phase's scope (money mutations —
purchase/exchange/withdraw — already have their own confirmation flows,
unchanged by 5.7; `GoalsPanel` has no delete action).

Rating tab empty/fallback copy (D-08, single-child "я против себя"):
- Heading (`achievements.soloCompareHeading`): ru **«Ты против себя»** ·
  en **"You vs. last week"**
- Body when no prior-week data exists yet (`achievements.soloCompareEmpty`):
  ru **«На следующей неделе здесь появится сравнение с этой неделей.»** ·
  en **"Next week you'll see how this week compares."**

## Motion Contract (phase-specific — supplements the base spec)

Locked by CONTEXT.md D-17–D-20; restated here as an executor-facing
checklist since it is the phase's second major deliverable alongside the
visual reskin.

| Trigger | Old behavior | New behavior | Guard |
|---------|-------------|---------------|-------|
| Day save credits coins | client-side confetti burst | `Stamp` "+N 🪙" (450ms) fires **only** after `/api/wallet/award` response resolves, keyed to `creditedCoins`; header balance `useCountUp`s to the new total | Never animate `coinsPreview` (client estimate) — server value only |
| Room/activity/book/custom checkbox toggle | per-toggle `triggerConfetti()` | micro-tick: checkmark draw-in + scale pulse, ≤200ms, no particles | Remove every `triggerConfetti` call site in `KidDayFillForm.tsx` (room ~635, activity ×2 ~681/1116, book ×2 ~923/1367, custom ~998) and the `<Confetti trigger={confettiTrig}/>` mount (~1028) |
| Streak bonus credited | — | confetti (`utils/confetti.ts`, unchanged mechanism) | Detected from the award response's per-source detail (additive `appliedSources` field, sourceType `'streak'`) — server-confirmed only |
| Level-up (XP threshold crossed) | — | confetti | Client compares `Math.floor(prevXp/1000)+1` vs `Math.floor(newXp/1000)+1` using the existing formula (`lib/services/badges.service.ts:582`) — pure-function extraction recommended for testability |
| Badge earned / goal reached / purchase or exchange approved | confetti (`CelebrationOverlay`, `GoalsPanel`'s `triggerGoalConfetti`) | `Stamp` ceremony: "ПОЛУЧЕНО" / "ЦЕЛЬ ДОСТИГНУТА" card, −8° tilt, 450ms entrance | Reuse the `Stamp` atom's existing reduced-motion double-guard; do not add a third guard layer |
| `prefers-reduced-motion: reduce` | — | every animation above degrades to an instant state change | Already implemented inside `Stamp`/`useCountUp` — new call sites inherit this for free; do not bypass by hand-rolling a fresh animation |
