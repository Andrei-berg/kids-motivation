# Phase 05.7 Plan 11: Day-save signature motion (stamp + count-up + streak/level-up confetti) Summary

Wires the phase's signature interaction: after a day save, the client parses the
`/api/wallet/award` response via `detectAward`, stamps the server-confirmed "+N 🪙",
count-ups the header balance, renders awarded items as `LedgerRow`s, and fires
confetti only for a credited streak source or an XP level-up crossing — never for
an ordinary save.

## What shipped

**Task 1 — `components/kid/KidDayFillForm.tsx`:**
- `handleSubmit`'s award-crediting logic was extracted into `creditAwardAndFinish()`:
  fetches `/api/wallet/award`, parses the response with `detectAward(res)` for
  `creditedCoins`/`hasStreak`, captures the child's `xp` before/after
  `checkAndAwardBadges` and computes `leveledUp` via `computeLevelUp`, then calls
  `onSaved({ creditedCoins, hasStreak, leveledUp, appliedItems })` — never
  `coinsPreview`.
- `triggerCoinFlyup(coinsPreview)` is unchanged — still the pre-save client estimate,
  now clearly off the stamp/confetti path.
- On an award-fetch failure, a `saveError` banner with a retry button appears (no
  stamp fires). Retry calls `creditAwardAndFinish()` directly rather than re-running
  `handleSubmit()`, since the day's rows (grades, room checks, activity logs…) are
  already persisted by the time the award step runs, and some of those writes
  (`saveSubjectGrade`) are insert-only — a full-form retry would create duplicates.
- `onSaved`'s type changed from `(coinsEarned: number) => void` to
  `(result: DaySaveResult) => void`; new exported types `DaySaveResult` and
  `AwardedItem`.

**Task 2 — `app/kid/day/page.tsx`:**
- `handleFillSaved(result: DaySaveResult)` stores `result` as `savedResult` and fires
  `triggerConfetti()` only when `result.hasStreak || result.leveledUp` (D-19),
  replacing the old `coinsPreview > 0 → setConfetti` gating. Removed the
  `<Confetti trigger={confetti}/>` mount and its state.
- `useCountUp(wallet?.coins ?? 0)` drives `headerCoins`, fed to `ScreenHeader`'s
  `coins` prop on both the mobile and desktop mounts (hook called unconditionally
  before the loading/no-child early returns, per Rules of Hooks).
- Added a `Stamp` "+N 🪙" (both the desktop stats-panel celebration card and the
  mobile teal summary card), keyed to `savedResult.creditedCoins`, rendered only
  when `creditedCoins > 0` — skipped entirely on an idempotent re-save.
- Added a day-summary ledger card rendering one `LedgerRow` (`theme="paper"`) per
  `savedResult.appliedItems` entry, `tone="earn"`/`"penalty"` by sign.

## Deviations from Plan

### Auto-fixed / Rule 2 additions

**1. [Rule 2 — missing critical functionality] Extended `/api/wallet/award`'s response with `appliedItems`**
- **Found during:** Task 2 (rendering "awarded items as ledger rows")
- **Issue:** The response shape pinned by the plan's `<interfaces>` block
  (`{ ok, creditedCoins, awards, appliedSources, streakEvents }`) only exposes
  `appliedSources` as an array of bare `sourceType` strings (e.g. `['room','grade','grade']`)
  — no per-item coin amount or description. Building genuinely accurate `LedgerRow`s
  (name + amount) from that alone would require fabricating numbers.
- **Fix:** Added an additive `appliedItems: { sourceType, coins, description, icon }[]`
  field to the JSON response in `app/api/wallet/award/route.ts`, built from the
  `applied: AwardIntent[]` array `creditAwards` already returns — no change to the
  money-mutation logic itself, and the existing `appliedSources` field (consumed by
  Plan 02's `detectAward`, covered by `tests/award-result.test.ts`) is untouched.
- **Files modified:** `app/api/wallet/award/route.ts` (not in the plan's declared
  `files_modified`, but purely additive/read-only — no money-mutation change).
- **Commits:** fc5db16

**2. [Rule 2] Added `saveError` state + retry UI, and `kidDay.*` i18n keys**
- **Found during:** Task 1 action text ("If the award fetch fails, surface
  `t('kidDay.saveError')` with a retry")
- **Fix:** Added `saveError` state, an inline error banner + retry button in
  `KidDayFillForm.tsx`, and new `kidDay.saveError` / `kidDay.retryBtn` /
  `kidDay.awardedHeading` keys in `public/locales/en.json` and `ru.json` (no
  `kidDay` namespace existed previously).
- **Files modified:** `public/locales/en.json`, `public/locales/ru.json`.
- **Commits:** fc5db16

None of the above touch money-mutation logic (`creditAwards`, `applyWalletDelta`)
— they are read-response enrichment and UI/i18n additions only, consistent with
CLAUDE.md's server-side-only rule for money writes.

## Threat Flags

None. `appliedItems` exposes per-item `description`/`coins`/`icon` for the
requesting child's own just-credited award — data already readable by the same
authenticated child via direct `wallet_transactions` SELECT (money tables are
RLS SELECT-only for clients, migration `04.4-03`). No new trust boundary.
