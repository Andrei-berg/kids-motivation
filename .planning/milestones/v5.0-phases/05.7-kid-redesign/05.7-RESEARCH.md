# Phase 5.7: kid-redesign - Research

**Researched:** 2026-07-16
**Domain:** React/Next.js UI redesign (design-system rollout) on an existing Supabase-backed PWA
**Confidence:** HIGH (codebase-verified — no external library research required; every claim below is grounded in the actual repo files read during this session)

## Summary

Phase 5.7 is a **rollout**, not a greenfield build: the design-token foundation
(`lib/design/tokens.ts`, `components/design/atoms.tsx` — `LedgerRow`/`Amount`/
`StatusChip`/`useCountUp`/`Stamp`) and the day-blocks data/render contract
(`lib/day-blocks.ts`'s `assembleDayBlocks`, the `block.block_type → renderer`
seam in `KidDayFillForm`/`DailyModal`) already exist and are proven on two
pilot surfaces (kid wallet history, parent dashboard activity). 5.7's job is to
extend that same atom kit and the same theme-aware pattern (`theme: 'paper' |
'ink'` prop, no context/provider) across every remaining kid screen, rebuild
the 6→5 nav, and replace the current "confetti on everything" motion policy
with the disciplined stamp/tick/count-up vocabulary. No money math, PIN auth,
or day-blocks award logic changes — this is render-layer work sitting on top
of already-shipped server logic.

Three concrete engineering facts should shape planning: (1) the `/api/wallet/
award` response today is `{ ok, creditedCoins, awards: <count>, streakEvents
}` — it does **not** expose per-source detail, so D-17 (stamp only on
server-confirmed numbers) and D-19 (confetti only for streak/level-up) need
one small additive response-shape extension exposing `applied` (already
computed server-side as `AwardIntent[]`, just not returned); (2) three modal
components explicitly listed as "in scope" by CONTEXT.md's canonical_refs —
`ShopModal.tsx`, `ExchangeModal.tsx`, `GoalsModal.tsx` — have **zero import
sites** anywhere in the app; they are dead code, not live kid-flow overlays;
and (3) `ChatThread.tsx` (the component D-11 targets for a paper/ink
theme-aware rebuild) is currently consumed **only** by `/kid/chat` — the
parent side (`components/parent-center/screens/ChatPanel.tsx`, 740 lines) is
a wholly separate, un-shared implementation, so "parent chat must not break"
has no code today that could break, and building an "ink variant" of
`ChatThread` produces a currently-unconsumed capability unless the plan also
wires `ChatPanel` to it (a materially bigger change than CONTEXT.md's framing
suggests). Both findings are flagged in Open Questions for confirmation
before planning locks task scope.

**Primary recommendation:** Treat every screen conversion as "swap bespoke
markup for `LedgerRow`/`Amount`/`StatusChip`/`Stamp`/`useCountUp` from
`components/design/atoms.tsx`, `theme='paper'`" — the same mechanical pattern
already proven in 05.3-03's two pilots — and budget the nav rebuild, the
KidDayFillForm decomposition, and the confetti-call-site sweep as the three
highest-risk/highest-effort waves.

## Project Constraints (from CLAUDE.md)

- **No money-mutation changes.** All money mutations stay server-side via
  service-role client; money tables are RLS SELECT-only for clients. 5.7 must
  not add any client-side wallet write, and the award route's `AwardIntent`/
  `creditAwards` idempotency machinery must be extended (additive fields only),
  never forked or reimplemented.
- **Preserve original logic; change only what's necessary.** KidDayFillForm's
  `handleSubmit`, block-save calls, and the `/api/wallet/award` POST must stay
  byte-identical in *behavior* — D-09 explicitly scopes this to render-only.
- **State assumptions explicitly; ambiguous → propose + ask.** Two material
  ambiguities were found this session (ChatThread parent-sharing claim;
  ShopModal/ExchangeModal/GoalsModal being dead code) — see Open Questions.
- **`localDateString()` for "today", never `toISOString().split('T')[0]`** —
  relevant if any new stamp/confetti trigger logic needs a "today" check
  (e.g. the badge-nudge Sunday-evening gate already follows this correctly and
  is a good reference).
- **Do not remove `globals.css` classes** — legacy pages depend on them; 5.7
  touches only kid-scoped files, so this should not come up, but the shared
  `app/globals.css` must not be edited for kid-only styling needs (use
  component-scoped inline styles / the token system instead, matching every
  existing kid screen's convention).
- **i18n:** all renamed/new labels need both `en.json` and `ru.json` dotted
  keys (existing pattern confirmed in `public/locales/{en,ru}.json`, e.g.
  `kidNav.*`).

### Pattern 1: Theme-aware atom consumption (the proven 05.3 pattern)
**What:** Every reskinned row/amount/status delegates to the shared atom with
an explicit `theme` prop; never hand-color new markup.
**When to use:** Any place currently rendering `T.coral`/`T.teal` colored
amounts, dotted-leader rows, or status pills.
**Example:**
```tsx
// Source: app/kid/wallet/page.tsx (05.3-03, as-built)
<LedgerRow theme="paper" name={label} sub={time} amount={x.coins_change}
           tone={isIn ? 'earn' : 'spend'}/>
```

### Pattern 2: Block-type → renderer contract (5.6, must not be forked)
**What:** `lib/day-blocks.ts`'s `assembleDayBlocks(blocks, dayType, date)` is
called identically by `KidDayFillForm` and `DailyModal`; each block row's
`block_type`/`legacy_key` selects a renderer. 5.7 reskins the renderers'
*inner JSX*, not the assembly/selection logic.
**When to use:** Decomposing `KidDayFillForm` (D-09) — extract each existing
inner-JSX block (room checklist, grade entry, custom-block toggle list, etc.)
into its own component file, but keep the `switch`/lookup that calls
`assembleDayBlocks`'s output untouched.
**Example:**
```tsx
// Source: components/kid/KidDayFillForm.tsx (05.6-04, as-built pattern)
// "Shared content-body extraction" — each built-in section's inner JSX is
// hoisted into a local const, referenced from BOTH the flag-off (hardcoded
// title) and flag-on (block-config title/icon) branches.
```

### Pattern 3: Stamp/count-up orchestration (D-17)
**What:** The `Stamp` atom replays on `trigger` change (key remount); combine
with `useCountUp` for the header balance.
**Example:**
```tsx
// Source: components/design/atoms.tsx (05.3-02, as-built)
const displayed = useCountUp(wallet.coins)           // header balance
// after POST /api/wallet/award resolves:
<Stamp trigger={awardResponse.creditedCoins /* or a monotonic counter */}>
  <span>+{awardResponse.creditedCoins} 🪙</span>
</Stamp>
```
`Stamp` and `useCountUp` already double-guard `prefers-reduced-motion` (CSS
media-query + JS post-mount check) — do not add a second guard layer.

### Pattern 4: Level-up crossing detection (D-19)
**What:** Read `child.xp`/`child.level` *before* calling
`checkAndAwardBadges(childId, date)` (which internally may update `xp`/
`level` via `awardBadge` in `lib/services/badges.service.ts`), then re-fetch
and compare `Math.floor(newXp / 1000) + 1` against the prior level.
**Example:**
```ts
// Formula source: lib/services/badges.service.ts:582 (as-built, unchanged)
const newLevel = Math.floor(newXP / 1000) + 1
// Achievements page already echoes this: xpInLevel = (child?.xp ?? 0) % 1000
```
This is a **client-side comparison of two already-fetched snapshots** — no
new DB field, per D-19's explicit constraint.

### Pattern 5: Prior-week comparison for "я против себя" (D-08)
**Example:**
```ts
// api.getWeekScore signature (lib/services/coins.service.ts) — reuse twice
const thisWeek = await api.getWeekScore(childId, getWeekRange(today).start)
const lastWeek = await api.getWeekScore(childId, getWeekRange(addDays(today, -7)).start)
```
**Caveat:** `getWeekScore` computes coins from a **hardcoded** local formula
(`GRADE_COINS` map, `roomOkDays*3`, `behaviorDays*5`) — it does not read
`wallet_settings` or `day_blocks` pricing (5.6). It is already used this way
today for the wallet page's "earned this week" chip, so reusing it for D-08
is *consistent with existing behavior*, but the comparison is an
approximation, not an exact reconciliation of actually-awarded coins. Do not
"fix" this in 5.7 (would be money-adjacent logic change, out of phase scope)
— just don't be surprised if it drifts slightly from `wallet_transactions`.

### Anti-Patterns to Avoid
- **Forking `creditAwards`/`AwardIntent` to add stamp/confetti detail:**
  extend the route's final `NextResponse.json(...)` call additively; never
  duplicate the idempotency/crediting logic client-side.
- **Stamping `coinsPreview`:** D-17 is explicit — the client-side day-form
  preview total is an estimate only; the stamp animates only the server's
  `creditedCoins`.
- **Re-deriving day-of-week/day-type filtering inside a new component:**
  reuse `lib/day-blocks.ts`'s `assembleDayBlocks` — do not write a third
  implementation of the `day_types`/`'always'`/empty-means-all convention.
- **Hardcoding a 2-child assumption in the Rating tab:** the existing
  leaderboard page's "me vs sibling" logic only ever compares the active
  member against the *first* other child found — a 3+-child family silently
  drops entries. D-07's podium+list must generalize to N children (top-3
  podium, remainder in the list), not just port the 2-child comparison UI.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Ledger-row rendering (name + dotted leader + amount) | A new bespoke row markup per screen | `components/design/atoms.tsx`'s `LedgerRow` | Already theme-aware, AA-contrast-verified, and the explicit brand signature per the design contract |
| Money figure display | Local `toLocaleString('ru-RU')` + manual gold coloring | `Amount` atom (`money` prop gates gold) | Gold-for-money is an enforced invariant (T-053-04) living inside the atom, not scattered across call sites |

## Common Pitfalls

### Pitfall 1: Stamping/confetti-ing an unconfirmed number
**What goes wrong:** A stamp or confetti burst fires off the client-side
`coinsPreview`/`totalCoins` estimate before the server round-trip completes,
so a child sees "+12" stamped, then the header balance count-up lands on a
different number (e.g. if a block was actually already-awarded and
idempotency skipped it).
**Why it happens:** `coinsPreview` already exists client-side for the flyup
UI and is very easy to reach for since it's available synchronously.
**How to avoid:** Wire the stamp trigger strictly to the `/api/wallet/award`
response (`creditedCoins`), per D-17. `coinsPreview` continues to exist for
the *pre-save* flyup estimate only, unchanged.
**Warning signs:** A stamp appears before the network request resolves, or a
stamp value doesn't match a subsequent `wallet_transactions` read.

### Pitfall 2: Confetti call-site sweep is easy to under-scope
**What goes wrong:** D-18 requires removing *every* per-toggle confetti call.
Confirmed call sites in `KidDayFillForm.tsx` alone: room-task toggle (line
~635), extra-activity toggle (×2, lines ~681/1116 — mobile+desktop render
paths), book-finished toggle (×2, lines ~923/1367), custom-block toggle (line
~998), plus the `<Confetti trigger={confettiTrig}/>` mount (line ~1028). Miss
one and SC3 fails a spot-check.
**Why it happens:** The desktop and mobile render paths in `KidDayFillForm`
duplicate several toggle blocks (visible in the line numbers above) — a
search-and-remove pass must catch both copies.
**How to avoid:** `grep -n "triggerConfetti"` across `components/kid/
KidDayFillForm.tsx` before calling the sweep done; also check
`components/kid/GoalsPanel.tsx` (`triggerGoalConfetti` on goal completion —
this one becomes a stamp per D-20, not a removal) and
`components/kid/CelebrationOverlay.tsx` (badge-earned confetti — also
converts to stamp per D-20).
**Warning signs:** A checkbox toggle still bursts confetti after the sweep.

### Pitfall 3: `ShopModal`/`ExchangeModal`/`GoalsModal` are dead code
**What goes wrong:** CONTEXT.md's canonical_refs lists `components/
ShopModal.tsx`, `ExchangeModal.tsx`, `GoalsModal.tsx` as in-scope "purchase/
exchange modals" for D-12. Grepped across the entire `app/` and `components/`
tree: **zero import sites** for any of the three (confirmed via `grep -rln
"from '@/components/ShopModal'" ...` and equivalents — no hits). The kid
shop flow's actual purchase confirmation UI is an inline bottom-sheet inside
`app/kid/shop/page.tsx` (the `pending`/`ApprovalSheet` state), not a modal
component.
**Why it happens:** These three files are very likely leftovers from an
earlier architecture (pre-server-action `requestPurchase`) that were never
deleted when the kid shop page's own inline confirm-sheet was built.
**How to avoid:** Do not spend a task reskinning these three files under the
assumption they're live UI. Verify with a fresh `grep -rln` before planning
a task against them. If genuinely unused, they're 5.11 (legacy-cleanup)
candidates, not 5.7 work — flag to the user rather than silently reskinning
dead code.
**Warning signs:** A reskin task references `<ShopModal>`/`<ExchangeModal>`/
`<GoalsModal>` and nothing renders differently in the running app.

### Pitfall 4: ChatThread is not actually shared with the parent side today
**What goes wrong:** CONTEXT.md's Integration Points claims "ChatThread is
shared with `/parent/chat`". Verified: `app/parent/chat/page.tsx` is a
2-line redirect to `/parent-center`; the actual parent chat UI is
`components/parent-center/screens/ChatPanel.tsx` (740 lines, its own
bespoke implementation, imports `T`/`Btn`/`Pill`/`Icon` from `../ui`/
`../tokens`, never imports `ChatThread`). `ChatThread.tsx` is imported
**only** by `app/kid/chat/page.tsx`.
**Why it happens:** The two chat surfaces likely diverged over time; the
CONTEXT.md framing may be describing an aspirational/planned state rather
than the current one.
**How to avoid:** Do not plan a task that assumes swapping `ChatPanel.tsx`'s
internals for `ChatThread` is a side-effect-free "add an ink theme" change —
that would mean replacing a 740-line component wholesale, which is 5.8-scale
work, not a theme prop. The safe reading of D-11 for 5.7: give `ChatThread`
a `theme: 'paper' | 'ink'` prop (paper is what's actually consumed today);
building out ink theme values is low-cost and forward-compatible, but wiring
`ChatPanel.tsx` to consume `ChatThread(theme='ink')` should be treated as
optional/discretionary, not required for SC1-3, unless the user confirms
otherwise (see Open Questions).
**Warning signs:** A task description says "ChatThread ink variant used by
parent chat" and cites `/parent/chat` — that route doesn't render chat UI.

### Pitfall 5: N-children generalization for the Rating tab
**What goes wrong:** The existing `/kid/leaderboard` page hardcodes a
"me vs one sibling" comparison (`const sibling = entries.find(e => e.id !==
activeMemberId)` — picks exactly one). A family with 3+ children would
silently only ever show the first other child.
**Why it happens:** The original app was built for exactly two kids (Adam &
Alim) — CLAUDE.md's Architecture section notes this legacy explicitly.
**How to avoid:** D-07's podium+list must read *all* `family_members` with
`role='child'` (the query pattern already exists in `/kid/leaderboard`'s
`useEffect` — reuse the query, not the 2-entry comparison logic) and rank by
whatever metric is chosen (weekly coins is the existing default), rendering
top-3 as podium and the rest as a list. D-08's single-child fallback is the
`entries.length === 1` case; the 2-child and 3+-child cases must both flow
through the same N-length podium+list code path, not a special "vs" layout.
**Warning signs:** A 3-child test family only ever shows 2 avatars.

### Pitfall 6: `useT()`/dotted-key i18n parity drift
**What goes wrong:** Every renamed/new label (nav labels, Awards tab labels,
screen-header contextual titles, stamp ceremony copy "ПОЛУЧЕНО"/"ЦЕЛЬ
ДОСТИГНУТА", new unread-badge aria-labels) needs both `en.json` and
`ru.json` entries or the English locale silently falls back/breaks.
**Why it happens:** High key-count phase (20 decisions, ~6 screens) makes it
easy to add a `ru.json` key and forget the `en.json` mirror (05.3-05a's
SUMMARY notes a prior instance of exactly this gap needing a 60+-key backfill
for onboarding).
**How to avoid:** Add/verify both files in the same commit per new key;
spot-check with the existing project convention (`grep -c` key parity check
is precedented in 05.3-01's own acceptance gate).
