## Critical Issues

### CR-01: `family_members_self_update` policy lets a child promote themselves to parent

**File:** `supabase/migrations/05.7-01-chat-read-marker.sql:16-19`
**Issue:** The new permissive UPDATE policy is scoped only by row (`user_id = auth.uid()`), not by column. Postgres RLS policies cannot restrict columns, and no column-level `GRANT`/`REVOKE` or guard trigger exists on `family_members` (verified across all migrations). A child with a valid session (every PIN-login child has one) can issue a direct PostgREST update against their own row:

```
PATCH /rest/v1/family_members?user_id=eq.<own-uid>
{ "role": "parent" }
```

`WITH CHECK (user_id = auth.uid())` passes because `user_id` is unchanged. The row now has `role = 'parent'`, and everything that trusts `family_members.role` falls:

- `requireParent()` (`lib/supabase/admin.ts`) — unlocks all parent-guarded money mutations: rewards CRUD, `wallet_settings` (e.g. set `coins_per_grade_5` to an arbitrary value), purchase approval/refund, `/api/set-child-pin` for siblings.
- `middleware.ts` role guards — access to `/parent/*`.

The same hole allows rewriting `child_id` (impersonate a sibling) or `family_id` (attach to another family, whose data then becomes readable via `get_my_family_ids()`).

**Fix:** Do not grant generic self-UPDATE. Two sound options:

1. Replace the client-side `update` with a SECURITY DEFINER RPC and drop the policy:

```sql
DROP POLICY IF EXISTS "family_members_self_update" ON family_members;

CREATE OR REPLACE FUNCTION mark_chat_read()
RETURNS void LANGUAGE sql SECURITY DEFINER SET search_path = public AS $$
  UPDATE family_members
  SET chat_last_read_at = now()
  WHERE user_id = auth.uid();
$$;
```

and in `lib/repositories/chat.repo.ts` call `supabase.rpc('mark_chat_read')` instead of `.update(...)`.

2. Or keep the policy but add a BEFORE UPDATE trigger that raises an exception when a non-parent's update changes any column other than `chat_last_read_at` (compare `OLD`/`NEW` field-by-field).

Option 1 is simpler and removes the entire attack surface.

## Warnings

### WR-01: Three of four wallet quick actions are dead — one navigates to a non-existent route

**File:** `app/kid/wallet/page.tsx:95-99`
**Issue:** The `QUICK_ACTIONS` hrefs do not resolve:
- `/kid/analytics` — no such route exists (`app/kid/` contains only `achievements, chat, day, login, shop, wallet`); tapping "stats" 404s.
- `/kid/wallet#goals` and `/kid/wallet#gift` — no element with `id="goals"` or `id="gift"` exists anywhere in `app/kid` or `components/kid` (verified by grep), so "save" and "gift" are no-ops on the same page.

Only the shop link works.
**Fix:** Point "stats" at an existing route (or remove it), add `id="goals"` to the `GoalsPanel` wrapper for the `#goals` anchor, and either implement the gift/P2P entry point or drop that action until it exists.

### WR-02: `normalizeDate(new Date())` used for "today" — the exact UTC-date bug CLAUDE.md forbids

**File:** `app/kid/wallet/page.tsx:58,89`; `app/kid/achievements/page.tsx:135`
**Issue:** `normalizeDate` is `toISOString().slice(0,10)` — the UTC date. CLAUDE.md explicitly mandates `localDateString()` for "today" because between 00:00–03:00 local (UTC+3) the UTC date is *yesterday*. Here it feeds `getWeekRange(today)`: on Monday 00:00–03:00 local, the wallet's "earned this week" chip and the rating tab compute *last* week's range, and the solo compare on the achievements rating tab shifts a week.
**Fix:** `const today = localDateString()` (already imported in sibling `app/kid/day/page.tsx`), then `getWeekRange(today)`.

### WR-03: Exception after a successful award silently strands the save — and invites duplicate grade rows

**File:** `components/kid/KidDayFillForm.tsx:513,529-537`
**Issue:** In `creditAwardAndFinish`, `checkAndAwardBadges(childId, date)` (line 513) and the post-award `getChild` chain run *after* `awardOk = true` but outside any try/catch that preserves the success. If `checkAndAwardBadges` throws:
- via `handleSubmit`: the outer catch logs and exits — `onSaved` is never called, `saveError` stays `false`, so the form silently stays in edit mode even though coins were credited. Pressing "Save" again re-runs `saveSubjectGrade` for entries still flagged `saved: false` → duplicate `subject_grades` rows → new row ids → new coin awards (the idempotency key is per row id, so duplicates are *not* deduped).
- via `handleRetryAward`: there is no catch at all — an unhandled promise rejection.

**Fix:** Once `awardOk` is true, make the remainder non-fatal:
```ts
try { await checkAndAwardBadges(childId, date) } catch (e) { console.warn('badges failed', e) }
```
and always reach `onSaved({...})`. Give `handleRetryAward` a catch that sets `saveError`.

### WR-04: `getMessages` loads the OLDEST 50 messages — new messages become unreachable, contradicting the new unread badge

**File:** `lib/repositories/chat.repo.ts:4-18` (interacts with new code at `183-206` and `components/chat/ChatThread.tsx:456`)
**Issue:** `order('created_at', { ascending: true }).limit(50)` returns the first 50 messages ever, not the latest 50. Once a family passes 50 messages, opening chat shows a frozen historical window. The ordering is pre-existing, but this phase's unread feature makes it user-visible and inconsistent: `getChatUnreadCount` counts *newer-than-last-read* messages (which exist beyond row 50), the nav badge lights up, the kid opens chat, cannot see the messages — and `markChatRead` then clears the badge anyway, silently losing them.
**Fix:**
```ts
.order('created_at', { ascending: false }).limit(limit)
// then: return (data ?? []).reverse()
```

### WR-05: ChatThread hardcodes Russian UI strings despite full i18n elsewhere in the phase

**File:** `components/chat/ChatThread.tsx:171,186-187,238,265,336; also 108-109`
**Issue:** The redesigned chat renders `'Лента'`, `'Раздельно'`, `'💬 Сообщения'`, `'✨ Активность'`, `'Нет активности'`, `'Новый значок!'`, `'Сегодня'`/`'Вчера'`, and `` `${count} события/событий` `` as literals, while the same file already uses `useT()` (`t('chat.noMessages')` etc.) and every other reviewed component is fully keyed in `en.json`/`ru.json`. English-language families get a mixed-language chat screen. The plural heuristic `count < 5 ? 'события' : 'событий'` is also wrong for counts ≥ 21 (21 события → should be «событие» for 21, «события» for 22–24).
**Fix:** Move these strings into `chat.*` locale keys (both files) and use the existing plural-aware `t()` pattern used by `kidDayPage.streakDays`.

### WR-06: Shop approval sheet stamps «ПОЛУЧЕНО» on a timer, not on a real state

**File:** `app/kid/shop/page.tsx:289-322`
**Issue:** `ApprovalSheet` advances `sent → seen → approved` purely on `setTimeout` (1.6s / 3.4s). At the timer-driven "approved" stage it fires the D-20 `Stamp` ceremony with `t('stamp.received')` («ПОЛУЧЕНО»/"Received") — while the actual purchase is still `pending` and the adjacent step label simultaneously reads "Ждём вручения от родителя" ("Waiting for parent to deliver"). The stamp ceremony is defined as a mark of a *confirmed* event (it is keyed to server-confirmed `creditedCoins` on the day screen); here it asserts receipt of a reward the parent has not approved, and directly contradicts the pending `StatusChip` shown in "Мои заявки" below.
**Fix:** Replace the stamp at this stage with a neutral "request sent / coins deducted" treatment (that part *is* server-confirmed), and reserve `stamp.received` for the moment a purchase reaches `approved`/`delivered` status — e.g. render it in the purchases list when `p.status === 'approved' || 'delivered'`.

### IN-05: Level computed from two different sources across screens

**Issue:** The achievements header derives level via `levelForXp(xp)`, while the day screen and rating entries display the DB column `child.level`. If the stored `level` ever lags the XP write, the two screens disagree.

### IN-06: Weekly "spent" figure computed from only the last 20 transactions

**File:** `app/kid/wallet/page.tsx:61,89-93`
**Issue:** `weekSpent` is derived from `getTransactions(activeMemberId, 20)`; a week with more than 20 transactions undercounts spending. Display-only, but the hero line presents it as the week's total.
**Fix:** Either fetch transactions since `weekStart` explicitly, or label the figure as "recent".

### IN-07: Parent-preview cookie value used unvalidated

**File:** `app/kid/layout.tsx:24-29`
**Issue:** `kid_preview` cookie content is passed straight into `KidInitializer` as the active member id without checking it identifies a child of the parent's family. RLS confines any forged value to empty screens (no data leak), but validating against the membership roster would fail fast and avoid a confusing blank preview.
**Fix:** Look up the cookie value in `children`/`family_members` for `membership.family_id`; fall back to the normal branch when it doesn't match.

### IN-08: Hardcoded bilingual literals in the grades body

**Fix:** Add `kidFillForm.digitalHint` / `kidFillForm.digitalShort` keys to both locale files.
