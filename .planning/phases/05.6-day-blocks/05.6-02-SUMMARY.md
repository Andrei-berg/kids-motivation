# Phase 05.6 Plan 02: Block Visibility/Assembly + Price/Multiplier Contract Summary

**Pure, unit-tested `assembleDayBlocks`/`resolveBlockPrice`/`resolveMultiplier` module in `lib/day-blocks.ts` (15 vitest cases, tsc-clean, no Supabase imports) plus a server-side `loadFeatureFlag` reader in `app/api/wallet/_lib.ts` — the single shared contract Plans 03 (award) and 04 (forms) will both consume, closing the D-06 drift risk before either downstream plan starts.**

## Accomplishments

- Created `lib/day-blocks.ts`: `assembleDayBlocks(blocks, dayType, date)` filters is_active blocks by the exact `day_types` (`'always'`/empty-means-all) and `days_of_week` (system weekday `jsDay===0?6:jsDay-1`) semantics already established by `getActivitiesForDay`/`getSectionsForDate` in `lib/repositories/expenses.repo.ts`, then sorts by `sort_order` ascending. `resolveBlockPrice(block, settings)` returns the explicit block price when set, else the `wallet_settings` flat-price fallback for room/behavior/book/exercise, else `null` for grade/sport/activity/custom-with-no-price (signaling "use existing per-row computation"). `resolveMultiplier(block, dayType)` returns `block.multipliers[dayType]` when present and `> 0`, else `1`.
- Created `tests/day-blocks.test.ts` — 15 vitest cases covering: grade-hidden-on-vacation/shown-on-school, empty-day_types-shows-always, `'always'`-sentinel, `days_of_week` Tue/Thu filtering + empty-means-all, inactive-blocks-dropped, sort_order preserved, price override vs settings fallback (room + grade/sport/activity + custom), multiplier present/missing/empty. All 15 pass; `npx tsc --noEmit` reports no errors introduced.
- Added `loadFeatureFlag(admin, familyId)` to `app/api/wallet/_lib.ts`, mirroring the existing `loadSettings` admin-select + `maybeSingle` + defaults-merge shape: reads `families.day_blocks_enabled`, defaults to `false` for a missing/legacy row. `creditAwards`, `AwardIntent`, and `loadSettings` left byte-for-byte unmodified (confirmed via `git diff` — additive-only change).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Plan's specified test file path (`lib/day-blocks.test.ts`) is outside vitest's configured include glob**
- **Found during:** Task 1, before writing the test file
- **Issue:** `vitest.config.ts` sets `include: ['tests/**/*.test.ts']`. The existing project convention (confirmed via `tests/day-type.test.ts` testing `lib/day-type.ts`, `tests/coins.test.ts`, `tests/streaks.test.ts`, `tests/helpers.test.ts`) is to place all test files under `tests/`, never co-located under `lib/`. Had the test been written at `lib/day-blocks.test.ts` exactly as the plan's frontmatter/task literally specified, `npm test -- day-blocks` would report "no tests found" (silently 0 tests), directly violating this same task's acceptance criteria ("`npm test -- day-blocks` passes with tests covering: ...").
- **Fix:** Created the test file at `tests/day-blocks.test.ts` instead, following the established `tests/day-type.test.ts` convention exactly (same import style, same `@/lib/...` alias). `lib/day-blocks.ts` itself was created exactly where the plan specified — only the test file's location changed.
- **Files modified:** `tests/day-blocks.test.ts` (created; plan said `lib/day-blocks.test.ts`)
- **Verification:** `npm test -- day-blocks` → "Test Files 1 passed (1), Tests 15 passed (15)"; full `npm test` afterward → "5 passed | 4 skipped (9), 62 passed | 26 skipped (88)" (integration suites skip without live DB creds, consistent with the rest of the repo's test run in this environment).
- **Committed in:** `4d74bc6` (part of Task 1 commit)
