## Accomplishments
- Created `day_blocks` (config, parent/extended-only writes) and `day_block_entries` (completions, family-wide writes) tables with RLS, mirroring the room_tasks/room_checks + calendar-writes-parent-only lineage.
- Idempotent `seed_default_day_blocks` RPC seeds 7 built-in blocks (room/behavior/grade/sport/activity/book/exercise) plus 3 first-class "previously-free" custom blocks (home_help, weekend_homework, reading_extra) that close the D-03/WR-05 false-coin-promise gap.
- Applied migration to prod; backfilled 11 existing families (77 legacy_key rows + 33 seed_key rows); re-ran the backfill and confirmed the counts were unchanged, proving the child_id-IS-NULL-scoped ON CONFLICT arbiters work.
- Ran all three money verify scripts (wallet-rls, award-idempotency, award-reads) against prod post-migration — all green, no regression.
- Built the browser-client repository (`day-blocks.repo.ts`) exposing reads, day-data entry upsert, flag read, and parent CRUD; wired non-fatal seeding into `createFamily`.

## Decisions Made
- Default-row unique indexes scoped to `child_id IS NULL` (with a separate override index reserved for Phase 5.8) — see key-decisions above; this is the exact bug the plan called out and it verified idempotent live (counts unchanged after a second backfill run).
- `day_block_entries` RLS mirrors `room_checks` (family-wide, no role clause) rather than the parent-only pattern used on `day_blocks` config writes — completions need to be kid-writable per D-06/D-07.
- Repo `getDayBlocks` dedupes by legacy_key when both a family-default and a per-child override row exist, preferring the child-specific row (forward-compat for Phase 5.8 per-child pricing, D-04) — not required by this plan's schema (no override rows exist yet) but matches the plan's explicit repo action spec.
