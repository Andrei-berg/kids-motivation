<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| kid client → /api/wallet/award | client sends only {childId, date}; server computes all amounts |
| day_blocks config → coin amounts | a crafted price/multiplier could inflate awards |
| custom_block source_type → idempotency index | a new source_type must not collide or bypass the unique key |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-056-07 | Elevation | award inflation via crafted block price/multiplier | mitigate | Prices/multipliers come from parent-only-writable day_blocks (RLS, Plan 01); server clamps every block coin (Math.round, abs ≤ 100000) so a malformed value cannot mint unbounded coins. |
| T-056-08 | Tampering | idempotency-key collision for custom_block | mitigate | sourceType 'custom_block' is new (no reuse); sourceId = day_block_entries PK; verify-day-blocks-award.mjs proves the 23505 unique-index rejection. |
| T-056-09 | Elevation | kid self-awards a parent-filled custom block | mitigate | Flag-on custom-block loop skips who_fills='parent' blocks when member.role==='child'; behavior/book parent-gates preserved. This is the authoritative who_fills enforcement (day_block_entries RLS is family-wide, so the server, not the DB, is the who_fills boundary). |
| T-056-10 | Spoofing | client forces flag-on to change math | mitigate | flagOn read server-side via loadFeatureFlag(families.day_blocks_enabled); request body cannot influence it. |
| T-056-11 | Repudiation | flag flip silently changes payouts | accept | Flag is operator SQL in v1; flag-off byte-parity proven so no silent change until an operator opts a family in. |
</threat_model>
