# Phase 05.6 Plan 04: Assembled Block Rendering in Both Day-Fill Forms Summary

**KidDayFillForm and DailyModal both branch on `families.day_blocks_enabled`: flag-off keeps every existing hardcoded section byte-for-byte; flag-on renders `assembleDayBlocks(dayBlocks, dayType, date)` — the SAME shared assembler in both forms — with built-in blocks reusing their exact existing inner JSX (extracted into shared content-body variables, never duplicated) and custom blocks (including the 3 seeded "previously-free" defaults) rendering a generic toggle that persists to `day_block_entries` via `saveDayBlockEntries`.**

## Performance

- **Duration:** ~70 min
- **Tasks:** 2 completed
- **Files modified:** 3 (app/kid/day/page.tsx, components/kid/KidDayFillForm.tsx, components/DailyModal.tsx)

## Accomplishments

- `app/kid/day/page.tsx` now loads `getFamilyDayBlocksEnabled` + `getDayBlocks` (active, family+child-scoped) alongside its existing day-type resolution, and passes `dayBlocksEnabled`/`dayBlocks` as two new props to `KidDayFillForm` — no existing props removed.
- `KidDayFillForm` branches its render: flag-off is the original 7 `FillSection` blocks completely unchanged (verified via `git diff` — the JSX moved into an `if/else`-equivalent ternary block, was not rewritten). Flag-on computes `visibleBlocks = assembleDayBlocks(dayBlocks, dayType, date)` and renders one `FillSection` per visible block, titled/iconed from the block's own `name`/`icon` (so a parent rename via Plan 05's settings editor shows up here too). Built-in blocks (room/grade/exercise/sport/book/activity) reuse their exact existing inner JSX via shared content-body consts (`roomBody`, `gradesBody`, `exercisesBody`, `sectionsBody`, `readingBody`, `extraActivitiesBody`) computed once and referenced from both branches. `'behavior'` renders nothing on the kid side — it never had kid-facing UI (good-behavior has always been parent-assessed). Custom blocks (incl. the 3 seeded `home_help`/`weekend_homework`/`reading_extra` defaults) render a generic toggle button (visually matching the extra-activities pattern); `who_fills === 'parent'` blocks render disabled/greyed for the kid (UI-level defense-in-depth — T-056-12 — the award route from Plan 03 is the authoritative backstop since `day_block_entries` RLS is family-wide).
- Save path (flag-on) collects ALL rendered custom-block toggle states and calls `saveDayBlockEntries(childId, date, entries)` — writing both `true` and `false` so unchecking persists — then POSTs `/api/wallet/award` exactly as before; coins remain 100% server-computed.
- `DailyModal` loads the same flag + blocks + today's `day_block_entries` in its existing `dayTypeInfo` recompute effect (keyed off the same deps plus `familyId`), and a `visibleDayBlocks` `useMemo` calls the identical `assembleDayBlocks(blocks, dayTypeInfo.type, date)` call the kid form uses — same inputs, same output, structurally guaranteeing D-06 parity. Flag-off renders the original parent sections completely unchanged, including the orphaned `homeHelp`/`homeHelpNote` fields (byte-parity verified — that whole block only got wrapped in a `!dayBlocksEnabled &&` fragment, its internals untouched). Flag-on renders one `BlockSection` (a new module-scope wrapper matching the existing `scroll-section`/`scroll-section-header` markup) per assembled block using the same shared-body extraction pattern; it explicitly omits the legacy `homeHelp`/`homeHelpNote` fields (the seeded 'home help' custom block supersedes them, D-03) while keeping `diaryNotDone`/`dayNote` always visible in their own section (same "stays outside the block system" pattern KidDayFillForm uses for Mood).
- Both forms' coin previews (`coinsPreview` / `totalCoins`) were extended to add the flat price of any toggled-on custom block when the flag is on, so the pre-save total isn't silently wrong for the new block type (preview only — the server remains the source of truth).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Extended coinsPreview/totalCoins to include toggled custom-block prices**
- **Found during:** Task 1 (KidDayFillForm) and Task 2 (DailyModal)
- **Issue:** The plan's task text didn't explicitly ask for this, but without it, checking a custom block (e.g. the seeded "home help" — 3 coins) would show `+0` in the coin flyup/footer even though the server (`resolveBlockPrice`) will actually credit the block's price — a materially misleading preview for the very feature this plan ships.
- **Fix:** Added a small addend to each form's existing coin-preview computation: sum of `block.price` for toggled-on custom blocks, gated on `dayBlocksEnabled`. Preview only — `/api/wallet/award` remains the sole source of truth for actual credited coins.
- **Files modified:** `components/kid/KidDayFillForm.tsx`, `components/DailyModal.tsx`
- **Verification:** `npx tsc --noEmit` clean; logic mirrors the existing preview patterns already in both files (e.g. `activityCoins`, `roomCoins`).
- **Committed in:** `1d76072`, `9069c9a` (part of each task's commit)
