# Phase 5.6: day-blocks - Context

**Gathered:** 2026-07-14
**Status:** Ready for planning

<domain>
## Phase Boundary

The child's day is assembled from data — day type (from 5.5's `getDayType`) ×
schedule × block rules — not from hardcoded form sections. A `day_blocks` config
per child (type, enabled, day types, who fills, coin price, multipliers, schedule
link, order) drives both day-fill forms; `/api/wallet/award` computes coins from
the assembled blocks keeping idempotency per `(child_id, source_type, source_id)`;
rollout is staged behind a per-family feature flag (own family → beta → all).

Full parent-facing block-management UI ("Day Constructor") is Phase 5.8 — 5.6
ships the data model, the assembled rendering, the award path, and whatever
minimal management surface is needed to configure blocks at all.

</domain>

<decisions>
## Implementation Decisions

### Framing principle (user's overarching directive)
- **D-01:** Maximum flexibility, no hard coupling. The block system must work for
  different ages, countries, school systems, and family formats. Nothing about a
  specific country's grading, a specific age's activities, or the original
  two-kid family may be baked into code paths. When in doubt, prefer the more
  configurable/data-driven option.

### Block model (generic-first)
- **D-02:** The foundation is a **generic block model described by data**. The 7
  existing hardcoded sections (room, behavior/mood, grades, home exercises,
  sport sections, reading, extra activities) become **built-in block types** —
  specializations of the generic model that keep their specialized UI renderers
  (grade entry, room checklist, coach rating…). Additionally, **custom blocks are
  first-class in v1**: a parent can create an arbitrary block (name, price,
  day-type applicability, who fills) rendered as a generic "did it / didn't"
  check. Direction of evolution: toward fully generic — built-in types should
  share as much of the generic pipeline (config, rendering shell, award) as
  possible so future phases can migrate them fully.
- **D-03:** Previously "free" activities that the 5.5 review stripped of false
  coin promises (home help, weekend homework, reading thresholds beyond the book
  award) are **not special-cased**: they become ordinary configurable blocks with
  a price the server actually credits. This closes the WR-05 gap properly.

### Prices & multipliers
- **D-04:** **Block price overrides; `wallet_settings` is the fallback.** A block
  row with a NULL price uses the family's `wallet_settings` value for that block
  type (custom blocks always carry their own price). This enables per-child
  pricing without breaking existing families.
- **D-05:** Day-type **multipliers are part of the v1 data model and the server
  award computation** (e.g. reading ×2 on vacation). User delegated the details
  ("сделать правильно и максимально удобно настраиваемо и гибко") — Claude's
  discretion on exact shape (per-day-type multiplier map, default ×1). Rich
  multiplier-editing UI belongs to 5.8's Day Constructor; 5.6 needs the model +
  award math + at least a minimal editing surface.

### Rendering & rollout
- **D-06:** **Both forms in 5.6:** the kid day form (`KidDayFillForm`) and the
  parent day form (`DailyModal`) render the same assembled block list — the two
  must never show a different day.
- **D-07:** **Staged rollout behind a per-family feature flag:** own family first,
  then beta families, then all. Flag-off families keep the current hardcoded
  forms and award behavior unchanged (byte-parity requirement — see D-09).
- **D-08:** On a vacation day school blocks disappear and schedule-driven blocks
  (section training, extra lesson) appear only on their scheduled days (ROADMAP
  SC2) — block visibility is a pure function of (day type, schedule link, block
  config).

### Award integrity
- **D-09:** `/api/wallet/award` computes coins from the assembled blocks while
  keeping idempotency per `(child_id, source_type, source_id)`. For flag-off
  families the award result must be **identical to today's behavior**. Custom
  blocks get their own source type keyed to the block row (server-side, service
  role, same guard stack). Verify scripts must pass.

### Claude's Discretion
- Storage schema: `day_blocks` table shape (per-child rows following the 5.2
  `room_tasks` precedent), seeding of defaults for new children and backfill for
  existing families, migration strategy.
- Feature-flag mechanics: where the flag lives (families column vs settings),
  how it's flipped (operator SQL in v1 is acceptable), how "beta families" are
  marked.
- Award mechanics: new source types, how block config feeds the seven existing
  award computations, multiplier application order, clamps.
- "Who fills" (kid/parent/both) semantics and enforcement in the two forms.
- Schedule-link mechanics (sections/lessons already have schedules — reuse).
- Block ordering representation and any minimal settings surface needed in 5.6
  (full Day Constructor is 5.8).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `components/kid/KidDayFillForm.tsx` — 7 hardcoded `FillSection` blocks (lines
  ~586-928); the assembled renderer replaces this section list. Already loads
  `wallet_settings`, `room_tasks`, sections, activities per date.
- `components/DailyModal.tsx` — parent-side equivalent; holds `familyCalendar`
  state and day-type recompute effect from 5.5.
- `lib/repositories/room.repo.ts` + `supabase/migrations/05.2-01` — the
  config-table pattern (seed RPC on family creation, backfill migration,
  legacy-row delete guard) to mirror for `day_blocks`.
- `lib/repositories/schedule.repo.ts` — subjects, per-weekday schedule,
  exercise types; `getSectionsForDate` in expenses.repo — schedule-link sources.
- `lib/day-type.ts` — stable `getDayType` output (`school|weekend|vacation|sick`)
  that block visibility keys on.
- `app/api/wallet/award/_lib.ts` + route — settings loading, clamps, idempotent
  `creditAwards` machinery to extend, not fork.

### Established Patterns
- No feature flags exist anywhere in the codebase — the per-family flag is a new
  pattern; keep it minimal (column + one read path), operator-flipped in v1.
- Config tables are family-RLS client-writable (room_tasks, family_calendar
  post-05.5-04: parent-only writes) — `day_blocks` writes should be parent-only
  from day one (mirror migration 05.5-04), since prices feed money computation.
- Award reads saved rows only (client sends `{childId, date}`) — block-driven
  award must keep this: config + saved rows in, coins out.
- i18n: block titles for built-in types via locale keys; custom block names are
  user data (stored as-is).

### Integration Points
- `app/kid/day/page.tsx` — computes dayType and passes to KidDayFillForm; the
  assembled block list load joins here.
- `/api/wallet/award` — single place where block config changes money; flag
  check must live server-side here too (not only in the UI).
- Phase 5.7/5.8 redesigns consume the assembled renderer — keep block-render
  contract (type → renderer component) clean for reskinning.

</code_context>

<specifics>
## Specific Ideas

- «Мы должны создать что гибко настраивается и не имеет жёсткой привязки — для
  разных возрастов, стран и т.д.» — the user's core directive; treat every
  hardcoded assumption (grading scale, activity set, weekday semantics) as a
  candidate for configuration, within this phase's scope.
- User explicitly delegated technical shape of multipliers and "free" blocks:
  «сделать правильно и максимально удобно настраиваемо и гибко».

</specifics>
