## Summary

Reviewed the day-blocks feature (config tables + RLS, seed function, feature-flagged award
route branch, both day-fill forms, settings manager, repo, tests, verify script) against
diff base `0d2845d8`. Verification performed: `npx tsc --noEmit` clean, `tests/day-blocks.test.ts`
15/15 pass, and a whitespace-insensitive diff confirmed the award route's flag-OFF branch is
byte-identical to the pre-5.6 intents block (D-09 parity holds; the only change is the
additive `is_sick` column in the hoisted `days` select).

Two Critical findings: (1) the kid day-fill form **lost the Room checklist section in the
flag-off render path** — a regression affecting every existing family that also silently
clobbers saved room state on submit; (2) the new `custom_block` award keys idempotency to
the `day_block_entries.id` UUID, which a child can rotate via REST delete+recreate to mint
unbounded coins. Both must be fixed before ship. The migration's RLS and unique-index design
is otherwise sound, and the server-side-flag / server-recompute architecture (T-056-05/07/10)
is correctly implemented.

## Critical Issues

### CR-01: Flag-off kid day form no longer renders the Room checklist — regression + silent data clobber

**File:** `components/kid/KidDayFillForm.tsx:1074-1357`
**Issue:** Pre-5.6, the kid form rendered the room `FillSection` unconditionally (old file
line 586, before the Mood section). In the new code, `roomBody` (defined at line 619) is
rendered **only** inside `renderBuiltinBlock('room')` (line 943), which is reachable only on
the flag-ON branch. The flag-OFF branch (line 1078-1357) renders activities, grades,
exercises, sections, and reading — **no room section, no photo-proof UI**. Since
`day_blocks_enabled` defaults to `false` for every family, all existing families lose the
room checklist from the kid form. Worse, `handleSubmit` still runs the room dual-write with
`roomChecked` stuck all-false: `saveDay` writes `roomBed..roomTrash = false` (line 463-467)
and `saveRoomChecks` writes `done: false` for every task (line 492-497), **overwriting any
previously saved room state** every time the kid saves, and `/api/wallet/award` then finds
`roomComplete === false` — the child can never earn room coins.
**Fix:** Render the room section in the flag-off branch, exactly as pre-5.6:
```tsx
{/* inside the flag-off <> branch, before or after Mood as pre-5.6 (it was before Mood) */}
<FillSection title={t('kidFillForm.roomSection')} icon="🏠"
  sub={`${Object.values(roomChecked).filter(Boolean).length}/${roomTasks.length}`}>
  {roomBody}
</FillSection>
```

### CR-02: `custom_block` award idempotency keyed to entry UUID — child can mint coins via delete+recreate

**File:** `app/api/wallet/award/route.ts:564-570` (with `supabase/migrations/05.6-01-day-blocks.sql:220-228`)
**Issue:** The custom-block intent uses `sourceId: entry.id`. `day_block_entries` RLS grants
**family-wide INSERT/UPDATE/DELETE with no role clause** (policy bodies at migration lines
195-228), so a child-role session can, via the REST API: insert an entry with `done=true` →
POST `/api/wallet/award` (credit) → DELETE the entry → re-INSERT it (new UUID, the
`(child_id,date,block_id)` unique constraint does not preserve the id across delete+insert)
→ POST award again → credit again. Loop is unbounded — a money-minting vector requiring only
the child's own token. The `(child_id, source_type, source_id)` unique index cannot stop it
because the source_id rotates. The table already has the natural key needed for stable
idempotency.
**Fix:** Use a deterministic source id derived from the natural key instead of the row UUID:
```ts
intents.push({
  coins: final,
  description: block.name,
  icon: block.icon ?? '⭐',
  sourceType: 'custom_block',
  sourceId: `${date}:${entry.block_id}`, // child_id is already part of the idempotency key
})
```
(Defense-in-depth: also consider removing the child-role DELETE capability on
`day_block_entries`; the forms only upsert, never delete.)

## Warnings

### WR-01: Award route reads `day_block_entries` without a family scope; entries RLS never ties `child_id` to `family_id`

**File:** `app/api/wallet/award/route.ts:550-554`; `supabase/migrations/05.6-01-day-blocks.sql:195-203`
**Issue:** The admin query filters only `child_id` + `date`. The INSERT policy's WITH CHECK
validates only that the **row's** `family_id` is one of the caller's families — it never
checks that `child_id` belongs to that family. Any authenticated user can therefore insert
`day_block_entries` rows for another family's child (with their own `family_id`). The award
route (service role, RLS bypassed) would then read those rows for the victim child.
Exploitation is gated on guessing the victim family's block UUIDs (`customBlockIds.has(...)`
check), so practical risk is low — but two cheap layers are missing.
**Fix:** Add `.eq('family_id', member.familyId)` to the entries query in the award route,
and tighten the RLS WITH CHECK to also require
`child_id IN (SELECT id FROM public.children WHERE family_id = day_block_entries.family_id)`.

### WR-02: Kid can pre-check a `who_fills='parent'` block that then credits on the next parent save

**File:** `app/api/wallet/award/route.ts:559`; `supabase/migrations/05.6-01-day-blocks.sql:177-183`
**Issue:** The server guard skips parent-only blocks only when `member.role === 'child'`
**at award time**. Since entries RLS is family-wide, a kid can insert `done=true` for a
parent-only block via REST; when the parent later saves the day (DailyModal POSTs
`/api/wallet/award` as parent), the entry credits. The parent modal does show the entry
pre-checked (loaded into `customBlockDone`), so it is visible — but an inattentive parent
saving the day launders the kid's self-award. T-056-09 claims "the server … is the
who_fills boundary", which is only true for kid-triggered awards.
**Fix:** Record who created/last-set the entry (e.g. a `filled_by` column set from the RLS
session or written by both forms) and have the award route require `filled_by != 'child'`
for `who_fills='parent'` blocks — or at minimum call this residual risk out in the plan/docs.

### WR-03: `saveVacationMultiplier` cannot clear a multiplier once set

**File:** `components/settings/DayBlocksManager.tsx:107-120`
**Issue:** When the input is cleared, `nextValue` is `undefined` and the update payload
spreads the existing map unchanged: `{ ...(block.multipliers ?? {}), ...({}) }` — the
`vacation` key persists forever (and since `undefined !== current`, a pointless no-op DB
write is issued). Also `Number('abc')` → `NaN`, which JSON-serializes to `null` inside the
JSONB map. A parent who set vacation ×2 can never turn it off from this UI.
**Fix:**
```ts
const next = { ...(block.multipliers ?? {}) }
if (nextValue === undefined || Number.isNaN(nextValue)) delete next.vacation
else next.vacation = nextValue
await updateDayBlock(block.id, { multipliers: next })
```

### WR-04: Flag-on empty-assembly fallback diverges between KidDayFillForm, DailyModal, and the award route

**File:** `components/kid/KidDayFillForm.tsx:1074-1078`; `components/DailyModal.tsx:1282-1288`; `app/kid/day/page.tsx:119-129`
**Issue:** With the flag ON and zero visible blocks (all filtered/inactive, or the
`getDayBlocks` fetch failing in `app/kid/day/page.tsx`'s catch → `dayBlocks=[]` while
`dayBlocksEnabled=true`), the kid form falls back to the **full legacy UI**
(`dayBlocksEnabled && visibleBlocks.length > 0 ? blocks : legacy`), while DailyModal renders
neither branch's sections and the award route credits nothing except streak. A kid fills
legacy sections, sees a non-zero `coinsPreview`, and the server awards 0 — the exact D-06
parity violation the shared assembler was meant to prevent.
**Fix:** Branch on `dayBlocksEnabled` alone (as DailyModal does); render an explicit empty
state when the assembled list is empty. Treat a failed block fetch under flag-on as an error
state, not as flag-off.

### WR-05: Book/exercise sign guards differ between flag-off and flag-on branches

**File:** `app/api/wallet/award/route.ts:289` vs `app/api/wallet/award/route.ts:499-501,526-527`
**Issue:** Flag-off credits the book bonus only when `settings.coins_per_book > 0`; flag-on
uses `final !== 0`, so a negative price override or negative `coins_per_book` produces a
**penalty award** under flag-on that flag-off would skip. Same for the new exercise source
(`final !== 0` with a settings-sourced base that can be negative). Divergent semantics for
the same configuration value across the flag.
**Fix:** Pick one rule; if negative book/exercise awards are not intended, use `final > 0`
in the flag-on branch.

### WR-06: Kid form resolves day type without sickness — flag-on visible set drifts from the award route on sick days

**File:** `app/kid/day/page.tsx:110-112`; `components/kid/KidDayFillForm.tsx:162-165`
**Issue:** The kid page calls `getDayType(today, false, …)` (isSick hardcoded false) and maps
`'sick'` → `'school'` before passing `dayType` to the form, so `assembleDayBlocks` in the kid
form filters with `school` on a sick day. The award route resolves `dayType` from
`days.is_sick` (route line 313-314) and filters/multiplies with `sick`. Result: on flag-on
sick days the kid sees (and previews coins for) blocks like the `{school}`-scoped grade
block that the server will not credit, and sick multipliers never show client-side.
**Fix:** Pass the day's `is_sick` into the kid page's `getDayType` call (it already loads
`todayDay`) and let the form receive the true day type for block assembly (UI can still
render the school layout).

### WR-07: `seed_default_day_blocks` is SECURITY DEFINER and executable by any authenticated user for any family

**File:** `supabase/migrations/05.6-01-day-blocks.sql:238-269`
**Issue:** `GRANT EXECUTE … TO authenticated` with an arbitrary `p_family_id` parameter and
no caller check means any logged-in user — including a child, or a member of a different
family — can insert the 10 default rows into **any** family. The built-ins are conflict-
protected, but the 3 seeded custom blocks are deletable by parents; anyone can re-create
them (priced 3/5/2 and immediately creditable) against the family's will, and a child can
re-seed blocks their parent deliberately deleted. Money-adjacent config should not be
writable cross-family.
**Fix:** Inside the function, require membership:
```sql
IF NOT EXISTS (
  SELECT 1 FROM public.family_members
  WHERE user_id = auth.uid() AND family_id = p_family_id AND role IN ('parent','extended')
) THEN RAISE EXCEPTION 'not a parent of this family'; END IF;
```
(Verify `lib/onboarding-api.ts` creates the creator's `family_members` row before Step 5c so
onboarding still passes, or move the seed call server-side.)

### WR-08: Coin previews claim to mirror the server but ignore block visibility, price overrides, and multipliers under flag-on

**File:** `components/DailyModal.tsx:648-698`; `components/kid/KidDayFillForm.tsx:302-359`
**Issue:** The DailyModal comment asserts "previews mirror exactly what /api/wallet/award
credits", but under flag-on `totalCoins` still sums room/behavior/grades/book/activities
from the hardcoded flag-off formula regardless of whether those blocks are visible today,
and neither preview applies `resolveBlockPrice` overrides or day-type multipliers (custom
blocks preview flat `b.price` only). Parents/kids are promised amounts the server will not
credit (or under-shown multiplied amounts) — the WR-05-class false-coin-promise this phase
set out to close.
**Fix:** Under flag-on, compute previews from `visibleBlocks` using the shared
`resolveBlockPrice`/`resolveMultiplier` helpers (they are isomorphic and already imported),
gating each source on its block's visibility.

### IN-04: DayBlocksManager numeric inputs are unvalidated

**File:** `components/settings/DayBlocksManager.tsx:62,97,109`
**Issue:** `Number(price)` / `Number(trimmed)` accept NaN (serialized to `null` by JSON),
negatives, and fractions with no client-side feedback. The server clamp (±100000) bounds
damage, but a parent typo silently produces surprising prices.
**Fix:** Validate with `Number.isFinite` and a sane range before calling the repo; show the
error inline.

### IN-05: `childId` string-interpolated into PostgREST `.or()` filters

**File:** `lib/repositories/day-blocks.repo.ts:28`; `app/api/wallet/award/route.ts:323`
**Issue:** `.or(\`child_id.is.null,child_id.eq.${childId}\`)` breaks (or widens the OR group)
if a `children.id` ever contains `,`/`(`/`)`. Currently mitigated — the award route runs
`assertChildInFamily` first and ids are app-generated — but it is an injection-shaped
pattern on a money-adjacent read.
**Fix:** Validate the id shape (e.g. `/^[A-Za-z0-9_-]+$/`) before interpolating, or split
into two queries.

### IN-06: No code path writes `families.day_blocks_enabled`

**File:** `supabase/migrations/05.6-01-day-blocks.sql:113`; (no writer anywhere in app/ components/ lib/)
**Issue:** The flag can only be flipped via direct DB access; DayBlocksManager edits config
that has no user-visible effect until then. If this is intentional staged rollout, fine —
otherwise a parent-facing toggle (parent-guarded API route; the `families` table must not be
client-writable for this column without a column-scoped policy) is missing.
**Fix:** Confirm intent in the phase docs; add the toggle in a follow-up if needed.

### IN-07: `reorderDayBlocks` issues N sequential non-atomic updates

**File:** `lib/repositories/day-blocks.repo.ts:197-206`
**Issue:** A failure mid-loop leaves a partially applied order (the caller reloads, so the
UI recovers, but the stored order is scrambled).
**Fix:** Use a single upsert of `{id, sort_order}` pairs or an RPC.
