### WR-01: Re-running migration 05.6-01 silently reverts the 05.6-02 hardening

**Files modified:** `supabase/migrations/05.6-01-day-blocks.sql`
**Commit:** 9d37040
**Applied fix:** Updated 05.6-01 in-place so a re-run converges on the 05.6-02
final state: the `day_block_entries` DELETE policy creation is replaced by a
drop-only statement with a do-not-recreate warning; the entries INSERT/UPDATE
policies now carry the 05.6-02 child_id-to-family WITH CHECK tie; the
`seed_default_day_blocks` body is the guarded 05.6-02 version pasted verbatim
(verified byte-identical via diff); the backfill is scoped with
`WHERE NOT EXISTS (... day_blocks ...)` so a re-run never invokes the guarded
function with a NULL `auth.uid()` (SQL editor context) — it selects zero
families on any already-seeded database, and a genuinely unseeded family fails
loudly instead of running unguarded. Header updated with a keep-in-sync warning.

### WR-02: Kid form flag-on with zero visible blocks falls back to the full legacy form

**Files modified:** `components/kid/KidDayFillForm.tsx`, `public/locales/en.json`, `public/locales/ru.json`
**Commit:** 532e2bf
**Applied fix:** The render branch is now `dayBlocksEnabled ? (blocks | empty
state) : (legacy form)` — flag-on is authoritative and the legacy form can no
longer render for a flag-on family, matching DailyModal. A "nothing to fill in
today" `FillSection` renders when `visibleBlocks.length === 0` (new locale keys
`kidFillForm.nothingTodaySection` / `kidFillForm.nothingToday` in en+ru).
`coinsPreview` gains a `legacyVisible(key)` gate: under flag-on, room /
activity / grade / sport(coach) / book contributions only count when the
corresponding built-in block is visible today, so the hero card and save
button never promise coins the server's flag-on path will not credit.
Note: the transient divergence when the kid page's flag *fetch* fails
(`app/kid/day/page.tsx` catch → flag treated as off) is unchanged — the
review's fix prescribed the form-side change only; the page-side catch is a
deliberate availability fallback.

### WR-03: loadFeatureFlag swallows query errors, silently mispricing awards

**Files modified:** `app/api/wallet/_lib.ts`
**Commit:** 17bc610
**Applied fix:** `loadFeatureFlag` now destructures `{ data, error }` and
throws on `error` (the award route's try/catch turns it into a 500 the client
can retry) instead of silently picking the flag-off branch and letting the
idempotency index lock in legacy-priced amounts. A missing row
(`data === null`) still means flag-off per D-09. Docstring updated; the sole
caller (`app/api/wallet/award/route.ts:115`) is inside the route's try/catch.

### WR-04: Clearing the vacation-multiplier input does not clear the stored multiplier

**Files modified:** `components/settings/DayBlocksManager.tsx`
**Commit:** cf9643e
**Applied fix:** `saveVacationMultiplier` builds the multipliers map
explicitly: `delete next.vacation` when the field is cleared or the value is
non-finite, `next.vacation = nextValue` otherwise — exactly the reviewer's
suggested code. A cleared field now genuinely deactivates the multiplier in
every award computation.

### WR-05: Award route dedupes per-child overrides differently from the client repo

**Files modified:** `app/api/wallet/award/route.ts`
**Commit:** 309141f
**Applied fix:** Added `.eq('is_active', true)` to the route's `day_blocks`
query so inactive rows are filtered in SQL *before* `dedupeBlocksForChild`
runs — identical semantics to the client repo's `getDayBlocks`
(`activeOnly: true`) used by both forms. An inactive per-child override can
no longer shadow an active family default server-side. Comment updated to
document the ordering requirement.

### WR-06: verify-day-blocks-award.mjs failure paths leave an active earnable synthetic block

**Files modified:** `scripts/verify-day-blocks-award.mjs`
**Commit:** fcb46dd
**Applied fix:** All failure paths inside the rotation-probe `try` now `throw`
instead of `process.exit(1)` (which bypassed the catch cleanup). The catch
cleanup was extended to also delete the natural-key `wallet_transactions` row
(`naturalSourceId` hoisted out of the try), the stale entry id is nulled when
the delete-then-reinsert rotation step fails half-way, and the probe block is
created with `is_active: false` so it can never render as an earnable block in
a real family's forms even mid-run (the probe never reads it back through the
assembler, so the flip does not weaken the exploit-path test).
