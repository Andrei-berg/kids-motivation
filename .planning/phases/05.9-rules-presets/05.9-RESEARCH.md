# Phase 5.9: rules-presets - Research

**Researched:** 2026-07-21
**Domain:** De-hardcoding a Next.js/Supabase money-adjacent settings model (rule presets, per-family grade scale, behavior tag set) — server-side award computation, RLS-aware schema migration, parent/child form UX
**Confidence:** HIGH

## Summary

This phase is three independent de-hardcoding features layered onto an already-established
architecture: server-side `wallet_settings` + idempotent `/api/wallet/award` recompute. All
three features slot into existing seams — `SETTINGS_DEFAULTS`/`loadSettings` in
`app/api/wallet/_lib.ts`, the grade/behavior branches in `app/api/wallet/award/route.ts`, and
the `CoinsRulesTab` editor in `components/parent-center/screens/Settings.tsx` — rather than
requiring new architecture. The behavior-tag-set feature is a direct third application of the
Room Tasks pattern (Phase 5.2): family-level definition table + per-child+date marks table +
legacy-delete guard + dual-write/fallback, this time with a propose/approve status column
borrowed from the `reward_purchases` request→approve→credit pattern.

Live prod DB verification (via `SUPABASE_DB_URL`) confirms CONTEXT.md's D-06 claim exactly:
`subject_grades.grade` is `INTEGER NOT NULL CHECK (grade >= 2 AND grade <= 5)` — note this
CHECK **excludes grade 1** even though `coins_per_grade_1` exists in `wallet_settings` and the
award route's `gradeCoins()` switch handles `case 1`. Grade 1 has never been storable in prod;
the CHECK-drop in this phase is what will make it possible for the first time. Research also
uncovered a **pre-existing, currently-broken code path** unrelated to this phase's headline
scope but directly touched by D-04: `lib/onboarding-api.ts`'s `completeOnboarding()` writes
`wallet_settings` via the **browser** Supabase client, but `wallet_settings` has been RLS
locked to SELECT-only for authenticated clients since migration `04.4-03`. Every new-family
onboarding flow that reaches this step almost certainly already fails today (RLS denies the
upsert, the error surfaces to the user via `handleFinish`'s catch block). This phase's planned
rework of `completeOnboarding` (replacing manual coin fields with a preset selection) is the
natural place to also fix this — route it through a server-side path
(new API route or server action), never re-introduce a client-side wallet write.

**Primary recommendation:** Extend `wallet_settings` with `grade_scale` (enum) and
`grade_coin_map` (JSONB) columns, keep `coins_per_grade_1..5` as the five-point default/
migration source (do not delete), widen `subject_grades.grade` from `INTEGER` to `TEXT` and
drop its CHECK constraint, and build `behavior_tags`/`behavior_marks` as a byte-for-byte
structural clone of `room_tasks`/`room_checks` (Phase 5.2) with an added `status` column on
the marks table for the propose/approve flow. Route every write through the existing
server-side seam (`/api/wallet/settings` PATCH for presets/grade_coin_map, a new admin-guarded
route or server action for behavior tag approval) — never a client-side wallet write.

|------------|-------------|----------------|-----------|

| Preset apply (write `wallet_settings`) | API / Backend (`/api/wallet/settings` PATCH) | — | `wallet_settings` is RLS SELECT-only for clients (04.4-03); every write is server-side, parent-guarded (`requireParent`) |
| Diff-preview computation (old→new values) | Frontend (Client Component) | — | Pure read+compare of already-loaded settings + a static preset-value table; no privileged data needed |
| `grade_scale`/`grade_coin_map` storage | Database / Storage (`wallet_settings`) | API (settings PATCH) | Same table, same write path as presets |
| Grade input UI (data-driven per active scale) | Frontend (Client Component) | — | Renders N rows/options from `grade_coin_map` keys, no privileged logic |
| `subject_grades` row write (grade entry) | Frontend → Database directly | — | `subject_grades` is family-isolation RLS (client-writable), **not** a money table — unchanged write path (`grades.repo.ts`), only the column type/value shape changes |
| Grade→coins award mapping | API / Backend (`/api/wallet/award`) | — | Money computation is exclusively server-side per CLAUDE.md; `gradeCoins()` must switch from a numeric switch to a `grade_coin_map[grade]` lookup |

| Behavior mark approval (coins credit) | API / Backend | — | Approval is the money-mutation trigger — must go through a service-role-guarded route/action, mirroring `app/parent/shop/actions.ts` approve/reject |
| Behavior coins award computation | API / Backend (`/api/wallet/award`) | — | Sums approved `behavior_marks` prices for the day — replaces the single `day.good_behavior` boolean check |

### Pattern 1: Preset Apply as a Table Lookup, Not a Branch Tree

**What:** Since D-01 establishes only **two** underlying numeric value sets (Classic vs. the
shared No-penalties/Bonuses-only set) behind **three** UI cards, model presets as a lookup
table keyed by preset id → a fixed `Partial<WalletSettings>` patch object, not as
if/else logic scattered through a component.

**When to use:** Both the onboarding preset picker and the Settings "Apply preset" flow should
call the same shared function, e.g. `getPresetValues(presetId): Partial<WalletSettings>`.

**Example:**
```typescript
// lib/presets.ts (new, shared by onboarding + Settings)
export type PresetId = 'classic' | 'no_penalties' | 'bonuses_only'

const CLASSIC: Partial<WalletSettings> = {
  coins_per_grade_5: 10, coins_per_grade_4: 5, coins_per_grade_3: -3,
  coins_per_grade_2: -5, coins_per_grade_1: -10,
  coins_per_room_task: 3, coins_per_good_behavior: 5, coins_per_exercise: 5,
  coins_per_coach_5: 10, coins_per_coach_4: 5, coins_per_coach_3: 0,
  coins_per_coach_2: -3, coins_per_coach_1: -10, coins_per_book: 20,
}
// D-01: No-penalties and Bonuses-only are numerically IDENTICAL — negatives floored to 0.
const NO_PENALTY_VALUES: Partial<WalletSettings> = {
  ...CLASSIC, coins_per_grade_3: 0, coins_per_grade_2: 0, coins_per_grade_1: 0,
  coins_per_coach_2: 0, coins_per_coach_1: 0,
}

export function getPresetValues(id: PresetId): Partial<WalletSettings> {
  if (id === 'classic') return CLASSIC
  return NO_PENALTY_VALUES // 'no_penalties' | 'bonuses_only' share one value set (D-01)
}
```
D-02 scope note: this patch object must touch **only** the fields D-02 lists (grade coins,
room/behavior/exercise, coach ratings, book) — never `streak_*`, `base_exchange_rate`, or P2P
limit fields.

### Pattern 2: `grade_coin_map` Lookup Replaces the Numeric Switch (Server AND Client)

**What:** `gradeCoins()` in `app/api/wallet/award/route.ts` currently switches on `grade: number`
(1-5). Per D-06 it must become a direct `settings.grade_coin_map[String(grade)]` lookup with no
numeric conversion in either direction.

**When to use:** Award route (server, authoritative) — and any client-side coin **preview**
that currently duplicates this switch (`DailyModal.tsx` `gradeCoinsFor`, `KidDayFillForm.tsx`
`coinsPreview` block, `BulkModal.tsx`'s hardcoded 2-5 range) should mirror the same lookup
shape so previews don't silently diverge once `grade_scale` is non-default.

**Example:**
```typescript
// app/api/wallet/award/route.ts — replace the switch statement
function gradeCoins(s: { grade_coin_map?: Record<string, number> }, grade: string): number {
  return s.grade_coin_map?.[grade] ?? 0
}
// subject_grades.grade is now TEXT — no Number(grade) cast anywhere in this path.
```

### Pattern 3: Behavior Tags as Room Tasks' Third Application

**What:** `behavior_tags` (family-level: id, family_id, name, icon, price, active, order) +
`behavior_marks` (child_id, date, tag_id, **status**, proposed_by, decided_by, decided_at) —
structurally identical to `room_tasks`/`room_checks` (`supabase/migrations/05.2-01-room-tasks.sql`)
with one addition: the `status` enum column on marks, absent from `room_checks` (room checks
have no approval step).

**When to use:** Migration DDL, repo layer (`behavior.repo.ts`), and the Settings manager
component should copy `room.repo.ts`/`RoomTasksManager.tsx` near-verbatim, changing only the
table/column names and adding propose/approve-specific functions (`proposeTag`,
`approveMark`, `rejectMark`).

**Example (migration skeleton, mirrors 05.2-01 exactly):**
```sql
CREATE TABLE IF NOT EXISTS public.behavior_tags (
  id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  family_id   UUID        NOT NULL REFERENCES public.families(id) ON DELETE CASCADE,
  name        TEXT        NOT NULL,
  icon        TEXT,
  price       INTEGER     NOT NULL,           -- signed: positive OR negative (D-10)
  legacy_key  TEXT,                            -- 'good_behavior' for the seeded default, else NULL
  sort_order  INT         NOT NULL DEFAULT 0,
  is_active   BOOLEAN     NOT NULL DEFAULT true,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CONSTRAINT behavior_tags_legacy_key_check CHECK (legacy_key IS NULL OR legacy_key = 'good_behavior')
);

CREATE TABLE IF NOT EXISTS public.behavior_marks (
  id           UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  child_id     TEXT        NOT NULL REFERENCES public.children(id) ON DELETE CASCADE,
  family_id    UUID        NOT NULL REFERENCES public.families(id) ON DELETE CASCADE,
  date         DATE        NOT NULL,
  tag_id       UUID        NOT NULL REFERENCES public.behavior_tags(id) ON DELETE CASCADE,
  status       TEXT        NOT NULL DEFAULT 'approved'  -- 'pending' | 'approved' | 'rejected' (D-12)
               CHECK (status IN ('pending','approved','rejected')),
  proposed_by  TEXT,   -- family_members.id or 'parent' sentinel — planning detail
  decided_by   TEXT,
  decided_at   TIMESTAMPTZ,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
  -- NOTE: unlike room_checks, this is NOT a UNIQUE(child_id,date,tag_id) — D-11 allows
  -- "multiple tags per child per day" as a SUM, and even the SAME tag could plausibly
  -- apply twice in one day (e.g. "Helped a sibling" twice) — confirm this is intended
  -- multiplicity before locking the schema; see Open Questions.
);
```
Default-tag seed mirrors `seed_default_room_tasks`: one row, `legacy_key='good_behavior'`,
price `+5`, `status` n/a (definition table has no status — only marks do).

### Anti-Patterns to Avoid
- **Re-implementing preset logic per-screen:** Onboarding and Settings must call the *same*
  `getPresetValues()` — do not hand-roll the value table twice (drift risk already flagged by
  D-01's "keep two cards, one value set" instruction).
- **Client-side wallet_settings writes:** `completeOnboarding()`'s existing direct browser-client
  upsert to `wallet_settings` is **already broken** by the 04.4-03 RLS lock — do not carry this
  pattern into the new preset-writing code. Every `wallet_settings` write, from onboarding or
  Settings, must go through `/api/wallet/settings` PATCH (parent-guarded, service-role).
- **Trusting client-supplied `grade_coin_map` shape blindly:** the existing PATCH route spreads
  `...safe` from the request body directly into the upsert with no per-field validation beyond
  the streak clamp. A `grade_coin_map` value must be defensively validated server-side (numeric
  values, sane bounds, keys matching the family's `grade_scale`) — see Common Pitfalls.
- **Numeric grade casts anywhere in the new path:** D-06 is explicit — no `Number(grade)` /
  lossy A→number mapping. Every place `subject_grades.grade` is read must treat it as an opaque
  string key into `grade_coin_map`, not as an orderable/comparable number (this breaks the
  existing badges/streaks grade<=2 / grade===5 comparisons for non-five-point families — D-07
  explicitly accepts this as out-of-scope, but the type change itself, `number → string`, will
  cause **TypeScript compile errors** at every comparison site listed below regardless of scale,
  so those call sites must be visited even though their *logic* stays 5-point-only).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Family-configurable list + per-day marks | A new one-off schema shape for behavior | Copy `room_tasks`/`room_checks` (05.2-01) structure verbatim, add only `status`/`proposed_by`/`decided_by`/`decided_at` | Third application of an already-proven, already-reviewed pattern in this codebase — inventing a new shape adds risk with zero benefit |
| Request→approve→credit money flow | A bespoke behavior-approval state machine | Mirror `reward_purchases`' `pending`→`approved`/`rejected` status + `app/parent/shop/actions.ts` approve/reject server actions | Same money-crediting shape (coins only move on approval, service-role only), already battle-tested with audit-event logging and push-notification hooks |
| Diff-preview computation | A generic "settings diff" library | Plain old→new field comparison in a component (small, fixed field list per D-02) | Scope is ~13 known numeric fields; a generic diffing library is unjustified complexity for this size |

**Key insight:** every piece of this phase already has a structural precedent shipped
somewhere in this codebase (Room Tasks for config+marks, shop purchases for propose/approve,
PeriodsManager for "apply a preset" UX, day_blocks JSONB fields for variable-length config).
The research task was locating those precedents, not inventing new architecture.

## Common Pitfalls

### Pitfall 1: `subject_grades.grade` Type Change Breaks TypeScript at Every Comparison Site
**What goes wrong:** Changing `grade: number` → `grade: string` in `SubjectGrade`
(`lib/models/child.types.ts`) will fail to compile everywhere `grade` is compared numerically
or arithmetic is done on it.
**Why it happens:** `grade` is read as a number in at least: `lib/services/badges.service.ts`
(`grade <= 2`, `grade === 5`, `grade === 5` count query filter `.eq('grade', 5)`),
`lib/services/streaks.service.ts` (grade-day-set logic, not value-comparing but typed via the
same row shape), `app/api/wallet/award/route.ts` (`gradeCoins(settings, g.grade)` switch),
`components/DailyModal.tsx` (`gradeCoinsFor`, badge/preview logic), `components/kid/
KidDayFillForm.tsx` (`GradeEntry.grade: number | null`, coinsPreview switch), and
`components/BulkModal.tsx` (`updateGrade(subjectIndex, date, grade: number)`, hardcoded
`grade >= 2 && grade <= 5` validation, hardcoded color-by-grade UI).
**How to avoid:** Enumerate every call site above as an explicit task; for badges/streaks
(D-07, explicitly 5-point-only and out of scope for *behavior* change) the fix is
**type-only** — cast/compare against the string literals `'1'..'5'` so the file compiles, no
logic change. For the award route and both day-fill previews, the fix is the *actual*
`grade_coin_map` lookup (Pattern 2 above). `BulkModal.tsx` belongs to a legacy single-family
page (`/records` or `/dashboard` per CLAUDE.md's Routing section) — confirm at planning time
whether it is in scope for this phase or can be left as a documented 5-point-only legacy
surface (mirrors the D-07 precedent for badges/streaks).
**Warning signs:** `npx tsc --noEmit` failing on `grade` arithmetic/comparison after the model
type changes — treat every such error as a required call site, not noise to suppress.

### Pitfall 2: `/api/wallet/settings` PATCH Has No Field Allowlist or Value Validation
**What goes wrong:** The current PATCH handler destructures `id`/`family_id`/`updated_at` out
and spreads *everything else* (`...safe`) directly into the `wallet_settings` upsert, with only
`clampStreakFields()` as defense-in-depth for streak keys. A malicious or buggy client PATCH
could write an arbitrary `grade_coin_map` shape (wrong keys, non-numeric values, absurd
magnitudes) directly to the row a parent controls.
**Why it happens:** The route was built when `wallet_settings` had a small, fixed set of flat
numeric columns; JSONB introduces a new class of shape risk this route wasn't designed for.
**How to avoid:** Add an explicit `grade_coin_map` validator (same defense-in-depth spirit as
`clampStreakFields`) before the upsert: verify it's a plain object, every value is a finite
number within the existing streak-style clamp bound (e.g. `[-100000, 100000]`), and — if
practical — that its keys match the family's declared `grade_scale`'s valid value set. The
*award route* already has its own authoritative clamp-free lookup (`grade_coin_map[grade] ?? 0`,
naturally 0 for an unknown key), so a malformed map can't mint unbounded coins server-side, but
validating at write-time avoids storing garbage that confuses the Settings UI.
**Warning signs:** A parent's grade-scale row list showing `NaN` or absurd numbers after a
manual PATCH; the award route silently awarding 0 for a grade because the stored key doesn't
match the literal the child entered.

### Pitfall 3: `completeOnboarding()`'s Direct Client Write to `wallet_settings` Is Already Broken
**What goes wrong:** `lib/onboarding-api.ts`'s `completeOnboarding()` calls
`createClient().from('wallet_settings').upsert(...)` from the browser. Since migration
`04.4-03` locked all money tables (including `wallet_settings`) to SELECT-only for
`authenticated` clients, this INSERT/UPDATE is denied by RLS — Postgres returns an error, which
`completeOnboarding` re-throws, which `handleFinish()` in `app/onboarding/v2/page.tsx` catches
and surfaces as an onboarding-blocking error message.
**Why it happens:** This code predates the 04.4-03 security hardening and was never updated
when the RLS lock landed — likely dead/broken since that migration, independent of this phase.
**How to avoid:** Since D-04 already requires reworking `completeOnboarding`'s signature (preset
selection instead of manual coin fields), use this as the opportunity to also route the write
server-side — either a new dedicated route (e.g. `POST /api/onboarding/complete-settings`) or
by having the onboarding wizard call the existing `PATCH /api/wallet/settings` once the family
exists and the user is authenticated as its parent (verify `requireParent()`'s membership
lookup succeeds at that point in the flow — it should, since `createFamily()` has already run).
**Warning signs:** If this is not fixed, onboarding will continue to fail after `createFamily`
+ `createChildWithWallet` succeed but before the invite-code screen shows — confirm via a live
prod signup or unit test before considering this phase done, since it may already be an
undiscovered production defect worth flagging to the user regardless of whether 5.9 assigns it
a plan.

### Pitfall 4: Day-Blocks `behavior` Legacy-Key Flat-Price Mapping Will Silently Diverge
**What goes wrong:** `lib/day-blocks.ts`'s `LEGACY_PRICE_SETTINGS_KEY.behavior:
'coins_per_good_behavior'` is read by the flag-ON path of `/api/wallet/award/route.ts`
(`resolveBlockPrice(behaviorBlock, settings) ?? settings.coins_per_good_behavior`) to price the
single boolean `day.good_behavior` flag, scaled by the day-type multiplier. Once behavior
becomes a per-tag-priced sum (D-11), this flat single-price resolution no longer makes sense —
each tag now carries its own price, and it's unclear whether the day-type multiplier should
still apply per-tag, or whether the `behavior` day_block's visibility gate (is it shown today?)
should still gate whether tags are creditable at all.
**Why it happens:** Phase 5.6 (day-blocks) and this phase were designed to layer on the same
"behavior" concept from two different angles, and CONTEXT.md's code_context section flags this
exact seam as unresolved ("behavior tags need to stay consistent with (or explicitly supersede)
this mapping so Phase 5.6's day-blocks feature flag branch doesn't regress").
**How to avoid:** Treat this as a required planning decision, not an afterthought — the planner
must explicitly decide (a) does the flag-ON day-blocks path's `behavior` block visibility gate
still apply to whether `behavior_marks` are creditable that day, and (b) does the day-type
multiplier apply per-mark. Whatever is decided, the flag-OFF path (D-09 byte-parity requirement
from Phase 5.6) is the simpler one to convert first: replace the single
`day.good_behavior && settings.coins_per_good_behavior !== 0` check with a loop over that day's
`approved` `behavior_marks`, summing `tag.price` per mark as its own `AwardIntent` (mirrors
the `extra_activities`/`activity_logs` per-item summation shape already in the same file, D-11).
**Warning signs:** Flag-ON families seeing behavior coins computed differently (or not at all)
compared to flag-OFF families after this phase ships — this is exactly the kind of drift Phase
5.6's byte-parity discipline was designed to prevent.

### Pitfall 5: `behavior_marks` Multiplicity Is Underspecified
**What goes wrong:** Room Tasks' `room_checks` has `UNIQUE(child_id, date, task_id)` — one row
per task per day. D-11 says "multiple tags can apply per child per day" (different tags), but
it's ambiguous whether the *same* tag can apply more than once on the same day (e.g. "Helped a
sibling" credited twice for two separate acts of help).
**Why it happens:** CONTEXT.md doesn't explicitly answer this — D-11 only confirms
cross-tag summation, not same-tag repeatability.
**How to avoid:** Flag as an explicit planning decision (see Open Questions) rather than
silently picking `UNIQUE(child_id, date, tag_id)` (which would forbid repeats) or no
uniqueness constraint at all (which would allow unlimited repeats) without surfacing the
tradeoff.

// app/api/wallet/_lib.ts — additive to the existing SETTINGS_DEFAULTS object
const SETTINGS_DEFAULTS = {
  // ...existing fields unchanged...

export async function loadSettings(admin: Admin, familyId: string) {
  const { data } = await admin.from('wallet_settings').select('*').eq('family_id', familyId).maybeSingle()
  const merged = { id: 'default', family_id: familyId, ...SETTINGS_DEFAULTS, ...(data ?? {}) }
  if (!merged.grade_coin_map) {
    merged.grade_coin_map = {
      '5': merged.coins_per_grade_5, '4': merged.coins_per_grade_4, '3': merged.coins_per_grade_3,
      '2': merged.coins_per_grade_2, '1': merged.coins_per_grade_1,
    }
  }
  return merged
}

### Behavior award computation (flag-OFF path, mirrors extra_activities summation shape)
```typescript
// app/api/wallet/award/route.ts — replaces the single day.good_behavior check
if (member.role !== 'child') { // parent-only crediting trigger, same guard as today
  const { data: marks } = await admin
    .from('behavior_marks')
    .select('id, tag_id, status')
    .eq('child_id', childId).eq('date', date).eq('status', 'approved')
  if (marks && marks.length > 0) {
    const tagIds = Array.from(new Set(marks.map(m => m.tag_id)))
    const { data: tags } = await admin.from('behavior_tags').select('id, name, icon, price').in('id', tagIds)
    const tagById = new Map((tags ?? []).map(t => [t.id, t]))
    for (const m of marks) {
      const tag = tagById.get(m.tag_id)
      if (!tag || tag.price === 0) continue
      intents.push({ coins: tag.price, description: tag.name, icon: tag.icon ?? '⭐', sourceType: 'behavior', sourceId: m.id })
    }
  }
}
```

## Open Questions

> **Status: all four questions RESOLVED during planning.**
> - OQ1 (day-blocks behavior seam) — RESOLVED, see Plan 05.9-06: uniform sum-over-approved-marks on both flag paths, no per-mark day-type multiplier, no visibility gate on creditability; zero-marks legacy fallback reproduces each path's prior behavior.
> - OQ2 (same-tag repeatability) — RESOLVED, see Plan 05.9-01: no `UNIQUE(child_id, date, tag_id)` constraint; marks are repeatable events, each its own idempotency slot.
> - OQ3 (parent review placement) — RESOLVED, see Plan 05.9-07: a dedicated `BehaviorApprovalQueue` surface (mirrors shop purchase-approval), not folded into DailyModal.
> - OQ4 (grade_coin_map backfill) — RESOLVED, see Plan 05.9-04: read-time fallback in loadSettings/getWalletSettings, no migration backfill UPDATE.
>
> Original analysis retained below for reference.


1. **Does the day-blocks (Phase 5.6) `behavior` block's visibility gate and day-type multiplier
   still apply to the new tag-sum model?**
   - What we know: the flag-OFF path (Phase 5.6 byte-parity requirement) is simpler to convert
     first — just replace the boolean check with a sum-over-approved-marks loop.
   - What's unclear: whether the flag-ON path's `resolveBlockPrice`/`resolveMultiplier` calls
     should be reapplied per-tag-mark, or whether the day_blocks `behavior` block's *visibility*
     (shown/hidden today) should still gate whether any tag can be credited that day at all.
   - Recommendation: resolve this explicitly during planning (see Pitfall 4) — do not leave it
     to implementation-time improvisation, since it directly affects whether flag-ON families'
     behavior coins are computed consistently with flag-OFF families.

2. **Can the same behavior tag apply more than once to the same child on the same day?**
   - What we know: D-11 confirms *different* tags sum; Room Tasks' analogous table
     (`room_checks`) has a `UNIQUE(child_id, date, task_id)` constraint (one row per task per
     day), but Room Tasks is a threshold/checklist model, not a repeatable-event model.
   - What's unclear: CONTEXT.md doesn't explicitly answer same-tag repeatability.
   - Recommendation: ask the user/planner to confirm intent before finalizing the
     `behavior_marks` schema's uniqueness constraint (or deliberate absence of one).

3. **Where exactly does the parent review pending child-proposed behavior tags?**
   - What we know: CONTEXT.md explicitly leaves this to "Claude's Discretion" — new dedicated
     queue vs. folded into the existing `DailyModal` behavior section.
   - What's unclear: which is a better UX fit given the existing DailyModal's per-child, per-day
     framing vs. a cross-child, cross-day pending-approvals inbox (which shop-purchase approval
     already uses, per `app/kid/(app)/shop/page.tsx`'s pending-status pattern referenced in the
     UI-SPEC).
   - Recommendation: the UI-SPEC already anchors the copy/visual pattern
     ("Approval queue" heading/empty-state copy) — this reads as leaning toward a dedicated
     queue (mirroring the shop purchase-approval surface) rather than folding into DailyModal;
     confirm with the user if ambiguity remains at plan time.

4. **Does `grade_coin_map` need backfilling from a family's already-customized
   `coins_per_grade_1..5` values, or is read-time fallback (Pattern in Code Examples above)
   sufficient?**
   - What we know: CONTEXT.md explicitly defers this to "Claude's Discretion" / planning.
   - What's unclear: whether a one-time backfill UPDATE in the migration is preferable to an
     eternal read-time fallback in `loadSettings`.
   - Recommendation: the read-time fallback shown in Code Examples is simpler, always-correct
     (works for both existing and brand-new families), and avoids a data migration entirely —
     recommend this over a backfill unless the planner identifies a reason a stored, queryable
     `grade_coin_map` is needed sooner (e.g. if the Settings UI reads `grade_coin_map` directly
     without going through `loadSettings`'s fallback logic — confirm the client-side
     `getWalletSettings()` equivalent in `lib/repositories/wallet.repo.ts` gets the same
     fallback treatment, since it's a separate code path from the server's `loadSettings`).

|---------|--------|----------------------|
| Child self-approves own proposed behavior tag to mint coins | Elevation of Privilege | Server-side role check in the approval route/action: `member.role === 'parent'` (or extended-with-permission) required, mirroring `requireParent()`; never trust a client-supplied "approved by" field |
