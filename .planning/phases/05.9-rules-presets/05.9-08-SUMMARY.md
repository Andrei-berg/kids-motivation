## Accomplishments

- `KidDayFillForm.tsx` grade input: `GradeEntry.grade` is now `string | null`; the grade-button grid maps over `GRADE_SCALE_VALUES[grade_scale]` (fallback `five_point`) instead of a hardcoded `[2,3,4,5]` block; `coinsPreview` sums `grade_coin_map[grade] ?? 0` (mirroring the award route's own lookup) instead of a numeric 1-5 switch; both the unsaved-entry button coloring and the saved-entry badge coloring are sign-based (`coins > 0` â†’ teal, `< 0` â†’ danger, `0` â†’ neutral) rather than a numeric threshold, so all three scales (five_point/twelve_point/a_f) render correctly. - `KidDayFillForm.tsx` behavior section: a new always-visible "Behavior" `BlockCard` (parallel to Mood) renders the family's active `behavior_tags` as toggle chips (`Tick` selected-state, matching the grade-button click-to-set idiom); a tag with an existing `pending` mark renders `StatusChip tone="warning" theme="paper"` "Waiting for parent" instead of being selectable; selecting one or more unmarked tags reveals a "Propose a tag" button that calls `proposeMark` per selected tag (client-writable, RLS forces `status='pending'`) and appends the returned marks to local state. Coins are never added to `coinsPreview` for proposed/pending marks.
- D-09 divergence documented inline (multi-line code comment at the state declaration and again above `behaviorBody`) and here: this form intentionally never writes `days.good_behavior` â€” behavior credit is parent-gated; the legacy dual-write/fallback lives entirely on the parent's `DailyModal` (plan 07) and the award-route's no-approved-mark fallback (plan 06).
- `BulkModal.tsx`: `SubjectRow.grades`, `updateGrade`, and `fillAll` now operate on string grade literals (`'2'`..`'5'`) instead of numbers; the hardcoded range check became `['2','3','4','5'].includes(grade)`; the color-by-grade UI is otherwise unchanged (still 5-point-only, per A1/D-07). A one-line comment marks this as the documented legacy compile-fix.
