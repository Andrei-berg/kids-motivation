<objective>
De-hardcode the award computation: replace the numeric grade switch with a `grade_coin_map[grade]` lookup at both call sites, and replace the single `day.good_behavior` boolean with a sum over that day's `approved` behavior_marks — while KEEPING `day.good_behavior` as the legacy fallback when a child+date has no APPROVED behavior_marks (D-09, the exact 05.2-03 Room Tasks pattern). Resolve the numeric→string `grade` type break in badges/streaks (type-only, D-07 keeps them 5-point-only). Cover it with integration tests.

RESOLVED Open Question 1 (Pitfall 4 — day-blocks behavior seam): behavior tags credit UNIFORMLY in the flag-OFF and flag-ON paths as a sum-over-approved-marks, each mark contributing exactly its `tag.price`. NO day-type multiplier is applied per mark, and the flag-ON `behavior` day_block's visibility gate does NOT gate whether marks are creditable. Rationale: a behavior mark is an explicit signed assessment (a parent's judgement or an approved child proposal), not a schedule-derived work slot — multiplying "Rude to a parent −10" by a vacation multiplier is semantically wrong, and running both paths through one loop preserves Phase 5.6's byte-parity discipline. The no-approved-mark LEGACY FALLBACK, by contrast, reproduces each path's EXISTING behavior verbatim (flag-OFF: flat `coins_per_good_behavior`; flag-ON: `resolveBlockPrice × resolveMultiplier`).

Purpose: The authoritative money computation for grades and behavior, with zero behavior change for the default/legacy case.
Output: Award route + day-blocks reconciliation + service type fixes + tests.
</objective>
