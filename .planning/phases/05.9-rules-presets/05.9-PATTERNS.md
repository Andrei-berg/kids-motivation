### `supabase/migrations/05.9-03-behavior-tags.sql` (migration)

**Analog:** `supabase/migrations/05.2-01-room-tasks.sql` (full file read — 221 lines)

This is a byte-for-byte structural template. Copy verbatim and adapt names:

**Table shape** (lines 13-51 of the analog):
```sql
CREATE TABLE IF NOT EXISTS public.room_tasks (
  id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  family_id   UUID        NOT NULL REFERENCES public.families(id) ON DELETE CASCADE,
  name        TEXT        NOT NULL,
  icon        TEXT,
  legacy_key  TEXT,
  sort_order  INT         NOT NULL DEFAULT 0,
  is_active   BOOLEAN     NOT NULL DEFAULT true,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CONSTRAINT room_tasks_legacy_key_check CHECK (
    legacy_key IS NULL OR legacy_key IN ('bed', 'floor', 'desk', 'closet', 'trash')
  )
);
CREATE INDEX IF NOT EXISTS room_tasks_family_id_idx ON public.room_tasks(family_id);
CREATE UNIQUE INDEX IF NOT EXISTS room_tasks_family_legacy_key_idx
  ON public.room_tasks (family_id, legacy_key) WHERE legacy_key IS NOT NULL;

CREATE TABLE IF NOT EXISTS public.room_checks (
  id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  child_id    TEXT        NOT NULL REFERENCES public.children(id) ON DELETE CASCADE,
  family_id   UUID        NOT NULL REFERENCES public.families(id) ON DELETE CASCADE,
  date        DATE        NOT NULL,
  task_id     UUID        NOT NULL REFERENCES public.room_tasks(id) ON DELETE CASCADE,
  done        BOOLEAN     NOT NULL DEFAULT false,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CONSTRAINT room_checks_child_date_task_unique UNIQUE (child_id, date, task_id)
);
```
For `behavior_tags`/`behavior_marks`: same shape, `price INTEGER NOT NULL` (signed, per D-10) instead of a bare checklist boolean, and `behavior_marks` gets `status TEXT NOT NULL DEFAULT 'approved' CHECK (status IN ('pending','approved','rejected'))` + `proposed_by`/`decided_by`/`decided_at` (per D-12) — and per the research's Pitfall 5/Open Question 2, decide the uniqueness constraint deliberately rather than copying `room_checks_child_date_task_unique` unreflectively (D-11 requires cross-tag summation; same-tag repeatability is unresolved).

**RLS policy block** (lines 61-156 of the analog) — copy verbatim, four policies (SELECT/INSERT/UPDATE/DELETE) per table, family-isolation subquery pattern:
```sql
CREATE POLICY "room_tasks_select_own_family"
  ON public.room_tasks
  FOR SELECT
  USING (
    family_id IN (
      SELECT family_id FROM public.family_members WHERE user_id = auth.uid()
    )
  );
```
Note per the Security Domain in RESEARCH.md: `behavior_marks` INSERT policy needs an *additional* constraint beyond straight family-isolation — a child-authored insert must not be able to set `status='approved'` directly (RLS is the actual boundary here, not app-layer discipline). This is a deviation from the room_checks analog that must be added, e.g. via a `WITH CHECK` clause keying off `auth.jwt()`/membership role, or a trigger.

### `supabase/migrations/05.9-01-wallet-settings-grade-scale.sql` and `05.9-02-subject-grades-widen.sql` (migration)

**Analog:** `supabase/migrations/04.4-03-wallet-rls-readonly.sql` (full file read — 66 lines) for the money-table RLS-scoping template, and the additive-column style used throughout `supabase/migrations/` (e.g. `05.2-01` adding new columns/tables without touching existing ones).

Key point from `04.4-03`: `wallet_settings` is **already RLS SELECT-only** for clients (this migration installed that lock) — the new `grade_scale`/`grade_coin_map` columns do NOT need a new RLS migration, they inherit the existing SELECT-only policy on `wallet_settings` automatically. `05.9-01` only needs `ALTER TABLE public.wallet_settings ADD COLUMN IF NOT EXISTS grade_scale TEXT NOT NULL DEFAULT 'five_point' CHECK (grade_scale IN ('five_point','twelve_point','a_f'))` and `ADD COLUMN IF NOT EXISTS grade_coin_map JSONB` — no policy changes.

`05.9-02` (widen `subject_grades.grade` to TEXT) is a schema change on a **non-money** table (`subject_grades` uses ordinary family-isolation RLS, confirmed live via `pg_policies` per RESEARCH.md Sources) — no RLS template needed there, just:
```sql
ALTER TABLE public.subject_grades ALTER COLUMN grade TYPE TEXT USING grade::text;
ALTER TABLE public.subject_grades DROP CONSTRAINT IF EXISTS subject_grades_grade_check;
```
(Constraint name must be confirmed against the live DB — RESEARCH.md flags the live CHECK as `INTEGER NOT NULL CHECK (grade >= 2 AND grade <= 5)`, exact constraint name not captured in RESEARCH.md; `\d subject_grades` or `information_schema.check_constraints` lookup needed before finalizing this migration.)

### `app/parent/behavior/actions.ts` (NEW — behavior mark approve/reject)

**Analog:** `app/parent/shop/actions.ts` (full file read — 173 lines)

**Guard + fetch + status-check pattern** (`approvePurchaseAction`, lines 42-54):
```typescript
export async function approvePurchaseAction(purchaseId: string): Promise<RewardPurchase> {
  const member = await requireParent()
  const admin = createAdminClient()
  const { data: purchase, error: fetchError } = await admin
    .from('reward_purchases').select('*').eq('id', purchaseId).eq('family_id', member.familyId).maybeSingle()
  if (fetchError || !purchase) throw new Error('Purchase not found')
  if (purchase.status !== 'pending') throw new Error('Purchase is not pending')
  // ... money mutation via adjustCoins/wallet_apply, then update status
}
```
This is the exact shape `approveBehaviorMark(markId)`/`rejectBehaviorMark(markId, note?)` should follow: `requireParent()` guard (never let a child self-approve — Security Domain V4 in RESEARCH.md), fetch-by-id scoped to `member.familyId`, check `status === 'pending'` before transitioning, then update `status`/`decided_by`/`decided_at`.

Money-mutation difference: shop's `adjustCoins` (lines 11-40) does an immediate wallet delta at approve-time because purchases pre-freeze/deduct coins. Behavior marks are simpler — approval does **not** itself move coins; it only flips `status` to `'approved'` so the *next* `/api/wallet/award` POST picks it up via the sum-over-approved-marks loop (idempotent, per source_id = mark id). Do not port `adjustCoins`/`applyWalletDelta` into the behavior-approval action — that would double-credit against the award route's own crediting. This is a deliberate divergence from the shop-approval analog, not an oversight.

**Push notification + audit-event side effects** (lines 81-101) — same optional, best-effort pattern (wrapped in try/catch, `void insertAuditEvent(...)` fire-and-forget) is worth reusing for behavior mark approve/reject notifications to the child.

**Reject/refund shape** (lines 106-173) is the template for `rejectBehaviorMark` — no coins to refund here (nothing was credited yet, since it was pending), so this is simpler than `rejectPurchaseAction`: just flip `status` to `'rejected'`, set `decided_by`/`decided_at`, notify child, done.
