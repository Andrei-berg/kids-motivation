# Phase 05.9 Plan 06: Award Computation De-Hardcoding (grade_coin_map + behavior-mark sum) Summary

**`/api/wallet/award` now looks up `grade_coin_map[grade]` (any scale — five_point/twelve_point/a_f) instead of a numeric switch, and behavior coins are a sum over that day's APPROVED `behavior_marks` (each mark its own idempotent source), with the legacy `day.good_behavior` credit preserved as a fallback whenever no approved mark exists — verified by 15 new/extended live-DB integration tests.**

## Accomplishments
- `gradeCoins()` in `app/api/wallet/award/route.ts` replaced its 5-case numeric switch with `s.grade_coin_map?.[grade] ?? 0` at both call sites (flag-OFF grade loop, flag-ON `clampBlockCoins(gradeCoins(...) * mult)`), driven entirely by the family's configured scale.
- New shared helper `pushBehaviorMarkIntents(admin, childId, date, member, intents)` sums that date's `status='approved'` `behavior_marks` (joined to `behavior_tags` for price/name/icon), pushing one `AwardIntent` per mark keyed by the mark's own id — so multiple marks per day each get an independent idempotency slot — and returns whether any approved mark exists.
- Both the flag-OFF and flag-ON award paths call the SAME helper uniformly (no per-mark day-type multiplier, block visibility never gates mark creditability — resolved Open Question 1) and fall back to the existing `day.good_behavior` legacy branch, UNCHANGED, whenever no approved mark exists for the date (zero rows, pending-only, or rejected-only).
- `lib/day-blocks.ts` gained a comment on `LEGACY_PRICE_SETTINGS_KEY.behavior` clarifying it now prices only the no-approved-mark fallback in the flag-ON path.
- `badges.service.ts` and `streaks.service.ts` compile cleanly against `subject_grades.grade` as TEXT: string-literal comparisons (`'5'`, `['1','2']`) replace numeric ones, `.eq('grade', 5)` → `.eq('grade', '5')`, each site carrying a D-07 5-point-only comment; `streaks.service.ts`'s `day.good_behavior` read is annotated as a Phase 5.9 known gap.
- `tests/integration/behavior-award.test.ts` (new, 8 tests): approved-sum across two tags including a negative-price tag (with idempotent re-POST), pending=0, same-tag-twice (resolved OQ2), the D-09 legacy fallback for zero-row/pending-only/rejected-only dates, no-double-credit when an approved default-tag mark and `good_behavior` both exist, and teardown.
- `tests/integration/award.test.ts` extended with one new test proving `grade_coin_map['11']` (twelve_point) and `grade_coin_map['B+']` (a_f) drive the award exactly, and that the literal grade value round-trips through insert→read without lossy numeric conversion.

## Known Gaps

1. **D-07 (grade scale):** `badges.service.ts` and `streaks.service.ts`'s grade-driven logic (the "Идеальная неделя"/perfect-week penalty check, the "Неделя отличника"/week-excellent all-fives check, and the `week_excellent` progress-bar count) remain 5-point-only assumptions and are NOT generalized by this phase — grade-driven badges may misfire or never trigger for families on the 12-point or A-F scales (e.g. a family on `a_f` will never earn `week_excellent` since no grade will ever equal the string `'5'`, and the `checkPerfectWeekAuto` penalty check will never fire since no grade will ever be `'1'`/`'2'`). Candidate future-phase fix: derive "excellent"/"penalty" thresholds from `grade_coin_map`'s sign/rank (e.g. the highest-value key = "excellent", any negative-value key = "penalty") instead of hardcoded five-point literals.

2. **`days.good_behavior` legacy readers:** `lib/services/coins.service.ts` (analytics `behaviorDays` count, ~line 34, NOT modified by this plan) and `lib/services/streaks.service.ts` (the behavior streak's `day.good_behavior` read, now commented) continue reading `days.good_behavior`, which tracks ONLY the seeded default tag (kept in sync by Plan 07's dual-write, not yet implemented at the time of this plan). Families using custom (non-default) behavior tags will see these two legacy readers diverge (undercount relative to their actual behavior-mark history) — e.g. a family that disables the default tag and only uses custom tags will see their behavior streak never increment even while earning behavior coins every day via marks. Documented known gap, candidate for a future phase generalizing both readers to sum `behavior_marks` directly.
