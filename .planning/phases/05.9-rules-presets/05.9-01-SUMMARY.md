# Phase 05.9 Plan 01: Rules/Presets Schema Migrations Summary

Authored (not applied) the three foundational SQL migrations for Phase 5.9's de-hardcoding work: `wallet_settings.grade_scale`/`grade_coin_map` additive columns, a `subject_grades.grade` INTEGER→TEXT widen with numeric CHECK drop, and a `behavior_tags`/`behavior_marks` table pair cloned from Room Tasks with a propose/approve status column and a child-cannot-self-approve RLS guard.

## What Was Built

**Task 1 — `05.9-01-wallet-settings-grade-scale.sql` and `05.9-02-subject-grades-widen.sql`:**
- `05.9-01`: `ALTER TABLE wallet_settings ADD COLUMN IF NOT EXISTS grade_scale TEXT NOT NULL DEFAULT 'five_point' CHECK (grade_scale IN ('five_point','twelve_point','a_f'))` and `ADD COLUMN IF NOT EXISTS grade_coin_map JSONB` (nullable, no backfill). `wallet_settings` is already RLS SELECT-only for clients (migration `04.4-03`) — these columns inherit that lock automatically; no policy statement was added.
- `05.9-02`: Before writing, connected read-only to the live prod DB (via the documented `pg` + `SUPABASE_DB_URL` one-liner from CLAUDE.md, run against the main repo checkout's `.env.local` since this worktree has none — `.env.local` is git-ignored by design) and queried `pg_constraint`/`information_schema.columns` for `public.subject_grades`. Confirmed: `grade` is `INTEGER NOT NULL`, CHECK constraint name is `subject_grades_grade_check`, definition `CHECK (((grade >= 2) AND (grade <= 5)))`. Authored `ALTER COLUMN grade TYPE TEXT USING grade::text;` followed by `DROP CONSTRAINT IF EXISTS subject_grades_grade_check;`.

**Task 2 — `05.9-03-behavior-tags.sql`:**
Byte-for-byte structural clone of `05.2-01-room-tasks.sql` (room_tasks→behavior_tags, room_checks→behavior_marks, task_id→tag_id), with the plan's four specified deltas:
1. `behavior_tags.price INTEGER NOT NULL` — signed, no `CHECK (price >= 0)` (D-10).
2. `behavior_marks` replaces `done BOOLEAN` with `status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','approved','rejected'))` plus `proposed_by`/`decided_by`/`decided_at` (D-12).
3. No `UNIQUE(child_id, date, tag_id)` constraint — deliberate divergence from `room_checks_child_date_task_unique`, documented in a leading comment (resolved Open Question 2).
4. `behavior_marks_insert_own_family` policy's `WITH CHECK` clause requires `status = 'pending'` in addition to family-isolation — the DB-level boundary preventing a child-authored insert from self-approving (resolved threat T-059-03).

Also included: `seed_default_behavior_tags(p_family_id)` (seeds one row: `'Good behavior'`, `'😊'`, price `5`, `legacy_key 'good_behavior'`), a backfill `SELECT` over `public.families`, and a legacy-delete guard trigger (`pg_trigger_depth() <= 1`) mirroring the room_tasks analog.
