## Decisions Made
See `key-decisions` in frontmatter above. The two most significant:
1. Added a DB migration (`05.9-04`) extending `parent_audit_events`'s CHECK constraint — not in the plan's stated file list, but required for the plan's own instruction to call `insertAuditEvent` with an "appropriate" `action_type`; without it the audit calls would compile and run without error but never actually persist a row (the DB would silently reject every insert against the old constraint, and `insertAuditEvent` swallows errors by design).
2. DailyModal's default-tag "un-apply" (goodBehavior flips to false after an approved mark already exists for that day) does a direct client delete on `behavior_marks` rather than calling `rejectBehaviorMark`, because Task 1's reject action has (and must keep, per its own acceptance criteria) a `pending`-only precondition. This is not a money table and the call site is parent-only (DailyModal), so no service-role path was needed for this specific removal.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Added parent_audit_events CHECK constraint migration for the new action_type values**
- **Found during:** Task 1, while implementing the plan's instruction to call `insertAuditEvent({...action_type appropriate...})`
- **Issue:** `lib/repositories/audit.repo.ts`'s `action_type` union (and the live DB's `parent_audit_events_action_type_check` CHECK constraint) had no `'behavior_approve'`/`'behavior_reject'` values. Extending only the TypeScript union would compile, but every `insertAuditEvent` call from the new actions would fail the DB constraint at runtime — silently, since `insertAuditEvent` is intentionally non-throwing (logs and swallows). The audit trail for behavior approvals would be permanently empty with no visible symptom.
- **Fix:** Added `supabase/migrations/05.9-04-behavior-audit-actions.sql` (drop-and-recreate of the named constraint, mirroring the existing `05.4-04-withdraw-audit-actions.sql` precedent) and applied it to the live DB via the `pg` connection-string method documented in CLAUDE.md.
- **Files modified:** lib/repositories/audit.repo.ts, supabase/migrations/05.9-04-behavior-audit-actions.sql
- **Verification:** Migration applied (`node --env-file=.env.local -e '...'` → `applied`); `npx tsc --noEmit` clean; the live-DB integration test's approve/reject cases (which trigger these `insertAuditEvent` calls, mocked to a no-op in the test suite but exercising the same action code path) pass.
- **Committed in:** 378d6b3 (Task 1 commit)

**2. [Rule 3 - Blocking] Default-tag un-apply routed around rejectBehaviorMark's pending-only precondition**
- **Found during:** Task 2, implementing DailyModal's D-09 dual-write reconciliation
- **Issue:** The plan's action text says "when [goodBehavior is] false ensure it is removed/rejected" for the default tag, naming `rejectBehaviorMark` as one of "the service-role actions from Task 1" to use for reconciliation. But Task 1's own action text and acceptance criteria require `rejectBehaviorMark` to assert `status === 'pending'` before transitioning — an already-`approved` default-tag mark (created by an earlier save) cannot be rejected through that action without either violating Task 1's stated precondition or silently loosening it (which would also weaken the guarantee that reject always mirrors a real pending-proposal decision).
- **Fix:** Used a direct client-side `supabase.from('behavior_marks').delete()` for this one case only (un-applying an already-approved default-tag mark from the parent-only DailyModal surface). `behavior_marks` is not a money table — plain family-isolation RLS permits the delete — so no service-role escalation was required, and Task 1's reject action keeps its stated pending-only contract intact.
- **Files modified:** components/DailyModal.tsx
- **Verification:** `npx tsc --noEmit` clean; manual trace of the reconciliation branch (insert when true+missing, delete when false+existing, no-op otherwise) confirmed against `behavior_marks`' RLS DELETE policy (family-isolation only, no status restriction).
- **Committed in:** 3aee16c (Task 2 commit)
