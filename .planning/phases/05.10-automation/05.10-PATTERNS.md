**Full analog file** (`2026-06-22-reading-verification.sql`, lines 1-11):
```sql
-- Reading verification: a child's "I finished a book" claim is held for parent
-- confirmation (the kid retells the book) before the book bonus is credited.
--   reading_log.verified: NULL = pending review, TRUE = approved, FALSE = rejected.
--   children.require_reading_check: per-child flexibility — default ON for everyone
--     (people get lazy/cunning), turn OFF for a kid you trust to read.
ALTER TABLE reading_log ADD COLUMN IF NOT EXISTS verified BOOLEAN;
ALTER TABLE children ADD COLUMN IF NOT EXISTS require_reading_check BOOLEAN DEFAULT true;

-- Grandfather existing finished-book entries as verified: they were already credited
-- under the old trust model, so parents shouldn't get a retroactive verification queue.
UPDATE reading_log SET verified = true WHERE book_finished = true AND verified IS NULL;
```

**Shape to copy for this phase:**
```sql
-- Per-child trust limit (D-01/D-04): 0 = disabled/off by default (opt-in only).
ALTER TABLE children ADD COLUMN IF NOT EXISTS trust_limit_coins INTEGER NOT NULL DEFAULT 0;

-- Per-child allowance (D-05/D-06/D-08): amount + period, off (NULL period) unless configured.
ALTER TABLE children ADD COLUMN IF NOT EXISTS allowance_amount INTEGER;
ALTER TABLE children ADD COLUMN IF NOT EXISTS allowance_period TEXT; -- 'weekly' | 'monthly' | NULL
ALTER TABLE children ADD COLUMN IF NOT EXISTS allowance_anchor INTEGER; -- day_of_week (1-7) for weekly, day_of_month (1-31) for monthly
```
No new idempotency index is needed — `04.4-02-wallet-tx-idempotency.sql`'s existing
`uniq_wallet_tx_source` on `(child_id, source_type, source_id)` already covers any
`source_type` value, including the new `'allowance'` (D-09).

### Per-child trust-limit/allowance server actions (new)

**Analog:** `app/parent/reading/actions.ts` lines 77-84 (`setRequireReadingCheckAction`) — the
established shape for "one per-child scalar column, one parent-guarded server action."

```typescript
'use server'

import { createAdminClient, requireParent, assertChildInFamily } from '@/lib/supabase/admin'

/** Per-child flexibility toggle: require parent verification of reading (default on). */
export async function setRequireReadingCheckAction(childId: string, value: boolean): Promise<{ ok: true }> {
  const member = await requireParent()
  const admin = createAdminClient()
  await assertChildInFamily(admin, childId, member.familyId)
  await admin.from('children').update({ require_reading_check: value }).eq('id', childId)
  return { ok: true }
}
```
Copy this exact shape for `setTrustLimitAction(childId, value: number)` and
`setAllowanceAction(childId, amount: number | null, period: 'weekly' | 'monthly' | null, anchor: number | null)`
— same `requireParent()` + `assertChildInFamily()` guard, same single-table `.update()` call,
same `{ ok: true }` return. Add server-side clamping (mirror `clampStreakFields` /
`clampGradeCoinMap` in `app/api/wallet/settings/route.ts` lines 16-63, e.g. floor at 0,
reject NaN) before the `.update()` — this codebase's established defense-in-depth idiom for
any parent-writable numeric field.

Note: unlike money-table mutations, `children.trust_limit_coins`/`allowance_*` are ordinary
(non-RLS-locked) columns, so a direct `admin.from('children').update()` from a server action
is the correct and sufficient pattern — no `wallet_apply` RPC or `wallet_transactions` insert
needed here (those are for `wallet`/`wallet_transactions` money tables specifically).

## Shared Patterns

### Cron auth guard
**Source:** `lib/cron-auth.ts` (`assertCronAuth`, full file, 39 lines)
**Apply to:** every new/modified cron route (`app/api/cron/allowance/route.ts`, any reminders
rework).
```typescript
export function assertCronAuth(request: NextRequest): NextResponse | null {
  const cronSecret = process.env.CRON_SECRET
  if (!cronSecret) {
    console.error('CRON_SECRET is not configured — rejecting request')
    return NextResponse.json({ error: 'Server misconfigured' }, { status: 500 })
  }
  const auth = request.headers.get('authorization') ?? ''
  if (!safeEqual(auth, `Bearer ${cronSecret}`)) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }
  return null
}
// Usage: const denied = assertCronAuth(request); if (denied) return denied
```

### Service-role client in cron context
**Source:** `app/api/cron/missed-tasks/route.ts` line 19, comment on lines 17-18.
**Apply to:** every cron route, and any reminder logic touching `streaks` (RLS SELECT-only).
```typescript

// not silently filtered to zero rows by the "TO authenticated" RLS policies.

```
**CAVEAT (flagged explicitly in CONTEXT.md canonical refs):** `notifyChild`/`notifyParent`
(`app/actions/push-notifications.ts`) internally call `createClient()` from
`lib/supabase/server.ts` — a cookie-session, RLS-bound client, NOT the admin client. In a
cron's session-less context these resolve 0 rows and silently no-op. Do NOT call
`notifyChild`/`notifyParent` from a cron route. Both `missed-tasks/route.ts` and
`reminders/route.ts` instead resolve `push_subscriptions` directly via the admin client and
call `sendPushToSubscription` (`app/actions/push.ts`) per subscription:
```typescript
import { sendPushToSubscription } from '@/app/actions/push'

sends.push(sendPushToSubscription(subJson, title, body, url))
```
This is the pattern the new allowance cron route (and any reminders rework) must reuse —
never `notifyChild`/`notifyParent` in a cron file.
