# Phase 05.10: automation - Context

**Gathered:** 2026-07-22
**Status:** Ready for planning

<domain>
## Phase Boundary

The routine runs itself: three independent automation features, continuing the
v5.0 de-hardcoding principle onto operational behavior rather than data model:

1. **Scheduled allowance** — cron credits coins per family/child schedule,
   idempotent per period.
2. **Trust-limit auto-approve** — purchases under a per-child coin threshold
   skip the parent approval queue; the parent still sees them in the journal.
3. **Schedule-driven reminders** — upcoming section training, day not filled
   by evening, streak about to expire.

No new capability beyond these three; this is HOW to implement what
ROADMAP.md already scoped for 5.10.

</domain>

<decisions>
## Implementation Decisions

### Trust-limit auto-approve (SC2)

- **D-01:** Trust limit is set **per child**, not one family-wide value —
  consistent with the roadmap's exact wording ("per-child trust limit") and
  lets a parent trust an older child with a higher auto-approve ceiling than
  a younger sibling.
- **D-02:** The new trust limit **OR-combines** with the existing per-reward
  `rewards.auto_approve` flag in `processPurchase()`
  (`app/api/wallet/_lib.ts`): a purchase auto-approves if EITHER the reward
  itself is flagged `auto_approve` OR `priceCoins <= child's trust limit`.
  No new tri-state / "always manual" override on rewards — additive only,
  nothing existing changes behavior.
- **D-03:** Auto-approvals via the trust limit must be **visually
  distinguishable** in the parent's purchase journal from auto-approvals via
  the existing reward flag — the parent should be able to tell *why* a
  purchase skipped review (e.g. "Auto-approved — under trust limit" vs.
  "Auto-approved — instant reward"). Needs a new `processed_by` value
  distinct from today's `'auto'` (exact value/label left to planning).
- **D-04:** Default trust limit for families who never touch this setting is
  **0 / disabled** — no purchase auto-approves via trust limit until a
  parent explicitly opts in and sets a value > 0. Matches this codebase's
  pattern of shipping new automation opt-in (day-blocks feature flag,
  presets), not on-by-default for anything money-adjacent.

### Allowance rules (SC1)

- **D-05:** Allowance amount and period are configured **per child**, not one
  family-wide schedule — same reasoning as D-01 (per-child granularity,
  partial adoption doesn't force an all-or-nothing family setting).
- **D-06:** Supported periods: **weekly and monthly**. Weekly needs a
  day-of-week anchor; monthly needs a day-of-month anchor. (Weekly-only was
  the fallback if monthly proved too complex to reason about for
  idempotency — it did not, so both ship.)
- **D-07:** A child added mid-period gets **no proration** — they simply
  aren't eligible for that period's allowance and receive the full amount
  starting the next scheduled date. No partial-period math to build or test.
- **D-08:** No pause/skip-single-cycle control in this phase — allowance is
  **on/off only** (configured with amount+period, or not configured at all).
  A parent who wants to withhold allowance for one cycle does it via a
  manual coin adjustment after the fact. Keeps the feature's scope tight per
  the roadmap's "needs no parent action" framing — a pause UI is itself a
  parent action to build and maintain.
- **D-09 (idempotency, follows from scouting):** Allowance credits reuse the
  existing `wallet_transactions(child_id, source_type, source_id)` unique-index
  idempotency mechanism (`04.4-02-wallet-tx-idempotency.sql` +
  `creditAwards()` in `app/api/wallet/_lib.ts`) — `source_type='allowance'`,
  `source_id` derived from the period key (e.g. `2026-W29` for weekly,
  `2026-07` for monthly). This is the same idempotency shape every other
  award source already uses; no new mechanism needed.

### Claude's Discretion (explicitly deferred, not decided)

- **Cron cadence / Vercel Pro tradeoff (SC3, reminders):** the codebase
  already has a fully-implemented `/api/cron/reminders` route
  (`app/api/cron/reminders/route.ts`) that is unwired from `vercel.json`
  because its 5-minute cadence needs Vercel Pro (Hobby allows 1 run/day per
  cron). The user did not decide between upgrading to Pro vs. redesigning
  reminders around coarser (daily/hourly-if-possible) cadence — this is a
  **real cost/infra decision**, not a technical implementation detail.
  Research should investigate current Vercel plan/limits and present both
  paths; **planning must surface this choice back to the user explicitly**
  before committing to an implementation shape for SC3 — do not silently
  pick Pro-upgrade or silently redesign around Hobby limits.
- **Reminder scope & timing (SC3):** which of the three reminder types ship,
  what "evening" means as a cutoff (fixed time vs. family-configurable), and
  whether reminders go to the child, the parent, or both, were not decided —
  left open pending the cadence decision above, since the answer partly
  depends on what cadence is actually available. Note per scouting:
  "streak about to expire" has no ready-made helper — inferring it requires
  factoring a forward-looking "is today's contribution filled yet" check out
  of the existing backward-walking streak calculators
  (`lib/services/streaks.service.ts`), which none of the four streak types
  (room/study/sport/strong_week) currently expose.
- Exact `processed_by` value/label for trust-limit auto-approvals (D-03).
- Exact UI/copy for where the per-child trust limit and allowance
  amount/period get configured (likely Parent Center → Settings, following
  the existing per-child settings card patterns — e.g. streaks, room tasks).

</decisions>

- `app/api/wallet/_lib.ts`: `SETTINGS_DEFAULTS`, `loadSettings`,
  `creditAwards()`, `processPurchase()` — extend with allowance
  amount/period fields and trust-limit field; both new features route
  through this same file's existing money-mutation helpers.

  adapt/re-enable rather than write from scratch, once the cadence decision
  (Claude's Discretion) is made.
- `app/api/cron/missed-tasks/route.ts` — template for a new allowance cron
  route (admin client, `assertCronAuth`, daily cadence is sufficient since
  the route just needs to check "is today the scheduled allowance day").

### Established Patterns
- Every award source uses the same `(child_id, source_type, source_id)`
  idempotency shape — allowance is the Nth application of this pattern, not
  a new one (D-09).
- Per-child settings stored and edited via a Settings card following the
  05.4/05.9 precedent (load → edit → PATCH → persist to `wallet_settings`
  or a per-child table).
- Money mutations are always server-side via service-role client — no
  exception for allowance or auto-approve.

### Integration Points
- `processPurchase()` — trust-limit check added alongside the existing
  `reward.auto_approve` check (D-02).
- New allowance cron route — mirrors `missed-tasks` structurally, calls
  `creditAwards()` with `source_type='allowance'`.
- `components/parent-center/ParentCenter.tsx` approval-queue rendering —
  needs the new "why auto-approved" distinction (D-03) surfaced in the
  journal list.
