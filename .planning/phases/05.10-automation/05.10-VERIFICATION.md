### Gaps Summary

**Primary blocker (SC3):** The reminders feature is fully implemented and unit/integration-tested at the code level (section-training query shape, day-not-filled detection, forward-looking streak-at-risk helper, push batching), but it cannot actually reach a single child in production today. Two tables it depends on — `push_subscriptions` (the delivery mechanism for every push in this codebase, not just this phase's reminders) and `schedule_items` (the source of the section-training check) — were never created in the production database. The migration that defines them, `supabase/migrations/01.3-categories-schedule.sql`, predates this phase (Phase 1.3) and is marked "Complete" in ROADMAP.md, but a direct query against the live Postgres instance (`to_regclass()`) confirms both tables are absent. This is independently reproducible via the phase's own live-DB integration test (`reminders-daily.test.ts`, 1/6 failing deterministically) — the executor of Plan 03 encountered this exact failure and logged it to `deferred-items.md` as an out-of-scope pre-existing issue, but did not connect it to SC3's actual non-functionality. Fixing this requires applying the missing migration to production and re-verifying end-to-end push delivery — outside the diff of any of the four 05.10 plans, but blocking for the phase goal as stated ("reminders need no parent action" — currently, reminders do not happen at all).

**Secondary gap (robustness, not blocking SC1/SC2):** The daily cron route has no `try`/`catch` anywhere, so any single exception (observed directly with a child missing its wallet row) aborts the whole run before any of that day's collected pushes are sent. This doesn't block SC1 (allowance crediting for well-formed data works and is verified) but is worth hardening before the push-delivery gap above is fixed, since the failure mode would otherwise re-surface as silent, whole-fleet reminder loss.

**Outstanding (SC2 config UI, not code-level failed but unverified):** Plan 04's human-verify checkpoint was explicitly bypassed by user decision; the Settings UI code is well-formed and type/unit-tested but has never been exercised in a real browser against the live DB. Recommend a follow-up spot-check per the five items above.

SC1 (allowance crediting) and the SC2 trust-limit auto-approve *engine* are solid: correct code, live-DB-tested, idempotent, and independent of the broken push infrastructure.
