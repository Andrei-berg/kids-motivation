## Accomplishments
- Migration `05.10-01-trust-limit-allowance.sql` adds `trust_limit_coins` (NOT NULL DEFAULT 0), `allowance_amount`, `allowance_period` (CHECK 'weekly'|'monthly', inlined), `allowance_anchor` to `children` — applied and verified live on prod (all 4 columns present; re-apply confirmed no-op).
- `processPurchase()` now OR-combines a per-child trust limit with the existing `reward.auto_approve` flag: a purchase priced at or below `trust_limit_coins` (when > 0) auto-approves with `processed_by='auto_trust'`, distinct from the existing `'auto'` marker for reward-flagged instant rewards.
- The parent journal (existing `wallet_transactions.description`, rendered in the Dashboard Activity feed) now carries a suffix (`· авто (лимит доверия)` or `· авто (мгновенная награда)`) so the parent can tell WHY a purchase skipped review, with zero UI changes required.
- Trust limit and `priceCoins` are read exclusively server-side via the admin client inside `processPurchase()` — never accepted from the request body — closing threat T-0510-01.

## Decisions Made
- `processed_by` precedence: reward-flag auto-approve always reports `'auto'` even when the trust limit would independently qualify — `'auto_trust'` is reserved for trust-only auto-approvals, so the marker always names the *most specific* applicable reason.
- Journal WHY-distinction implemented as a description-string suffix rather than a new join from the Dashboard Activity feed to `reward_purchases.processed_by` — avoids introducing new UI/query surface for a phase whose SC2 scope is the auto-approve engine itself (per 05.10-PATTERNS.md's "No Analog Found" note, this was explicitly left to planning to pick one option; the simpler no-join option was chosen).
