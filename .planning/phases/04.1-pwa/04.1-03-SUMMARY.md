## What Was Built

### notifyParent (app/actions/push-notifications.ts)

New exported function following the same pattern as `notifyChild`:
- Queries `family_members` where `family_id = X` and `role = 'parent'`
- Fetches `push_subscriptions` for all matched member IDs
- Sends to each via `sendPushToSubscription` using `Promise.allSettled`
- Never throws — all errors caught and logged as warnings

### requestPurchase server action (app/kid/shop/actions.ts)

New `'use server'` file that:
- Calls `createPurchaseRequest(childId, rewardId)` server-side
- If `purchase.status === 'pending'`: resolves `family_id` via `family_members.child_id` and calls `notifyParent`
- Push failure is completely non-blocking (try/catch)
- Returns the `RewardPurchase` object unchanged

### app/kid/shop/page.tsx

- Removed `createPurchaseRequest` from wallet.repo import
- Added `import { requestPurchase } from '@/app/kid/shop/actions'`
- `handleConfirm` now calls `requestPurchase` instead of `createPurchaseRequest`
- All other logic (error handling, state updates, loading state) unchanged

### public/sw.js

- Appended `pushsubscriptionchange` handler at the end of the file
- Logs a warning when browser rotates push credentials
- All existing handlers preserved: `push`, `notificationclick`, `install`, `activate`, `fetch`

## Decisions Made

1. **notifyParent only fires for pending-status purchases** — auto-approved purchases skip push to avoid noise for parents
2. **Push failure is non-blocking** — caught in try/catch in `requestPurchase`, purchase flow always completes regardless of push outcome
3. **pushsubscriptionchange logs warning only** — full auto-resubscription requires VAPID public key unavailable in SW context; NotificationSettings component handles re-subscription on next app open
4. **Server action composition** — `requestPurchase` must be a `'use server'` file because calling `notifyParent` (a server action) from a client component's dynamic import would not execute it server-side
