# Phase 5.8 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Nature of this phase:** rollout + two screen upgrades, not invention. The
palette/type/atom system was locked in Phase 5.3 against the approved design
contract artifact «семейный банк» v2.3
(https://claude.ai/code/artifact/ab9621cc-2f84-42ff-a873-d07f8b841715) and is
already fully specified for the **ink** (parent) theme. This UI-SPEC does not
re-litigate that palette — it (1) confirms the narrowed D-09 recolor scope
found by `05.8-RESEARCH.md`, (2) locks concrete layouts for the two new
screens (Day Constructor, Year Calendar) and the Weekly Summary card, and
(3) closes the two open design questions RESEARCH.md flagged (`ChatPanel`
recolor depth, `ui.tsx` cyan variant).

## Spacing Scale

Declared values (must be multiples of 4) — same working discipline 5.7
established, now binding for every Parent Center surface this phase touches:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, calendar grid cell gap, chip internal gaps |
| sm | 8px | Compact row spacing, `Btn`/`Pill` internal gaps |
| md | 16px | Default card padding (`Card pad={16}`), gap between stacked cards, screen-edge padding |
| lg | 24px | Section-to-section spacing (e.g. calendar month header → grid → edit affordances) |
| xl | 32px | Major screen-region breaks (e.g. above the Weekly Summary card) |
| 2xl | 48px | Empty-state illustration/message spacing |
| 3xl | 64px | Not expected — Parent Center screens are mobile-first, short viewports |

Exceptions:
- **44×44px minimum touch target** (carried over, still binding, UX-02):
  every calendar grid day-cell, every Day Constructor row control (move
  up/down, toggle-active, delete, "Link to schedule", "Override for…"),
  every `Tabs`/child-selector button.
- **Calendar grid cell gap is 4px (xs), not 8px** — a 7-column grid at
  mobile width (375px viewport, ~343px content width after 16px edge
  padding) needs the tightest gap on the scale to keep each cell ≥44px wide
  (343px − 6×4px = 319px ÷ 7 ≈ 45.6px). Do not loosen this to 8px — it drops
  cells under the touch-target floor.
- **Day Constructor row internal padding becomes 12px 12px** (updated from
  the existing `DayBlocksManager` row padding, `'10px 12px'` — the legacy
  10px vertical figure is retired here because 10px is not a multiple of 4
  and the spacing grid rule has no carve-out for "existing convention"; 12px
  is the nearest compliant value). Do not force it to exactly 16px either —
  that would waste horizontal space needed for the row's 5–6 inline controls
  (icon, name, price, move×2, toggle, delete); bumping vertical padding from
  10px to 12px is a 2px change and the row still fits comfortably.

## Typography

`lib/design/tokens.ts` fonts are locked. This phase declares the general ink
UI type scale for the two new/upgraded screens and the D-09 recolor sweep —
existing `components/settings/*` managers currently scatter 10–17px sizes and
600–800 weights inline (see `DayBlocksManager.tsx`, `CalendarSettingsManager.tsx`,
`PeriodsManager.tsx`); recoloring should consolidate onto this scale, not
preserve the sprawl.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 13px | 500 (Golos Text) | 1.5 |
| Label | 11px | 600 (Golos Text, uppercase, 0.06em tracking) | 1.3 |
| Heading | 17px | 700 (Bitter) | 1.2 |
| Display | 24px | 700 (Bitter) | 1.15 |

Two weights only: **500** (default/body) and **700** (emphasis/headings/labels).

**Atom-owned type (exception, already shipped — do not redesign):**
- `Amount` (`components/design/atoms.tsx`): sm 13px/700, md 15px/800,
  lg 22px/800, xl 30px/800 — JetBrains Mono, tabular-nums. The Weekly
  Summary card's coins-earned figure uses `size="lg"` (22px/800); this is
  the only place weight 800 appears in this phase's new UI.
- `LedgerRow` name (14px/600), `StatusChip` label (11px/600) — reuse verbatim,
  unchanged from 5.3/5.7.
- **Calendar grid day-number**: reuses the **Label role's size/weight**
  (11px/600) — not a new 5th size — with a font-family override to
  JetBrains Mono for tabular-numeral legibility in the dense 7-column grid
  (Label role's default family, Golos Text, is proportional and reads worse
  for stacked digits at this density). This is a font-family swap on an
  existing role, scoped to the new `CalendarGrid` component only, not a new
  size token added to the four-role scale above.
- **Weekday header row** (Mon…Sun labels above the grid): Label role (11px/600,
  uppercase) — reuse the general scale, no exception needed.

## Color

Base palette and ink theme are locked (`lib/design/tokens.ts` `ink` object,
re-exported as `T` in `components/parent-center/tokens.ts`). Every Parent
Center screen uses the **ink** theme exclusively — this phase adds no new
theme.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `T.bg1` `#141021` | Screen background on every Parent Center page |
| Secondary (30%) | `T.card` `#1D1830` / `T.cardHi` `#232040` | Cards, settings-manager panels, calendar grid cell base, row backgrounds |
| Accent (10%) | `T.indigo` `#6C5CE7` / `T.indigoHi` `#8B7BF5` | **Primary CTA buttons ("+ Add block", "Save calendar settings", "Apply preset"), active `Tabs`/child-selector tab, focus states, "Link to schedule" / "Override for…" triggers, today's calendar-cell ring, active weekend-day toggle in `CalendarSettingsManager`.** Nothing else |
| Destructive | `T.danger` `#FF8A96` | Delete-block / delete-period buttons, "remove override" confirmation, penalty rows |

**Gold is a separate, absolute invariant — not part of the 10% accent
budget:** `ink.goldText` `#F0BE6A` (there is **no** `gold`/`goldText` key on
`components/parent-center/tokens.ts`'s own `T` object — always reach through
`Amount`/`LedgerRow` from `components/design/atoms.tsx` with `theme="ink"`,
never hand-roll a gold hex in a parent-center file). Reserved exclusively for:
- Weekly Summary card's "coins earned this week" figure (`Amount money={true} theme="ink" size="lg"`)
- Day Constructor price fields, rendered as `Amount` where structurally
  possible (existing raw `<input type="number">` price fields may stay plain
  inputs — gold applies to the *displayed* value, not to every number input
  chrome; see Component Inventory)
- `ChatPanel.tsx`'s `coins_gain` activity entries (money) — see D-09 Recolor Scope

**Gold must never appear on:** the calendar grid (day-type coloring uses
`success`/`warning`/`muted`, never gold — day types are not money), the
streak-highlight fact on the Weekly Summary card (streaks are not money —
use `T.warning`, matching the established `05.7`/RESEARCH recolor
convention), badge/status chips, buttons, icons.

### Calendar grid day-type color mapping (D-05, new — concrete, not discretionary)

| Day type | Cell background | Cell text/number | Rationale |
|----------|-----------------|-------------------|-----------|
| `school` | `T.card` (default surface, no tint) | `T.text` | The "normal" day — no color budget spent on the majority case |
| `weekend` | `T.bg1` (one shade darker/quieter than `T.card`) | `T.muted` | Reads as "quiet, non-working," no semantic tone color needed |
| `vacation` | `hexAlpha(T.warning, 0.14)` (soft warm tint, same recipe as `StatusChip`) | `T.warning` | Warm/away signal; explicitly **not** gold — vacation is a schedule state, not money |
| `sick` (family-wide, i.e. `days.is_sick` for the currently-filtered single child) | `hexAlpha(T.danger, 0.14)` | `T.danger` | Only applies when a single child is filtered and that child was sick that day; in the family-wide (all-children) view, sick is shown as the D-06 overlay dot instead, never as a full-cell tint (a cell can't take two conflicting fills) |
| **Today** | (whatever the day-type fill is) **+ 2px `T.indigo` ring** (`box-shadow: 0 0 0 2px {T.indigo} inset` or a real border) | unchanged | Accent used as a state marker, matching "indigo = interactive/focus" precedent |
| **Selected/editing cell** (parent tapped a cell to add/edit a period) | `T.indigoSoft` background override | `T.text` | Matches the accent-for-active-state convention used by `Tabs` |

### D-06 sick-day overlay — concrete design (closes Claude's Discretion)

**Approach: small colored dots, bottom-right of the cell, tap-to-reveal
tooltip.** Concrete spec:
- When the grid is in **family-wide view** (no single child filtered), each
  day cell may render up to **3** small dots (5px diameter, 2px gap,
  bottom-right corner, 4px inset) — one per sick child that day, colored
  from `CHILD_ACCENTS` (`components/parent-center/tokens.ts`) in the same
  order children appear elsewhere (Dashboard, Analytics). If more than 3
  children are sick the same day (rare), render 2 dots + a `+N` 8px mono
  label instead of a 4th dot.
- Tapping a cell with ≥1 sick dot opens a small popover (anchored to the
  cell, `Card`-styled, `T.cardHi` background) listing the sick child
  name(s) for that date — "{Name} was marked sick." One line per child, no
  header needed for 1 child, `t('...multiple')` heading for 2+.
- In **single-child filtered view**, sick renders as the full cell tint
  (`sick` row in the table above) — no dot needed since there's only one
  child to represent.
- This is a **read-only** overlay in v1 of this screen — the sick flag
  itself is still set from the kid/parent day-fill form (`days.is_sick`),
  not editable from the calendar grid. No write path needed here.

## Accessibility & Visual Anchors

Non-blocking checker recommendation, addressed while in the file.

### Icon-only control labels (aria-label / visually-hidden text required)

Every icon-only button this phase touches or introduces must carry an
explicit accessible label — the visible glyph alone is not sufficient:

| Control | Screen | Accessible label (en / ru) |
|---|---|---|
| `chevL` / `chevR` month-nav | Year Calendar | "Previous month" / "Next month" · «Предыдущий месяц» / «Следующий месяц» |
| Row ↑ / ↓ (move up/down) | Day Constructor | "Move block up" / "Move block down" · «Переместить блок вверх» / «Переместить блок вниз» |
| Row ● (toggle active) | Day Constructor | "Enable block" / "Disable block" (state-dependent) · «Включить блок» / «Отключить блок» |
| Row 🗑 (delete) | Day Constructor | "Delete block" · «Удалить блок» |
| Schedule-link trigger chevron (`chevD`) | Day Constructor | trigger already carries visible text (`scheduleLink.none`/`.synced`) — the chevron itself needs no separate label, it's decorative inside a labeled button |
| "Override for…" trigger chevron (`chevD`) | Day Constructor | same as above — the button's visible text is the label, chevron is decorative |
| `[Custom ✕]` StatusChip remove | Day Constructor | "Remove {{name}}'s override" · «Удалить персональную настройку для {{name}}» — matches the destructive-confirmation copy already declared |

Implementation: `aria-label` on the `<button>` itself where there is no
visible text (move/toggle/delete rows, month-nav chevrons); where the button
already has visible text (schedule-link and override-for triggers), no
additional label is needed — `aria-label` must never contradict visible text.

### Primary visual anchor per screen (parity with Weekly Summary / Year Calendar)

- **Weekly Summary card:** the coins-earned figure (`Amount size="lg"`,
  gold) is the hero — already stated in the Screen Contract below.
- **Year Calendar:** the month grid itself is the hero; the settings form
  and period list are demoted to collapsed accordions below it — already
  stated in D-05.
- **Day Constructor (new statement, closes the gap the checker flagged):**
  there is no single hero figure — this is a management/utility screen, not
  a content-display screen. The **block list** (rows) is the primary visual
  anchor; the layout-toggle `Tabs` (By child / By block) sit above it as a
  secondary navigation control, and the "+ Add block" CTA sits below the
  list, not competing with it for attention. No color or size treatment
  should make any single row or control read as more important than the
  list as a whole.

### D-02 — Schedule-link picker (new UI element)

Added to the **per-child editor context only** (see D-03 below — a
family-default row has no single schedule to link to). Concrete layout,
placed as a new line inside the existing per-child control row (after the
existing who-fills/day-types line):

```
┌─────────────────────────────────────────────────────┐
│ [icon] Block name                     [price] [↑][↓][●][🗑]│
│  → who fills: [kid|parent|both]  day types: [☐school ☐weekend ☐vacation]│

└─────────────────────────────────────────────────────┘
```

- Trigger: a `Btn variant="ghost" size="sm"` labeled with the current state

- Once linked, show a small `StatusChip tone="success"` reading the synced
  copy next to the trigger, plus a small "↻ Re-sync" ghost icon-button (in
  case the underlying `schedule_items` row's days change later — this is a
  manual re-copy, not a live join, consistent with the denormalize approach).
- Accent color (`T.indigo`) on the trigger button only while unlinked
  (inviting the tap); once linked, the trigger reads in `T.text` with the
  `StatusChip` carrying the "linked" signal instead — do not double-signal
  with both indigo AND a success chip simultaneously.

## Screen Contract: Year Calendar (`CalendarSettingsManager` + `PeriodsManager` redesign, D-05/D-06/D-07)

### D-07 recolor — same substitution table as D-04, applied to both files
(`CalendarSettingsManager.tsx`'s inline rgba/hex form chrome,
`PeriodsManager.tsx`'s Tailwind `bg-gray-*`/`bg-amber-*`/`text-amber-*`

a hard design-contract rule, not just an inventory note.

### D-05 — Visual month/year grid (new component, concrete layout)

```
┌ Year Calendar ───────────────────────────────────────┐
│ [👦 Adam] [👧 Alim] [Family]  ← child filter Tabs     │
│                                                        │
│  ◀   September 2026   ▶        [ Today ]              │

│  ├──┼──┼──┼──┼──┼──┼──┤                                │
│  │ 6│ 7│ 8│ 9│10│11│12│  ← 9 has today's indigo ring   │
│  ├──┼──┼──┼──┼──┼──┼──┤                                │
│  │..│..│..│..│..│..│..│  ← vacation week, warm tint,   │
│  └──┴──┴──┴──┴──┴──┴──┘     sick dots bottom-right     │
│                                                        │
│  ▸ Edit school year, term mode, weekend days, region   │
│    (CalendarSettingsManager form — collapsed by default,│
│     opens as an accordion below the grid)              │
│  ▸ Vacation periods (PeriodsManager list + preset      │
│    picker — collapsed by default, same accordion pattern)│
└────────────────────────────────────────────────────────┘
```

Concrete decisions:
- **Grid columns are Monday-first** (Mon…Sun display order) even though
  the underlying `weekend_days`/`getDay()` convention is Sun=0…Sat=6
  internally — this is a pure rendering-order choice (RU calendar
  convention), not a data-model change; the grid component maps
  `getDay()` results to the Mon-first column index at render time only.
- Child filter `Tabs` at the top: "Family" (default, family-wide day types
  + all children's sick dots) or a specific child (single-child view:
  full-cell sick tint per the Color table, no dots needed).
- Month nav: `Icon name="chevL"`/`"chevR"` ghost icon-buttons flank a
  Heading-role month/year label (Bitter, 17px/700); a `Btn variant="ghost"
  size="sm"` "Today" jumps back to the current month.
- The existing settings form (`CalendarSettingsManager`) and the period
  list (`PeriodsManager`) become **collapsed accordion sections below the
  grid**, not separate settings sub-tabs removed — D-01 already locked "no
  new top-level nav items" and this keeps the sub-tab count in `Settings.tsx`
  unchanged (the `calendar` sub-tab now renders grid + accordion; `vacations`
  sub-tab can either stay separate or fold into the same accordion —
  planner's call, both are valid under D-01 since neither adds new nav
  depth). Their internal logic (materialization, preset-source marker,
  "manual wins") is untouched, per CONTEXT.
- Tapping a day cell with no vacation period opens `PeriodsManager`'s
  existing add-period form pre-filled with that date as `startDate`
  (a superficial pre-fill convenience, not a new data path); tapping a
  cell inside an existing vacation period opens that period's edit form
  (`openEdit`, already exists).
- New data need: `getDaysInRange(childId, start, end)` (per RESEARCH Code
  Examples) — one call per visible month per filtered child, to source the
  D-06 sick dots. Not needed for family-wide day-type coloring itself
  (that's computed from `getDayType()` + `getFamilyCalendar()` +
  `getVacationPeriods()`, no new query).

## Screen Contract: Weekly Summary card (Analytics, D-08)

Insertion point: `components/parent-center/screens/Analytics.tsx`, as a new
`Card` placed **above** the existing KPI grid (the "hero" position — it's
the one truly-real content on a screen whose charts below stay mocked).
Computed in `ParentCenter.tsx`'s existing `loadAll()` `Promise.all` batch and
passed down as new props (matches the established data-flow pattern per
RESEARCH's Open Question 3 recommendation — **locked**, not left open).

```
┌ This week ──────────────────────────────────────────┐
│  Coins earned            Tasks done      Streak      │
│  1,240 🪙 (gold, lg)      82% [chip]      Alim — 6-day│
│                                            streak     │
└───────────────────────────────────────────────────────┘
```

Concrete layout: `Card pad={16}`, heading (`analytics.weeklySummary.title`,
Heading role, 17px/700) at top, then a 3-column flex row (wraps to 1 column
under ~360px) — each fact is a small stat block: Label-role caption above,
value below.
- **Fact 1 (coins earned):** value via `Amount value={coinsThisWeek}
  theme="ink" money={true} size="lg"` (gold, per the correct-query Pitfall 3
  fix — `getTransactions(undefined, 500, weekStartISO)` summed over positive
  `coins_change`, **not** `getWeekScore.total`).
- **Fact 2 (completion rate):** value rendered as a `StatusChip` (not
  `Amount` — this isn't money) with tone `success`/`warning`/`danger` at the
  same 80/50 thresholds `AnalyticsScreen`'s existing "today %" `Pill` already
  uses (`>=80` success, `>=50` warning, else danger) — consistency with the
  rest of the screen's rate-coloring convention.
- **Fact 3 (streak highlight):** if any child has `current_count >= 5`
  (per CONTEXT's "≥5 days" threshold) in the already-loaded `streaks` data,
  show the highest one via the `streakHighlight` copy in `T.warning` color
  (not gold, not success — matches the streak-is-not-money convention
  locked in the D-09 recolor table). If no child qualifies, show the
  `noStreak` empty copy in `T.muted`, not a blank gap.
- **Card-level empty state:** if the family has zero transactions this week
  AND zero completed tasks (first-week-of-use case), the whole card shows
  the `weeklySummary.empty` copy centered, no 3-column layout attempt.
- **Card-level error state:** if the week-scoped query fails, show
  `weeklySummary.loadError` in `T.dangerSoft`/`T.danger` (same visual
  language as every other manager's error banner) in place of the 3 facts.
