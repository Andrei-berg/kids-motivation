# Phase 5.8: parent-redesign - Context

**Gathered:** 2026-07-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Parent Center moves fully onto the unified «family bank» ink theme (only the
Dashboard activity list and one other pilot spot were migrated in 5.3 — every
remaining screen/component still carries the old cyan/neon-green/hardcoded-hex
dark theme and must be recolored, no exceptions, per ROADMAP SC1). Two
functional-but-unstyled settings sub-tabs get promoted to full designed
screens: **Day Constructor** (`DayBlocksManager` → per-child block management +
schedule-link wiring) and **Year Calendar** (`CalendarSettingsManager` +
`PeriodsManager` → an actual visual month/year grid). Analytics gains a
**Weekly Summary card** with real narrative copy from simple existing
aggregates (not the full analytics engine — that stays mocked/deferred).

Out of scope: money logic, award math, day-block/calendar data models and RLS
(built in 5.6/5.5) — this phase is UI/screen work on top of them. Legacy pages
(`/dashboard`, `/wallet`, etc.) and globals.css purge are Phase 5.11. Kid UI is
already done (Phase 5.7).

</domain>

<decisions>
## Implementation Decisions

### Information architecture
- **D-01:** Day Constructor and Year Calendar **stay nested at their current
  depth** — Settings → Schedule → Blocks / Calendar / Vacations sub-tabs. No
  new top-level Parent Center nav items. (Parent Center bottom nav already has
  7 items + a separate Settings button; adding more was rejected as clutter.)
  Redesign is a reskin + capability upgrade in place, not an IA promotion.

### Day Constructor (`DayBlocksManager` redesign)
- **D-02:** **Wire `schedule_link` to `schedule_items`.** A block's schedule
  picker lists the child's existing lessons/sections/routines (e.g. "Football
  — Tue/Thu" from `schedule_items.title` + `day_of_week[]`); picking one makes
  the block auto-appear only on that item's scheduled weekdays, layered on top
  of the existing `day_types` filter. This was the field 5.6 explicitly
  reserved for this phase.
- **D-03:** **Per-child override UI — build both candidate layouts, decide at
  verification** (explicit user direction, same pattern as 5.7's D-11 chat
  theme): (a) a child selector/tabs above the block list showing that child's
  effective blocks (family default + their overrides highlighted, with a
  "family default" view always available), and (b) an inline per-block
  "override for…" expander listing all children with per-child fields on one
  screen. Both must actually work against the real repo (`getDayBlocks(familyId,
  {childId})` already resolves child-specific rows over family defaults); pick
  the one that reads better in the family-bank visual language, or keep both
  behind a toggle if neither clearly wins.
- **D-04:** Full ink-theme recolor of the manager itself (currently raw
  `rgba(255,255,255,…)` / hardcoded hex, dark-purple palette) — replace with
  `lib/design/tokens.ts` ink theme + shared atoms (`LedgerRow`/`Amount`/
  `StatusChip` where applicable, e.g. price fields as `Amount`).

### Year Calendar (`CalendarSettingsManager` + `PeriodsManager` redesign)
- **D-05:** **Build an actual visual month/year grid** — a calendar view with
  day cells color-coded by day type (school/weekend/vacation), not just the
  current settings-form + flat period list. The existing form (year start/end,
  term mode, weekend-day picker, region preset) and the period list
  (add/edit/delete vacation periods, preset apply/re-apply) become edit
  affordances layered on/around the grid — their underlying logic
  (materialization, preset-source marker, "manual wins") is untouched.
- **D-06:** **Add a per-child sick-day overlay on the family grid** — beyond
  family-wide day type (same for every child), show which child(ren) were
  marked sick on a given day directly on the grid (e.g. a child-color dot/mark
  on the cell), on top of the family-wide coloring. Needs a child-filter or
  multi-child day-cell design — Claude's discretion on exact visual.
- **D-07:** Full ink-theme recolor of both editors, matching D-04's approach.

### Analytics — Weekly Summary card
- **D-08:** Real narrative copy computed from simple, already-available
  aggregates (no new pipeline): **(1)** family total coins earned this week
  (SUM of positive `wallet_transactions` this week across children), **(2)**
  task/day-completion rate this week (reuse existing day-fill data — family-
  wide or per-child, Claude's discretion on presentation), **(3)** streak
  highlights (call out any child on a notable streak, e.g. ≥5 days, from the
  already-server-computed `streaks` table). This is a small additive card, not
  a rebuild of the (currently mocked/random) charts above it on the Analytics
  screen — those stay out of scope per ROADMAP wording ("static text for now").

### Ink-theme recolor scope (SC1)
- **D-09:** Full sweep — every Parent Center screen/component still on the old
  palette gets recolored via `components/parent-center/tokens.ts` (already
  re-exports the unified palette per 5.3; screens just haven't adopted the new
  values/atoms yet where they hardcode colors inline). In scope: `Dashboard`
  (finish beyond the 5.3 pilot), `WalletsScreen`, `ChildProfile`,
  `ChildrenTasksShop`, `ExpensesPanel`, `AuditScreen`, `ChatPanel` (parent side
  of the theme-aware `ChatThread` from 5.7 D-11 — finalize which theme parent
  chat uses), `Settings.tsx` and all its sub-managers (`RoomTasksManager`,
  streak/coins-rules managers, family manager, `PeriodsManager`,
  `CalendarSettingsManager`, `DayBlocksManager`), `ParentGoals`, `ActionModal`.
  Verification bar: no cyan/neon-green/hardcoded-hex remnants, gold only on
  money amounts (same contract as 5.3/5.7).

### Claude's Discretion
- Exact child-selector vs inline-expander visual polish for D-03; whether both
  ship behind a toggle or one is dropped after comparison.
- Exact multi-child sick-day cell design for D-06 (dots, stacked avatars,
  tooltip-on-tap, etc.).
- Presentation details of the Weekly Summary card layout (order of the three
  facts, copy tone) within the family-bank writing rules (plain verbs,
  sentence case, actions named by outcome).
- Whether `AnalyticsScreen`'s existing mocked bar/line charts get touched at
  all — default is no (out of scope; only the new Weekly Summary card is real).
- Order of recolor work across the D-09 screen list; whether any screen needs
  a `LedgerRow`/`Amount`/`StatusChip` atom adoption beyond pure recolor (atoms
  are available, adoption elsewhere is a bonus not a requirement of SC1).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `components/design/atoms.tsx` — LedgerRow/Amount/StatusChip/Stamp/useCountUp,
  theme-aware, already piloted on Dashboard's activity list and kid wallet.
- `getDayBlocks(familyId, { childId, activeOnly })` — already merges
  family-default + child-override rows by `legacy_key`, preferring the
  child-specific row; the repo-level machinery for D-03 already exists, only
  the management UI needs to write child-scoped rows (currently `addDayBlock`/
  `updateDayBlock` calls in `DayBlocksManager` never pass a `child_id`).
- `schedule_items` (`lib/repositories/schedule.repo.ts`) — per-child schedule
  entities with `day_of_week[]`, reusable for the D-02 picker.
- `getFamilyCalendar`/`upsertFamilyCalendar` (`lib/repositories/calendar.repo.ts`)
  and `vacation_periods` CRUD (`lib/vacation-api.ts`) — data layer for the new
  calendar grid; no schema changes needed, purely a new rendering surface.
- `days.is_sick` per child/date — source for the D-06 sick overlay (already
  used by `getDayType`).

### Established Patterns
- Settings sub-manager components (`RoomTasksManager`, `DayBlocksManager`,
  `CalendarSettingsManager`, `PeriodsManager`, streak/coins-rules managers) all
  currently hardcode `rgba(255,255,255,…)`/hex inline styles from before the
  5.3 token system — none have adopted `components/parent-center/tokens.ts`
  yet, despite tokens already carrying correct palette values.
  `components/parent-center/screens/Analytics.tsx` is the one screen already
  fully on `T`/atoms — use it as the reference for how a screen should look
  post-recolor.
- i18n via dotted keys in `lib/i18n` (en/ru json); every existing manager
  already has translated strings under `settings.dayBlocksManager.*` /
  `settings.calendarSettingsManager.*` — new UI additions need new keys in
  both locales following that pattern.
- `prefers-reduced-motion` double-guard pattern established in the Stamp atom
  (05.3-02 decision) — apply to any new grid/overlay animation.

### Integration Points
- `components/parent-center/screens/Settings.tsx` lines ~396-432 — the
  Schedule tab's sub-tab switch (`calendar`/`vacations`/`room`/`blocks`/
  `activities`) is the mount point for both redesigned screens; no routing
  changes needed (confirms D-01).
- `ChatThread` (shared kid/parent component, made theme-aware in 5.7 D-11) —
  `components/parent-center/screens/ChatPanel.tsx` is where the parent-side
  theme choice gets finalized per D-09's ChatPanel recolor item.
- `/api/wallet/award` and award computation are NOT touched by this phase —
  D-02's schedule-link only changes block *visibility*, which 5.6's award
  logic already reads from `day_blocks` config at render/save time.
