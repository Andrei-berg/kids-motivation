# Phase 05.4: Code Review Report

**Reviewed:** 2026-07-07T10:49:56Z
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

The phase itself is well-executed on its stated scope: the migration is idempotent, the six streak columns are threaded consistently through `SETTINGS_DEFAULTS`, the `WalletSettings` type, the repo fallback, the award route, and the Parent Center editor; write-side clamps ([1,365] days, [0,100000] bonus) and independent award-side clamps are both present; the six new i18n keys resolve in both en and ru; and the integration tests genuinely exercise the settings-driven path against the live DB.

However, the streak-bonus award path that this phase rewrote contains an exploitable economic hole: the bonus is keyed solely by a **client-supplied date string** and is computed from a **client-writable** `streaks` table, with no check that the date is today, real, or related to the streak. A child can mint unbounded coins. This predates 05.4, but 05.4 rewrote `computeStreakBonus` (commit `1b5e0ff`) and shipped "server-side clamps" as a goal — the date/replay dimension was left unclamped. Additional warnings cover a focus-loss React anti-pattern in the new streak editor rows, silent divergence between server-clamped and UI-displayed values, a stats-corrupting race-compensation path in `creditAwards`, and legacy client-side money mutators that still have live callers.

## Critical Issues

### CR-01: Streak bonus is mintable for arbitrary client-supplied dates (unbounded coin creation)

**File:** `app/api/wallet/award/route.ts:43,231-241,270-291`
**Issue:** Every other award source is anchored to a saved DB row (`day.id`, `subject_grades.id`, `section_visits.id`, etc.) — no data, no award. The streak bonus is not. Its idempotency key is `sourceId: date`, where `date` comes straight from the request body and is validated only by the regex `/^\d{4}-\d{2}-\d{2}$/` (which also accepts impossible dates like `9999-99-99` and future dates). `computeStreakBonus` never looks at `date` at all — it reads the child's *current* `streaks.current_count` and pays the bonus if it meets the threshold.

Consequence: a child (allowed to call this route for their own `childId`, route.ts:50-54) with any qualifying streak can loop `POST /api/wallet/award {childId, date}` over arbitrary distinct date strings and collect `bonus × N` coins — +100 per request at defaults, up to 100,000 per request at the clamp ceiling a parent can legitimately configure. Compounding factor: `streaks.current_count` is written **client-side** (`updateStreaks` in `lib/services/streaks.service.ts` runs in the browser via the RLS-bound client and the day-fill forms call it, `KidDayFillForm.tsx:510`), so a tampering child does not even need a real streak — they can upsert `current_count: 999` directly and then farm dates. This violates the project's core money invariant that award amounts derive only from server-verified saved data. The suite's own child-role gating test silently demonstrates the hole: the lingering room streak mints a +100 `streak` transaction for `CHILD_ROLE_DATE` under a `role: 'child'` caller (the test only asserts on `behavior` rows, so it passes).

**Fix:** In the award route, refuse to pay a streak bonus for any date other than the server-computed "today" (and validate the date is a real calendar date):
```ts
// route.ts — before pushing the streak intent
import { localDateString } from '@/utils/helpers'

const parsed = new Date(`${date}T00:00:00`)
if (Number.isNaN(parsed.getTime())) {
  return NextResponse.json({ error: 'invalid date' }, { status: 400 })
}
const streakBonus = date === localDateString()
  ? await computeStreakBonus(admin, childId, settings)
  : 0
```
Longer-term, key the bonus to the milestone rather than the day (e.g. `sourceId: \`${streak_type}:${threshold}:${streaks.last_updated}\``) and/or move `updateStreaks` server-side so `current_count` is not client-writable money input.

## Warnings

### WR-01: `RuleRow` is defined inside `CoinsRulesTab` — every keystroke remounts the input and drops focus

**File:** `components/parent-center/screens/Settings.tsx:161-174` (used at 182-186, 192-194, 200-205, 214-216)
**Issue:** `RuleRow` is an arrow-function component recreated on every render of `CoinsRulesTab`. Each `up()` call triggers `setSettings` → re-render → new `RuleRow` function identity → React sees a different element type → unmounts and remounts the row, destroying the `<input>` DOM node. Typing a multi-digit value (e.g. `365` into the new `streak_room_days` field, or `250` into a bonus field) loses keyboard focus after every character. The pattern predates this phase, but 05.4 added six more affected rows and the streak fields are the ones most likely to receive multi-digit input.
**Fix:** Hoist `RuleRow` to module scope and pass the change handler as a prop:
```tsx
function RuleRow({ icon, label, value, rkey, up }: {
  icon: string; label: string; value: number
  rkey: keyof WalletSettings; up: (k: keyof WalletSettings, v: string) => void
}) { /* unchanged body */ }
```

### WR-02: Saved settings UI silently diverges from server-clamped values

**File:** `components/parent-center/screens/Settings.tsx:144-159`; `app/api/wallet/settings/route.ts:16-35`
**Issue:** `save()` calls `updateWalletSettingsApi(settings)` and discards the return value, while the PATCH route clamps streak fields before persisting (days → [1,365], bonus → [0,100000]) and returns the authoritative row. A parent who enters `streak_room_days: 0` (which `up()`'s `Number(v) || 0` also produces for an emptied field) sees "saved" and `0` on screen, while the DB holds `1`. The state stays wrong until a full reload. `updateWalletSettingsApi` already returns the clamped `settings` object (lib/wallet-client.ts:33-36).
**Fix:**
```ts
const saved = await updateWalletSettingsApi(settings)
if (saved) setSettings(saved)
```

### WR-03: `creditAwards` race compensation permanently corrupts lifetime earned/spent totals

**File:** `app/api/wallet/_lib.ts:316-324`
**Issue:** When the 23505 backstop fires, the original apply added `earnedCoins: +N` (for a positive intent). The reversal applies `coins: -N` with `spentCoins: N` — so `total_earned_coins` stays inflated by N **and** `total_spent_coins` is inflated by N. Balance nets out, but both lifetime counters (surfaced in wallet UI/analytics) are permanently wrong by N per raced intent. A correct reversal must subtract from the same counter it credited.
**Fix:**
```ts
await applyWalletDelta(admin, childId, {
  coins: -intent.coins,
  earnedCoins: intent.coins > 0 ? -intent.coins : 0,
  spentCoins: intent.coins < 0 ? -Math.abs(intent.coins) : 0,
})
```
(Verify `wallet_apply` accepts negative counter deltas; if not, add a dedicated reversal mode.)

### WR-04: Client-side money mutators still exported from wallet.repo.ts with live callers

**File:** `lib/repositories/wallet.repo.ts:39-116` (`updateWalletCoins`), `339-416` (`purchaseReward`), `466-580` (`createPurchaseRequest`), `762-819` (`exchangeCoins`), `870-927` (`approveWithdrawal`), `1177-1213` (`createP2PTransfer`)
**Issue:** These functions perform direct client-side writes to RLS-locked money tables — exactly what CLAUDE.md forbids ("Never reintroduce a client-side wallet write"). RLS makes them fail at runtime (`.single()` on a 0-row update throws), but they still have **live callers**: `components/ShopModal.tsx:77` (`purchaseReward`), `components/ExchangeModal.tsx:87` (`exchangeCoins`), `components/P2PTransferModal.tsx:118` (`createP2PTransfer`), `components/CoachRatingModal.tsx:130` (`awardCoinsForSport` → `updateWalletCoins`). Any legacy page mounting these modals has silently broken money flows, and the exported functions are a standing drift risk (a future caller "works" in a dev DB without the RLS lock). `createP2PTransfer` is additionally non-atomic (two sequential `updateWalletCoins` calls — sender can be debited while recipient credit fails) and fabricates a fake transfer row (`id: Date.now().toString()`) that never touches `p2p_transfers`. Pre-existing, but this file was modified in-phase and the functions remain.
**Fix:** Delete the mutators (or replace their bodies with `throw new Error('moved server-side — use /api/wallet/*')`) and repoint the four legacy callers at `lib/wallet-client.ts` wrappers.

## Info

### IN-01: `coins_per_book` missing from `WalletSettings` type and repo fallback despite "mirror" claim

**File:** `lib/models/wallet.types.ts:96-126`; `lib/repositories/wallet.repo.ts:186-216`; `app/api/wallet/_lib.ts:35,54`
**Issue:** `_lib.ts` line 35 says its defaults "mirror lib/repositories/wallet.repo.ts getWalletSettings() fallback", but the repo fallback and the `WalletSettings` interface both omit `coins_per_book` (present in `SETTINGS_DEFAULTS` and used by the award route). The phase updated all three places for the six streak fields, so the sets now diverge only by this key. Also means the Coins Rules editor has no field for the book bonus.
**Fix:** Add `coins_per_book: number` to the interface and `coins_per_book: 20` to the repo fallback (and optionally a RuleRow for it).

### IN-02: `getStreakBonuses` is dead code with stale hardcoded thresholds

**File:** `lib/services/streaks.service.ts:250-283`
**Issue:** The client-side function the award route was "ported from" still hardcodes 7/14/7 and reads the legacy global `settings` key/value table. It has no callers (only re-exported via `lib/streaks.ts`). If ever reused, it will disagree with the per-family `wallet_settings` values that now drive actual awards.
**Fix:** Delete it (and its re-export), or port it to read `wallet_settings` like the server.

### IN-03: Settings PATCH validates only the six streak fields

**File:** `app/api/wallet/settings/route.ts:37-59`
**Issue:** All other numeric columns pass through unvalidated: a tampered parent PATCH with `coins_per_grade_5: "abc"` produces a raw Postgres type error surfaced verbatim in the 500 body via `errorResponse`, and absurd magnitudes are accepted. Parent-only and self-family, so low impact, but the clamp discipline introduced in-phase is inconsistent.
**Fix:** Extend the numeric coercion/clamp loop to an allowlist of all editable numeric columns; strip unknown keys.

### IN-04: Award route re-implements `authorizeChildAction` inline

**File:** `app/api/wallet/award/route.ts:47-57`; `app/api/wallet/_lib.ts:15-24`
**Issue:** The child-role + `assertChildInFamily` sequence duplicates the shared `authorizeChildAction` helper used by sibling routes. Duplication invites drift in the auth check.
**Fix:** `await authorizeChildAction(admin, member, childId)` after `createAdminClient()`.

### IN-05: Streak "days" fields rendered as coin values in the editor

**File:** `components/parent-center/screens/Settings.tsx:200-205,161-174`
**Issue:** `RuleRow` appends a 🪙 suffix and colors positive values green — correct for bonuses, misleading for the three day-count thresholds (a "days to bonus" number displayed as a green coin amount). Risks parent misconfiguration (entering a coin-sized number into a days field; server clamps at 365 silently — see WR-02).
**Fix:** Add a `unit?: 'coins' | 'days'` prop to `RuleRow` and render 📆/plain text without the earn/penalty coloring for days fields.

### IN-06: Deploy-order coupling — settings save breaks if code ships before the migration is applied

**File:** `app/api/wallet/settings/route.ts:48-57`; `lib/repositories/wallet.repo.ts:209-214`
**Issue:** The client fallback settings object now always contains the six `streak_*` keys, and the PATCH upserts every key it receives. In the window where this code is live but `05.4-01-streak-settings.sql` is unapplied, every Coins Rules save fails with "column does not exist" (500) — not just the streak fields. The award route degrades gracefully (defaults win); the settings route does not. Migrations are applied manually in this project, so worth knowing.
**Fix:** None required if the migration is already applied in prod; otherwise apply before deploy. For future phases, consider tolerating unknown-column upsert failures or gating new keys.

### IN-07: award.test.ts tests are order-coupled and under-assert on unexpected credits

**File:** `tests/integration/award.test.ts:190,319-350,352-361`
**Issue:** (a) The idempotency test depends on the first test's seeded data, and the final "teardown" test destroys the shared family — running a filtered subset (`vitest -t`) or reordering breaks the suite. (b) The child-role gating test asserts only that no `behavior` row exists; it does not assert the total transaction delta, so the +100 `streak` credit that the child-role call actually mints (see CR-01) passes unnoticed. Asserting `creditedCoins` from the response body would have caught the exploit.
**Fix:** Assert `body.creditedCoins === 0` (or the exact expected value) in the child-role test; move family destruction to `afterAll` only.

### IN-08: ru.json missing 32 `dailyModal.*` keys present in en.json

**File:** `public/locales/ru.json` (vs `public/locales/en.json:` `dailyModal.*`)
**Issue:** Flattened key-set diff shows 32 en-only keys, all under `dailyModal.*` (e.g. `dailyModal.roomBed`, `dailyModal.coachRatingLabel`). Pre-existing, unrelated to this phase (the six new streak keys and the `streaks` card title are present in both locales), but Russian users of the daily modal fall back to raw keys or English.
**Fix:** Backfill the `dailyModal` block in ru.json.
