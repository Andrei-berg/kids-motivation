# Phase 05.4 Plan 02: Settings-Driven Streak Bonus Summary

**`/api/wallet/award` now computes streak bonuses from per-family `wallet_settings` (streak_room/study/sport_days+bonus) instead of hardcoded 7/14/7/100 or the legacy `settings` key/value table, with lower+upper clamps on both the read and write side**

## Performance

- **Duration:** 25 min
- **Started:** 2026-07-07T13:07:00Z
- **Completed:** 2026-07-07T13:16:54+03:00
- **Tasks:** 2
- **Files modified:** 4

## Accomplishments
- `computeStreakBonus` now takes the already-loaded `WalletSettingsRow` and reads `streak_room_days/bonus`, `streak_study_days/bonus`, `streak_sport_days/bonus` — the legacy `admin.from('settings').select(...)` key/value read is gone entirely
- Authoritative clamp inside `computeStreakBonus`: `Math.max(1, Math.floor(...))` for thresholds, `Math.min(100000, Math.max(0, Math.floor(...)))` for bonuses — a negative/zero/absurd stored value can never produce a negative award, an always-fires threshold, or an unbounded bonus
- `PATCH /api/wallet/settings` clamps the same six `streak_*` fields before upsert (days -> [1,365], bonus -> [0,100000]) as defense-in-depth against a tampered client PATCH
- Integration suite (`tests/integration/award.test.ts`) extended with default parity (room streak, no wallet_settings row, exactly +100) and a settings-driven case (study streak, `streak_study_days=1`/`streak_study_bonus=250`, exactly +250)
- `destroyTestFamily` gained a family_id-scoped `wallet_settings` delete as a teardown safety net (no `ON DELETE CASCADE` from `families`)

## Decisions Made
- `computeStreakBonus` sums the bonus across every qualifying streak type for the child on each award call — this is pre-existing, unchanged behavior, not something this plan altered. The new settings-driven test accounts for it by setting `streak_room_days: 365` in the same settings upsert used for the study case, which pushes the shared permanent room streak (`current_count: 7`) below its new threshold so it contributes 0 — isolating the assertion to exactly the study bonus (250) as the plan's acceptance criteria required, rather than a combined 350.
- Copied `.env.local` from the main repo checkout into the worktree (gitignored, untracked, never committed) to obtain `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` for running the live-DB integration suite — same operational step as 05.4-01, no secret fabricated.
