## Phase Boundary

Remove all hardcoded `'adam'`/`'alim'` references from TypeScript files. Every page loads children dynamically from the authenticated family context. Zustand store migrates from legacy string IDs to UUID-based fields. No new features — pure data wiring refactor.

<decisions>
## Implementation Decisions

### Child switcher UI
- Replace the hardcoded `<select>` in NavBar with **pill buttons** (same pattern already used in wallet and coach-rating pages)
- Buttons load children dynamically from DB (via authenticated family context)
- Consistent visual language across the app — no new patterns needed

### Auth guard behavior
- Instant redirect to `/login` if unauthenticated — no spinner or skeleton before redirect
- No loading flash; redirect happens server-side or as fast as possible client-side

### Store migration cut-over
- Clean cut-over: remove legacy `childId: 'adam' | 'alim'` and `setChildId` from `store.ts`
- All 4 pages (expenses, coach-rating, wallet, dashboard) updated in the same pass
- `activeMemberId` (UUID) becomes the sole child identifier throughout the app
- `familyId` stays (already in store from Phase 1.3)

### P2P sibling auto-select
- When family has exactly 2 children: auto-select the sibling as P2P recipient (current scenario)
- When family has 1 child: hide or disable P2P transfer UI
- When family has 3+ children: show a dropdown to pick recipient
- No hardcoded `'adam'`/`'alim'` comparisons — derive sibling from family members list

### Claude's Discretion
- How to fetch the children list (single hook, context provider, or per-page fetch)
- Exact loading state within each page while children list loads
- How to persist `activeMemberId` to localStorage across sessions (already in `name: 'v5_child'` persist key — may need to clear/migrate old persisted value)

</decisions>
