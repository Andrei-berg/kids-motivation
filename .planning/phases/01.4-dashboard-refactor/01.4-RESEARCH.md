## Summary

Phase 1.4 is a pure refactoring phase with no new features. The goal is to eliminate all hardcoded `'adam'` and `'alim'` string references from the TypeScript codebase and replace them with UUID-based multi-tenant patterns already established in Phases 1.1–1.3.

The existing middleware (`middleware.ts`) already handles the auth guard: unauthenticated users are redirected to `/login`, and authenticated users without a family are redirected to `/onboarding`. This means the "auth guard" requirement for pages is already satisfied at the infrastructure level — pages do not need their own client-side redirect logic. The only work needed client-side is loading `familyId` and `activeMemberId` from Supabase auth context.

The Zustand store already has `activeMemberId`, `familyId`, `setActiveMemberId`, and `setFamilyId` from Phase 1.3. The legacy `childId: 'adam' | 'alim'` and `setChildId` fields must be removed, and all 9 files currently referencing hardcoded adam/alim must be updated to use `activeMemberId` + a dynamically-loaded children list.

**Primary recommendation:** Build a single `useFamilyMembers()` hook that fetches the authenticated user's family UUID and children list from `family_members`, writes `familyId` into the Zustand store, and returns the members array. NavBar calls this hook to render dynamic pill buttons. All other pages read `activeMemberId` from the store (which NavBar keeps current).

### Pattern 1: useFamilyMembers Hook

**What:** A single custom hook that:
1. Calls `supabase.auth.getUser()` to get the authenticated user
2. Queries `family_members` for the user's membership row (gets `family_id`)
3. Queries all children in that family (`role = 'child'`)
4. Writes `familyId` into the Zustand store
5. Auto-selects `activeMemberId` if not already set (first child in list, or the child whose `user_id` matches the logged-in user if role is child)
6. Returns `{ members, loading, error }`

**When to use:** Called from `NavBar` (which renders on every page). Pages read `activeMemberId` from the store — they do not call this hook themselves.

**Example:**
```typescript
// lib/hooks/useFamilyMembers.ts
'use client'

import { useEffect, useState } from 'react'
import { createClient } from '@/lib/supabase/client'
import { useAppStore } from '@/lib/store'

export interface FamilyMember {
  id: string           // family_members.id (UUID)
  display_name: string
  avatar_url: string | null
  role: 'parent' | 'child' | 'extended'
}

export function useFamilyMembers() {
  const { familyId, setFamilyId, activeMemberId, setActiveMemberId } = useAppStore()
  const [members, setMembers] = useState<FamilyMember[]>([])
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState<string | null>(null)

  useEffect(() => {
    async function load() {
      try {
        setLoading(true)
        const supabase = createClient()

        const { data: { user } } = await supabase.auth.getUser()
        if (!user) return  // middleware already handles redirect; no-op here

        // Get this user's family membership
        const { data: membership, error: memberError } = await supabase
          .from('family_members')
          .select('family_id, id, role')
          .eq('user_id', user.id)
          .maybeSingle()

        if (memberError || !membership) return

        // Write familyId to store (idempotent)
        if (membership.family_id !== familyId) {
          setFamilyId(membership.family_id)
        }

        // Fetch all children in this family
        const { data: children, error: childError } = await supabase
          .from('family_members')
          .select('id, display_name, avatar_url, role')
          .eq('family_id', membership.family_id)
          .eq('role', 'child')
          .order('display_name')

        if (childError) throw childError

        const childList = (children || []) as FamilyMember[]
        setMembers(childList)

        // Auto-select: if no activeMemberId is set yet, default to first child
        if (!activeMemberId && childList.length > 0) {
          setActiveMemberId(childList[0].id)
        }
      } catch (err) {
        setError((err as Error).message)
      } finally {
        setLoading(false)
      }
    }

    load()
  }, []) // runs once on mount — NavBar mounts on every page

  return { members, loading, error }
}
```

### Pattern 2: Updated NavBar with Dynamic Pill Buttons

**What:** Replace the `<select>` with mapped pill buttons over the `members` array from `useFamilyMembers()`.

**Example:**
```typescript
// components/NavBar.tsx (updated switcher section only)
'use client'

import { useFamilyMembers } from '@/lib/hooks/useFamilyMembers'
import { useAppStore } from '@/lib/store'

export default function NavBar() {
  const { activeMemberId, setActiveMemberId } = useAppStore()
  const { members, loading } = useFamilyMembers()

  return (
    <div className="nav">
      {/* ... brand ... */}
      <div className="navR">
        {/* Dynamic child switcher */}
        {!loading && members.map((member) => (
          <button
            key={member.id}
            className={activeMemberId === member.id ? 'btn primary' : 'btn'}
            onClick={() => setActiveMemberId(member.id)}
          >
            {member.avatar_url && !member.avatar_url.startsWith('http')
              ? member.avatar_url  // emoji
              : '👦'
            } {member.display_name}
          </button>
        ))}
        {/* ... nav pills ... */}
      </div>
    </div>
  )
}
```

### Pattern 3: Page Store Consumption (no per-page fetch)

Pages read `activeMemberId` (not `childId`) from the store. No page fetches the children list — that's NavBar's job.

```typescript
// Any page — before (legacy)
const { childId } = useAppStore()

// After (Phase 1.4)
const { activeMemberId } = useAppStore()

// Pass to API calls
await api.getWeekData(activeMemberId!, today)
```

**Guard for null activeMemberId:** Pages that were previously guaranteed `childId` now receive `string | null`. Add a loading/null guard:

```typescript
const { activeMemberId } = useAppStore()

useEffect(() => {
  if (!activeMemberId) return  // wait for NavBar hook to set it
  loadData()
}, [activeMemberId])
```

### Pattern 4: P2P Sibling Derivation

**What:** Instead of `childId === 'adam' ? 'alim' : 'adam'`, derive sibling from the members list.

**Example:**
```typescript
// app/wallet/page.tsx — P2P sibling logic
const { activeMemberId } = useAppStore()
const { members } = useFamilyMembers()

const siblings = members.filter(m => m.id !== activeMemberId)

// 2-child family: auto-select sibling
const toMember = siblings.length === 1 ? siblings[0] : null
// 3+ child family: render a <select> from siblings
// 1-child family: siblings.length === 0 → hide P2P button
```

### Pattern 5: localStorage Migration for v5_child Persist Key

The Zustand store uses `name: 'v5_child'`. The persisted value currently contains `childId: 'adam'` (string). After Phase 1.4 removes the `childId` field, the old persisted object in localStorage will contain a stale `childId` key that Zustand will silently ignore (it only restores keys that match the store interface). No explicit migration needed — old `childId` key is ignored, and `activeMemberId` (already in the store schema) will be restored from localStorage if it was set.

**Verification:** On first load after deploy, if `activeMemberId` is null in localStorage (which it will be for existing sessions), the `useFamilyMembers` hook auto-selects the first child. This is correct behavior.

### Anti-Patterns to Avoid

- **Fetching children list in each page:** This causes N+1 auth calls. Only NavBar calls `useFamilyMembers()`. Pages read from the store.
- **Using `childId` string comparisons for display names:** All display data comes from `members[].display_name` — never from a hardcoded switch on ID.
- **Passing `activeMemberId` as `null` to API calls:** Guard with early return when `activeMemberId` is null. Legacy `lib/api.ts` functions expect a non-null string.
- **Client-side auth redirect logic in pages:** The middleware already handles this. Adding `useRouter` + auth check in pages creates double redirect and flicker.
- **Mutating `familyId` in store from multiple places:** Only `useFamilyMembers` writes `familyId`. Pages are read-only consumers.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|

**Key insight:** The auth + redirect infrastructure from Phase 1.1 is complete and correct. Phase 1.4 only needs to wire the already-existing `family_members` table data into the UI layer.

## Common Pitfalls

### Pitfall 1: activeMemberId null on initial render

**What goes wrong:** Pages that previously had `childId: 'adam'` (guaranteed non-null) now read `activeMemberId: string | null`. If a page runs `api.getChild(activeMemberId)` before the hook sets it, it passes `null` to the API and gets an error.

**Why it happens:** Zustand state initializes from localStorage (which may not have `activeMemberId` set yet), and the `useFamilyMembers` hook inside NavBar runs asynchronously.

**How to avoid:** In every `useEffect` that reads `activeMemberId`, add:
```typescript
useEffect(() => {
  if (!activeMemberId) return
  loadData()
}, [activeMemberId])
```

**Warning signs:** Console errors like "invalid input for type uuid" or "column child_id does not exist" when passing null.

### Pitfall 2: Legacy API functions expect `child_id = 'adam'` in old tables

**What goes wrong:** `lib/api.ts` functions (e.g., `getChild`, `getWeekData`, `getStreaks`) query the old `children` table using `child_id` TEXT. After Phase 1.4, these functions receive UUIDs from `family_members.id` — which don't exist in the old `children` table.

**Why it happens:** The data migration from the old `children` table to `family_members` is separate work (Phase 2.x). Phase 1.4 only updates the UI; the underlying API functions still talk to legacy tables.

**How to avoid:** Recognize that the legacy API functions will return "not found" errors when called with UUIDs. Phase 1.4 must scope its API calls carefully:
- Functions that query `family_members` directly (from `lib/onboarding-api.ts`) are safe.
- Functions in `lib/api.ts` that query the old `children` table by string ID are NOT UUID-safe.
- For this phase: the store's `activeMemberId` is the `family_members.id` UUID. Pages currently using `api.getChild(childId)` may break unless either: (a) the old `children` rows are updated to have UUID IDs matching `family_members.id`, or (b) the page skips calling that function.

**Recommendation for planning:** The planner should flag which legacy `lib/api.ts` calls remain valid after the store migration and which need to be stubbed/skipped during this phase. The safest approach is to confirm with the seed migration that the legacy `children` rows have `id = 'adam'`/`'alim'` and the new `family_members` rows have different UUIDs — meaning ALL `lib/api.ts` calls will break with UUIDs until Phase 2.x remaps the data. Plan tasks with this constraint in mind.

**Warning signs:** TypeScript complains about `string | null` where `string` expected; runtime "not found" from Supabase queries.

### Pitfall 3: NavBar hook running on every page mount

**What goes wrong:** `useFamilyMembers` in NavBar makes 2 Supabase queries on every page navigation (since NavBar is rendered inside each page, not in the layout). This means every page visit makes 2 DB round-trips just for the children list.

**Why it happens:** NavBar is rendered inside each page file (not in `app/layout.tsx`), per the project's established pattern (CLAUDE.md).

**How to avoid:** Write `familyId` and the children list into the Zustand store. On subsequent renders, check if `familyId` is already set and `members` is already loaded — skip the fetch if data is fresh. OR: keep the fetch fast (only 2 simple queries) and accept the cost — for a family app with 2-3 children this is negligible.

**Warning signs:** Network tab showing repeated identical queries on every page click.

### Pitfall 4: Stale childId reference in ExpenseModal and AuditLogViewer

**What goes wrong:** `components/ExpenseModal.tsx` has `const [childId, setChildId] = useState('adam')` — a local component state defaulting to `'adam'`. This is not from the Zustand store. After Phase 1.4, this component must receive the active member UUID (or the family members list) as a prop so it can render a dynamic child picker.

**Why it happens:** ExpenseModal was written before multi-tenancy; it manages its own child selection state.

**How to avoid:** Add `members: FamilyMember[]` prop to `ExpenseModal`. Replace the radio buttons with a dynamic list from the prop. The parent page (`app/expenses/page.tsx`) passes `members` from the Zustand store (or from a hook call).

**Warning signs:** ExpenseModal radio buttons still show "Адам" and "Алим" hardcoded.

### Pitfall 5: AuditLogViewer displayName lookup

**What goes wrong:** `components/AuditLogViewer.tsx` line 126: `case 'child': return '👦 ' + (childId === 'adam' ? 'Адам' : 'Алим')` — hardcoded name lookup.

**How to avoid:** Pass `displayName: string` as a prop instead of deriving it from ID comparison. The parent (`app/audit/page.tsx`) reads `activeMemberId` from store and passes the display name resolved from the members list.

## Open Questions

1. **Legacy API functions and UUID compatibility**
   - What we know: `lib/api.ts` functions pass `childId` directly to `.eq('child_id', childId)` on the old `children` table. The old table has `id = 'adam'` and `id = 'alim'`. After Phase 1.4, the store passes `family_members.id` UUIDs to these functions.
   - What's unclear: Do the `family_members` rows for Adam and Alim have the same IDs as the old `children` rows? If not, all `lib/api.ts` data-fetching calls will return "not found".
   - Recommendation: The planner should add a Wave 0 investigation task — check the actual `family_members.id` values for Adam and Alim against the `children.id` values. If mismatched, the page data will be empty but the refactor can still proceed — the broken API calls are pre-existing issues from the legacy data model, not caused by Phase 1.4.

2. **`activeMemberId` persistence across browser sessions**
   - What we know: Zustand persist with `name: 'v5_child'` will restore `activeMemberId` from localStorage. Old sessions have `activeMemberId: null`. The `useFamilyMembers` hook auto-selects the first child if null.
   - What's unclear: Should the hook validate that the persisted `activeMemberId` actually belongs to the current authenticated user's family? (Defensive check against stale localStorage from a different user's session.)
   - Recommendation: Add a validation step in `useFamilyMembers`: after loading the members array, if the persisted `activeMemberId` is not in the array, reset it to `members[0].id`. This handles cross-user session contamination.
