## Known Remaining Issues / Residuals

### app/wallboard/page.tsx (known residual — documented per plan)

- **Status:** Not modified — not in the plan's `files_modified` list
- **Current behavior:** Uses `api.getChildren()` from the legacy `children` table (TEXT primary key `'adam'`/`'alim'`). Uses a local `childId` variable as a dictionary key — not from the Zustand store. No TypeScript errors introduced.
- **Hardcoded strings:** None — no `'adam'` or `'alim'` string literals in the file
- **Impact:** Wallboard will not display data from the authenticated family context (UUID-based) after Phase 1.4. It reads from the old `children` table.
- **Resolution:** Deferred to Phase 2.x when the legacy data model is migrated to the `family_members`-based schema.

### P2P with 2+ siblings (known gap — TODO commented in code)

- When `siblings.length >= 2`, the P2P button is shown but the modal won't open (because `p2pRecipient` is `null`). A sibling selector dropdown needs to be added.
- Deferred: current family has 2 children (1 sibling per user), so this is not a regression.

### Legacy API functions returning empty for UUIDs

- `api.getChild(activeMemberId!)` will return `null` for UUID-based activeMemberIds because the `children` table has TEXT PKs `'adam'`/`'alim'`. The dashboard already handles `child === null` gracefully.
- Pre-existing issue — not caused by Phase 1.4. Resolved in Phase 2.x data migration.
