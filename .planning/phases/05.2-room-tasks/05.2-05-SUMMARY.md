# Phase 05.2 Plan 05: DailyModal renders room_tasks + dual-write Summary

**Parent DailyModal Room checklist is now data-driven from `room_tasks` (active, ordered) instead of the hardcoded 5-item array with per-item useState booleans, and `handleSave` dual-writes `room_checks` (every rendered task) plus the legacy `days.room_*` columns for tasks carrying a `legacy_key` — sick-day grace branch preserved byte-for-byte.**

## Accomplishments
- Removed the 5 hardcoded room `useState` booleans (`roomBed`/`roomFloor`/`roomDesk`/`roomCloset`/`roomTrash`) and the hardcoded 5-element checklist array; the Room section now maps over `roomTasks` (`getRoomTasks(familyId)`, active, `sort_order` ascending), rendering `task.icon ?? '🏠'` + `task.name` inside the unchanged `premium-checklist`/`premium-checkbox` markup. (Closet emoji converges from 👕 to the seeded 🚪 on both surfaces — expected per the plan's read_first note.)
- Added `roomTasks`/`roomChecked` state (`roomChecked` keyed by `task.id`). Loaded in `loadData`: `family_id` resolved from the `children` table (store `familyId` as fallback); pre-fill uses `getRoomChecks(childId, date)` when any rows exist, otherwise falls back to the legacy `dayData.room_*` booleans mapped by each task's `legacy_key`, so an already-legacy-saved day pre-checks correctly.
- `roomScore`/`roomOk`/`roomCoins` preview derives from the dynamic checked-state: `roomOk` when `doneCount >= max(1, ceil(0.6 * taskCount))` — byte-parity with the Plan 03 server threshold (3/5 for the default 5-task family). The section badge shows `doneCount/taskCount` instead of the literal `/5`.
- `handleSave` dual-write: builds `legacyDone` (all 5 `RoomLegacyKey`s explicitly defaulted `false`, then set from `roomChecked` for any task with a matching `legacy_key`), passes them into the existing `api.saveDay` call's `roomBed/…/roomTrash` params, and adds `saveRoomChecks(childId, date, roomTasks.map(...))` (one row per rendered task) to the same `Promise.all`. A legacy task that's deactivated/not rendered writes `false` rather than letting `saveDay`'s `params ?? roomData?` fallback-merge resurrect a stale prior value (same pattern as Plan 04).
- Sick-day grace branch untouched: `dayType === 'sick'` still renders the grace note (🤒 badge, no checklist, no room penalty); `goodBehavior` sick auto-true handling, proof-photo display, the `/api/wallet/award` POST, and every non-room field unchanged.
