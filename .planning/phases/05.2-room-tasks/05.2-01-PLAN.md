Create the two new additive tables that make the room checklist family-configurable: `room_tasks`
(per-family checklist definition) and `room_checks` (per child+date completion), both family-isolated
via RLS. Seed the current default 5 tasks for every EXISTING family and provide a reusable
`seed_default_room_tasks(family_id)` function for new-family creation (wired in Plan 02). Harden the
dual-write era with a delete guard so the 5 legacy-mapped tasks can only be deactivated, never deleted.

Purpose: First de-hardcoding step of v5.0 — nothing about the checklist lives in code or fixed columns.
Output: One additive migration applied to the live DB + a verification script proving tables, RLS, seed
and the delete guard.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client (browser supabase, anon/JWT) → room_tasks/room_checks | Family members write config + day data directly; anon key is public |
| migration (service-role) → all families | Backfill loops over every family row |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-052-01 | Information Disclosure | room_tasks/room_checks SELECT across families | mitigate | RLS SELECT policy `family_id IN (SELECT family_id FROM family_members WHERE user_id = auth.uid())` — a family can only read its own rows; verify script confirms RLS enabled |
| T-052-02 | Tampering | Family A writing family B's tasks/checks | mitigate | RLS INSERT/UPDATE/DELETE WITH CHECK on the same family_id subquery (mirrors 04.4-01) |
| T-052-03 | Tampering | Deleting a legacy-mapped task and breaking dual-write/analytics | mitigate | BEFORE DELETE trigger `room_tasks_block_legacy_delete` blocks delete when legacy_key IS NOT NULL (deactivate-only) |
| T-052-04 | Elevation of Privilege | seed_default_room_tasks SECURITY DEFINER used to seed foreign family | mitigate | Function only INSERTs into room_tasks for the passed family_id; backfill is a one-time service-role migration statement, not client-callable with arbitrary side effects; idempotent guard prevents duplicate seeding |
| T-052-05 | Tampering | Backfill double-seeding on re-run | accept/mitigate | Idempotent seed (partial-unique-index / NOT EXISTS guard) — re-running the migration is safe; verify asserts exactly 5/family |
| T-052-SC | Tampering | `npm i pg --no-save` supply chain | accept | `pg` is the CLAUDE.md-sanctioned, already-used migration tool; pinned by npm resolution; no new runtime dependency added to the app bundle |
</threat_model>
