    Rewrite middleware.ts preserving all existing behavior and adding role-based routing. The final middleware must implement this exact logic in order:

    1. Call updateSession(request) → { supabaseResponse, user, supabase }
    2. Classify the path:
       - isPublicPath = pathname.startsWith('/auth') || pathname === '/'
       - isOnboardingPath = pathname.startsWith('/onboarding')
       - isParentPath = pathname.startsWith('/parent')
       - isKidPath = pathname.startsWith('/kid')
       - isFamilyPath = pathname.startsWith('/family')

    3. If !user and !isPublicPath and !isOnboardingPath → redirect to /  [UNCHANGED]

    4. If user is present and not on a public/onboarding path, fetch role:
       Query: supabase.from('family_members').select('id, role').eq('user_id', user.id).maybeSingle()
       Store result as: membership (may be null if no family yet)

    5. If membership is null (no family) and !isPublicPath and !isOnboardingPath → redirect to /onboarding  [UNCHANGED logic, just moved inside the role-fetch block]

    6. Role-based post-login redirect (REQ-ROLE-001):
       If pathname === '/':
         - If !membership → redirect to /onboarding
         - If membership.role === 'parent' → redirect to /parent/dashboard
         - If membership.role === 'child' → redirect to /kid/day
         - Otherwise (extended or unknown) → redirect to /dashboard (safe fallback)

    7. Route guards (only when membership exists):
       If isParentPath and membership.role !== 'parent' → redirect to /kid/day  [REQ-ROLE-002]
       If isKidPath and membership.role !== 'child' → redirect to /parent/dashboard  [REQ-ROLE-003]
       isFamilyPath → no redirect, allow through  [REQ-ROLE-004]

    8. Return supabaseResponse

    IMPORTANT implementation notes:
    - Single DB query per request (one .from('family_members') call — do NOT duplicate)
    - The role query replaces the old separate membership-existence query. Combine both checks into the one query by selecting 'id, role' (id presence confirms membership, role enables routing).
    - Keep the existing supabase client pattern: use the supabase instance returned by updateSession, not a new client
    - Do NOT import createClient from lib/supabase/client — middleware uses the server client from updateSession
    - The middleware.config matcher (if it exists) must NOT exclude /parent/*, /kid/*, or /family/* paths

    Verify the matcher in the existing middleware.ts export (at the bottom of the file) and preserve or update it so it covers the new route namespaces. Standard matcher pattern:

