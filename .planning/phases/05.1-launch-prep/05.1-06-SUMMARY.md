## Accomplishments

- `tests/integration/purchase.test.ts` — 6 tests under `describe.skipIf(!hasIntegrationEnv)`:
  1. Request deducts coins immediately (100 → 60) and lands `status: 'pending'`, `frozen_coins: 0`; exactly one `spend_coins` transaction (`coins_change: -40`) linked via `related_id`.
  2. Insufficient coins: requesting a 1000-coin reward against a 60-coin balance returns HTTP 400 `'Insufficient coins'`; balance unchanged; zero `reward_purchases` rows created for that reward.
  3. Approve keeps the deduction: `approvePurchaseAction` on the pending purchase (mocked parent caller) sets `status: 'approved'`, `processed_by: 'parent'`; wallet coins stay at 60 — no second deduction on the `frozen_coins=0` path.
  4. Reject refunds: a second request (60 → 20) followed by `rejectPurchaseAction` sets `status: 'rejected'`, refunds coins back to 60, and creates one `earn_coins` refund transaction (`coins_change: +40`).
  5. Non-pending guard: approving the already-approved purchase from Test 3 rejects with `'Purchase is not pending'`.
  6. Teardown: `destroyTestFamily` leaves zero `reward_purchases`/`wallet_transactions` rows for the test child.
- `tests/integration/exchange-withdraw.test.ts` — 7 tests under `describe.skipIf(!hasIntegrationEnv)`:
  1. Exchange math: 100-coin balance, exchanging 50 coins hits the `>=100` bonus tier (rate `10 * 1.1 = 11`), credits exactly 550 money; `coin_exchanges` row and `exchange` transaction both pin `exchange_rate: 11`, `bonus_rate: 10`.
  2. Overspend: exchanging 60 coins against a 50-coin balance returns 400 `'Insufficient coins'`, balances unchanged.
  3. Input validation: `coinsAmount` of `0`, `-5`, and `1.5` all return 400.
  4. Withdraw request is NOT a deduction: requesting a 200 withdrawal creates a `cash_withdrawals` row but wallet `money` stays at 550.
  5. Withdraw insufficient: requesting 10000 against a 550 balance returns 400 `'Insufficient money'`; no row created.
  6. **KNOWN GAP** (literal marker in-file): with the Test-4 pending 200 withdrawal still open, a second request for 400 still succeeds against the unchanged 550 balance — documents that the request route checks only current balance, not balance-minus-pending, because no approval/deduction endpoint exists to test instead.
  7. Teardown: zero `coin_exchanges`/`cash_withdrawals` rows remain for the test child.
- Full `npm run test:integration` run: **3 files / 18 tests, all green** (5 award + 6 purchase + 7 exchange-withdraw), ~18-19s wall time against the live rotated DB.
- Plain `npm test`: **3 passed | 3 skipped test files, 31 passed | 18 skipped tests** — all three integration suites correctly skip without env keys.
