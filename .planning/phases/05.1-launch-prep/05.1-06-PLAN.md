Key contracts (verified this session):

From app/api/wallet/_lib.ts processPurchase:
- Validates reward (family-scoped, is_active), debits price_coins atomically with `minCoins: 0` floor → 'Insufficient coins' AuthError(400) on overspend; inserts reward_purchases with status 'pending' (or 'approved' when reward.auto_approve), frozen_coins 0; inserts a spend_coins wallet_transactions row.

From app/parent/shop/actions.ts:
- `approvePurchaseAction(purchaseId)` — requireParent → pending guard → (frozen_coins=0 path: NO extra deduction) → status 'approved'. Dynamically imports '@/app/actions/push-notifications' (mock it) and calls insertAuditEvent from '@/lib/repositories/audit.repo' (mock it — it uses the browser supabase singleton).
- `rejectPurchaseAction(purchaseId, note?)` — pending guard → refunds price_coins via adjustCoins (frozen_coins=0 path) → status 'rejected'.

From app/api/wallet/exchange/route.ts POST:
- Body `{ childId, coinsAmount }` (positive integer). Rate = base_exchange_rate * (1 + bonus/100); bonus tier from CURRENT balance (>=100 → bonus_100_coins, >=500, >=1000). Defaults: base 10, bonus_100 10. Debits coins/credits money atomically with minCoins 0; inserts coin_exchanges + 'exchange' transaction. 'Insufficient coins' → 400.

From app/api/wallet/withdraw/route.ts POST:
- Body `{ childId, amount }`. Request only — NO balance change; inserts cash_withdrawals with balance_after_money = money - amt; 'Insufficient money' 400 when amt > balance. Deduction happens in /api/wallet/withdraw/approve (READ that route before writing tests — its status guard + floor is the double-spend protection under test).

Auth mocking pattern (established in plan 05 award.test.ts): partial vi.mock('@/lib/supabase/admin') overriding requireFamilyMember/requireParent; NextRequest handler invocation; fixture from tests/integration/family-fixture.ts.
