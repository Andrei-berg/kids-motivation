## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Added `experimental.instrumentationHook: true` to next.config.js**
- **Found during:** Task 2
- **Issue:** The plan's `<action>` did not mention this flag. On Next.js 14.2.35 (this project's pinned version), `instrumentation.ts` is silently ignored at build time unless `config.experimental.instrumentationHook` is `true` (confirmed by reading `node_modules/next/dist/build/index.js`, where the file-detection regexp is only added to the build's root-file scan when this flag is set). Without it, `instrumentation.ts` would exist but never run — server/edge Sentry init would never fire, silently defeating the entire plan.
- **Fix:** Added `experimental: { instrumentationHook: true }` to the `nextConfig` object before wrapping with `withSentryConfig`.
- **Files modified:** `next.config.js`
- **Verification:** `npm run build` output shows `Experiments (use with caution): · instrumentationHook` in the Next.js build banner, confirming the flag is active.
- **Committed in:** `ff0c98c` (Task 2 commit)

**2. [Rule 1 - Bug] Replaced deprecated `disableLogger: true` with `webpack.treeshake.removeDebugLogging: true`**
- **Found during:** Task 2 (surfaced by `npm run build` output: `[@sentry/nextjs] DEPRECATION WARNING: disableLogger is deprecated...`)
- **Issue:** The top-level `disableLogger` Sentry build option is deprecated in the installed 10.x SDK and scheduled for removal.
- **Fix:** Moved the same behavior to `webpack: { treeshake: { removeDebugLogging: true } }` per the SDK's own recommendation.
- **Files modified:** `next.config.js`
- **Verification:** Re-ran `npm run build` — deprecation warning no longer appears.
- **Committed in:** `ff0c98c` (Task 2 commit)

## User Setup Required

**External service (Sentry) requires manual dashboard configuration and env vars — no code change possible for this part.**

1. Create a Sentry project (platform: Next.js) on the free tier at sentry.io → Projects → Create project.
2. Copy the project DSN from Sentry → Project → Settings → Client Keys (DSN).
3. Add to Vercel project env (Production, and Preview/Development as desired):
   - `NEXT_PUBLIC_SENTRY_DSN` = the DSN
   - `SENTRY_DSN` = the same DSN value (used server/edge-side)
4. Redeploy.
5. **Manual prod capture confirmation (not yet performed — requires the DSN to exist first):** after deploy with the DSN set, visit `https://kids-motivation.vercel.app/api/health?boom=1` once. It should return HTTP 500, and a "Sentry test error: /api/health?boom=1" event should appear in the Sentry Issues stream within a minute or two. This step is intentionally left for the operator to run after adding the env vars above — the executor has no way to add secrets to Vercel or confirm delivery to a Sentry project it cannot access.
6. Optional (source maps): if later desired, add `SENTRY_ORG`, `SENTRY_PROJECT`, and `SENTRY_AUTH_TOKEN` to Vercel — `next.config.js` already gates the upload on `SENTRY_AUTH_TOKEN`'s presence, so no code change is needed to turn it on.
