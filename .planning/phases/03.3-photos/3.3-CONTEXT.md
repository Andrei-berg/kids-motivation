# Phase 3.3: photos - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Family members can share photos in the family chat. Children can attach a photo as proof when completing room cleaning; parent sees the proof inline before approving or rejecting the task. This phase introduces Supabase Storage for the first time (no file upload infrastructure exists yet).

</domain>

<decisions>
## Implementation Decisions

### Photo attach UX in chat
- Two separate buttons alongside the sticker button (🎭): one for camera, one for gallery
- After selecting a photo, show a preview above SendBox with a send button — user can cancel before sending
- Optional caption field in the preview step; if provided, it renders below the image in the thread
- Client-side compression before upload: **30% JPEG quality** (very aggressive — keeps file sizes tiny); Claude decides max dimension

### Image rendering in chat thread
- Inline thumbnail: ~150px wide (compact, doesn't dominate the scroll)
- Tap photo → full-screen lightbox with a close button
- Caption renders below the thumbnail, inside the message bubble
- Photo messages support the same reaction bar (❤️ 👍 🔥 🏆) as text messages — `ReactionPickerBar` renders under them

### Task proof UX (kid side)
- Photo proof applies to **room cleaning only** (not grades, sport, or section visits)
- Proof is **optional** — kid can submit room cleaning without a photo; no friction
- Camera-first: tap to shoot in-the-moment. No gallery picker for proof (reduces ability to pick old photos)
- After capture: small thumbnail + retake button appears in the room cleaning section of the day-fill form

### Parent proof review
- Proof photo appears **inline in the existing parent daily confirmation view** — no new screen
- Tap the proof thumbnail → full-screen lightbox (same behavior as chat photos)
- If no proof provided: no photo section shown (no "No proof provided" label — proof is optional so absence is normal)
- Storage retention: keep photos permanently by default

### Claude's Discretion
- Max pixel dimension for client-side compression (user only specified 30% quality)
- Exact lightbox implementation (overlay + backdrop, close on tap outside)
- Supabase Storage bucket structure (e.g., `family-photos/{family_id}/chat/` vs `task-proof/{family_id}/`)
- Signed URL expiry duration (short enough for security, long enough for scrolling through chat)
- Upload progress indicator design

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `components/chat/ChatThread.tsx` — message list with `message_type` switch. Currently handles `text`, `sticker`, `system`. Add `photo` case here.
- `components/chat/SendBox.tsx` — currently text-only. Camera/gallery buttons will be added alongside existing sticker button (which is in ChatThread, not SendBox).
- `components/chat/MessageReactions.tsx` / `ReactionPickerBar` — already used for text and sticker messages; extend to photo messages with no changes.
- `lib/models/chat.types.ts` — `MessageType = 'text' | 'sticker' | 'system'`. Needs `'photo'` added. `ChatMessage` needs a `photo_url: string | null` field.

### Established Patterns
- Real-time subscriptions via `subscribeToMessages` and `subscribeToReactions` in `lib/repositories/chat.repo` — photo messages flow through same channel; no new subscription needed.
- Optimistic UI: sticker messages are inserted optimistically then replaced on save — same pattern for photo messages (insert with local blob URL, replace with signed URL after upload).
- Tailwind CSS for styling — no custom CSS classes needed for this phase; use Tailwind directly.

### Integration Points
- `KidDayFillForm` in `components/kid/KidDayFillForm.tsx` — room cleaning section is where the proof photo attach UI goes.
- Parent daily confirmation view (in `app/parent/daily/page.tsx` or parent dashboard) — proof photo appears inline here.
- Supabase Storage: new resource to provision. Will need bucket policies (RLS equivalent for Storage) and signed URL generation.
- `lib/repositories/chat.repo` — `sendMessage()` will need to accept a `photoUrl` param for photo messages.

</code_context>
