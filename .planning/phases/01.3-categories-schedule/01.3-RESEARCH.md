## Summary

Phase 1.3 has three interlocking domains: (1) a new Supabase schema for categories/tasks/schedule, (2) a complete redesign of the settings page UI, and (3) Web Push API integration via service worker. The schema work is the critical foundation because every other piece depends on it. The settings UI redesign is substantial but follows established project patterns. Web Push requires adding `web-push` npm, generating VAPID keys, creating `public/sw.js`, and `app/manifest.ts` — all of which Next.js 14 supports natively without third-party PWA plugins.

The existing codebase has NO `categories` or `tasks` tables yet — the Phase 1.2 onboarding wizard wrote categories as UI-only (no DB writes). Phase 1.3 must create these tables from scratch. The existing `schedule` table (used by `flexible-api.ts` / `ScheduleEditor.tsx`) only covers school lessons (Mon–Fri, 8 slots). The new unified schedule must extend to cover lessons, sections, and routine slots with push reminder times. The old `ScheduleEditor.tsx` component and the old `schedule` table are superseded — the new design builds from scratch.

Push notifications for web: the official Next.js docs (updated 2026-02-27) document the full pattern: `app/manifest.ts`, `public/sw.js`, `web-push` npm with Server Actions, VAPID keys. iOS 16.4+ supports Web Push for PWAs installed to Home Screen. This pattern is stable and well-documented.

**Primary recommendation:** Build the SQL migration first (categories, tasks, schedule_items, push_subscriptions), then the API layer (`lib/categories-api.ts`), then the settings page redesign, then the push notification plumbing. Each plan is a clean dependency chain.

### Pattern 1: Categories + Tasks Schema (family-scoped)
**What:** Two new tables — `categories` (family-level) and `tasks` (category-level, optionally child-specific). Both scoped by `family_id` for RLS isolation.
**When to use:** Every read/write in categories-api.ts

```sql
-- Source: ARCHITECTURE.md schema design + existing RLS pattern from rls.sql
CREATE TABLE public.categories (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  family_id   UUID NOT NULL REFERENCES families(id) ON DELETE CASCADE,
  name        TEXT NOT NULL,
  icon        TEXT NOT NULL DEFAULT '📋',
  color       TEXT,
  type        TEXT NOT NULL CHECK (type IN ('study','home','sport','routine','custom')),
  is_active   BOOLEAN DEFAULT true,
  is_default  BOOLEAN DEFAULT false,
  sort_order  INT DEFAULT 0,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE public.tasks (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  family_id         UUID NOT NULL REFERENCES families(id) ON DELETE CASCADE,
  category_id       UUID NOT NULL REFERENCES categories(id) ON DELETE CASCADE,
  child_member_id   UUID REFERENCES family_members(id) ON DELETE CASCADE,  -- NULL = all children
  title             TEXT NOT NULL,
  description       TEXT,
  coins_reward      INT NOT NULL DEFAULT 0,
  coins_penalty     INT DEFAULT 0,
  is_required       BOOLEAN DEFAULT false,
  is_active         BOOLEAN DEFAULT true,
  reminder_time     TIME,   -- e.g. '07:30' — null means no reminder
  monthly_cap       INT,    -- null means no cap; for grade tasks this is always null
  sort_order        INT DEFAULT 0,
  created_at        TIMESTAMPTZ DEFAULT NOW()
);
```

### Anti-Patterns to Avoid
- **Using old `lib/supabase.ts` for new API functions** — Use `createClient()` from `lib/supabase/client.ts` instead. The new client pattern (from Phase 1.1) handles auth cookies correctly.
- **Storing push subscriptions in module-level variable** — The Next.js docs example uses a module variable (for demos). Production: store in `push_subscriptions` Supabase table.
- **Using `schedule` table for new schedule items** — Old `schedule` table has no `family_id`, no `type`, no `location`. Create new `schedule_items` table. Old table stays for backward compat until Phase 1.4.
- **Hardcoding familyId** — Always get familyId from Supabase session via `supabase.auth.getUser()` + a DB lookup for `family_members`.
- **Blocking signup with DB trigger side effects** — Pattern established in Phase 1.1: wrap trigger logic in EXCEPTION block.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Push notification delivery | Custom WebSocket/SSE push system | `web-push` npm + VAPID | Handles VAPID signing, GCM, encryption, retry; official Next.js recommendation |
| PWA manifest serving | Manual JSON file in public/ + link tag | `app/manifest.ts` (Next.js built-in) | Type-safe, auto-served at /manifest.webmanifest, integrates with App Router metadata |
| Service Worker registration | Complex registration logic | Single `navigator.serviceWorker.register('/sw.js')` call | Browser handles lifecycle; no framework needed |
| RLS family isolation | Custom auth checks in API code | Supabase RLS with `get_my_family_ids()` helper (already established) | Already implemented in Phase 1.1; reuse identical policy pattern |
| Default category seeding | Complex migration trigger | Simple INSERT call from `onboarding-api.ts` after createFamily | Already have the function call site; seeds are static data |

**Key insight:** Web Push is deceptively complex internally (VAPID signing, encryption, push service protocols) but `web-push` npm wraps all of it in a single `sendNotification()` call. Never build push delivery from scratch.

## Common Pitfalls

### Pitfall 1: iOS Push Requires PWA Installation First
**What goes wrong:** Push permission request silently fails on iOS Safari if the app is not installed to Home Screen.
**Why it happens:** iOS 16.4+ only supports Web Push for PWAs in standalone mode (installed to Home Screen), not for browser tabs.
**How to avoid:** Show `InstallPrompt` component that detects iOS + non-standalone mode and instructs user to install first. Only show push subscribe button after `window.matchMedia('(display-mode: standalone)').matches`.
**Warning signs:** `PushManager` is undefined on iOS Safari in browser tab (not an error — just not supported).

### Pitfall 2: VAPID Keys Not in Environment on Vercel
**What goes wrong:** Push notifications work locally but fail in production with "invalid VAPID" error.
**Why it happens:** VAPID_PRIVATE_KEY and NEXT_PUBLIC_VAPID_PUBLIC_KEY not added to Vercel project settings.
**How to avoid:** Document both env vars in `.env.local.example`. Add to Vercel dashboard before deployment. NEXT_PUBLIC_ prefix required for the public key (used in client JS).
**Warning signs:** `webpush.sendNotification` throws `Error: Invalid VAPID key`.

### Pitfall 3: Service Worker Scope and Caching
**What goes wrong:** Old service worker serves stale push handler after update.
**Why it happens:** Browser caches service worker. Without `Cache-Control: no-cache` header on `/sw.js`, old version may persist.
**How to avoid:** Add service worker headers in `next.config.js` as shown in official docs. Use `updateViaCache: 'none'` in `navigator.serviceWorker.register()` call.
**Warning signs:** Code changes to sw.js don't take effect after reload.

### Pitfall 4: Supabase RLS on New Tables Blocks Writes
**What goes wrong:** Inserting categories or tasks returns 403 permission denied.
**Why it happens:** New tables need explicit RLS policies. Tables created without `ENABLE ROW LEVEL SECURITY` default to allow-all for superuser but block anon/authenticated.
**How to avoid:** Migration SQL must include both `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` AND `CREATE POLICY` for each new table. Use same `family_id IN (SELECT get_my_family_ids())` pattern from `rls.sql`.
**Warning signs:** Supabase returns `{ error: { code: '42501', message: 'new row violates row-level security policy' } }`.

### Pitfall 5: Monthly Cap Logic Complexity
**What goes wrong:** Monthly cap computed incorrectly — cap applies per task per child per calendar month. If enforced at DB level (CHECK constraint), it requires aggregating prior transactions. If enforced in API, race conditions possible.
**Why it happens:** Monthly rollups require summing coins_earned for the current month across task_completions.
**How to avoid:** Enforce cap in API code (not DB constraint). On task completion, sum existing coins for that task+child+month, compare to cap, reject or warn if exceeded. Claude's discretion: hard stop is simpler to implement; warning is better UX.
**Warning signs:** Children accumulating unlimited coins from a single repeated task.

### Pitfall 6: Old schedule Table Conflict
**What goes wrong:** New `schedule_items` queries accidentally target the old `schedule` table (still used by `flexible-api.ts`).
**Why it happens:** Both tables exist simultaneously during transition. TypeScript imports from wrong module.
**How to avoid:** Name new API file `lib/schedule-api.ts` (distinct from `lib/flexible-api.ts`). New `ScheduleEditor` component imports from `schedule-api.ts` only. Old `ScheduleEditor.tsx` → rename to `ScheduleEditor-old.tsx` or delete.
**Warning signs:** Schedule changes in new UI don't appear; data going to wrong table.

### Pitfall 7: familyId Not Available in Client Components
**What goes wrong:** Settings page tries to call categories-api.ts but doesn't have the familyId.
**Why it happens:** Phase 1.3 adds real multi-tenant data. The old store only has `childId: 'adam'|'alim'`.
**How to avoid:** Settings page reads familyId from Supabase session on mount: `supabase.auth.getUser()` → query `family_members` to get `family_id`. Store in component state. Extend Zustand store with `familyId` (initialized on app load in layout or dashboard).
**Warning signs:** `family_id: null` or `family_id: undefined` in INSERT statements → RLS failure.

import { createClient } from '@/lib/supabase/client'

export async function getCurrentFamilyId(): Promise<string | null> {
  const supabase = createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return null

  const { data } = await supabase
    .from('family_members')
    .select('family_id')
    .eq('user_id', user.id)

### categories-api.ts CRUD skeleton
```typescript
// lib/categories-api.ts
import { createClient } from '@/lib/supabase/client'

export interface Category {
  id: string
  family_id: string
  name: string
  icon: string
  color: string | null
  type: 'study' | 'home' | 'sport' | 'routine' | 'custom'
  is_active: boolean
  is_default: boolean
  sort_order: number
}

export interface Task {
  id: string
  family_id: string
  category_id: string
  child_member_id: string | null  // null = all children
  title: string
  coins_reward: number
  coins_penalty: number
  is_required: boolean
  is_active: boolean
  reminder_time: string | null  // e.g. "07:30:00"
  monthly_cap: number | null
}

export async function getCategories(familyId: string): Promise<Category[]> {
  const supabase = createClient()
  const { data, error } = await supabase
    .from('categories')
    .select('*')
    .eq('family_id', familyId)
    .order('sort_order')
  if (error) throw error
  return data
}

export async function createCategory(
  familyId: string,
  fields: Pick<Category, 'name' | 'icon' | 'type'>
): Promise<Category> {
  const supabase = createClient()
  const { data, error } = await supabase
    .from('categories')
    .insert({ family_id: familyId, ...fields })
    .select()
    .single()
  if (error) throw error
  return data
}

export async function getTasks(familyId: string, categoryId: string): Promise<Task[]> {
  const supabase = createClient()
  const { data, error } = await supabase
    .from('tasks')
    .select('*')
    .eq('family_id', familyId)
    .eq('category_id', categoryId)
    .order('sort_order')
  if (error) throw error
  return data
}
```

// Source: https://nextjs.org/docs/app/guides/progressive-web-apps (2026-02-27)
function urlBase64ToUint8Array(base64String: string) {
  const padding = '='.repeat((4 - (base64String.length % 4)) % 4)
  const base64 = (base64String + padding).replace(/-/g, '+').replace(/_/g, '/')
  const rawData = window.atob(base64)
  const outputArray = new Uint8Array(rawData.length)
  for (let i = 0; i < rawData.length; ++i) {
    outputArray[i] = rawData.charCodeAt(i)
  }
  return outputArray
}

async function subscribeToPush(memberId: string) {
  const registration = await navigator.serviceWorker.ready
  const sub = await registration.pushManager.subscribe({
    userVisibleOnly: true,
    applicationServerKey: urlBase64ToUint8Array(
      process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!
    ),
  })
  // Save to Supabase push_subscriptions table
  await savePushSubscription(memberId, JSON.parse(JSON.stringify(sub)))
}

## Open Questions

1. **Where to trigger push sends (scheduled reminders)**
   - What we know: Web Push requires a server to call `webpush.sendNotification()`. A Server Action can do this but needs to be triggered somehow.
   - What's unclear: The CONTEXT.md says "reminder time is set per task" — but what triggers the send? Vercel doesn't have built-in cron in free tier. Supabase Edge Functions can schedule. Or: a Next.js API route polled by a background job.
   - Recommendation: For Phase 1.3, implement push subscription storage + manual "test send" button in settings. Scheduled delivery (cron-triggered) is Phase M2 infrastructure. Document the `app/api/push/send/route.ts` stub so Phase M2 can hook into it.

2. **familyId in Zustand store**
   - What we know: Current store has `childId: 'adam'|'alim'` (legacy). Phase 1.3 needs real UUIDs.
   - What's unclear: Should Phase 1.3 extend the store, or should settings page fetch familyId on mount only?
   - Recommendation: Extend Zustand store with `familyId: string | null` in Phase 1.3. Initialize in `app/layout.tsx` using a `useEffect` that calls `supabase.auth.getUser()`. This sets up Phase 1.4 dashboard refactor cleanly.

3. **Old `schedule` table backward compatibility**
   - What we know: `flexible-api.ts` and old `ScheduleEditor.tsx` use the old `schedule` table. Phase 1.4 will refactor dashboard.
   - What's unclear: Should Phase 1.3 delete the old table or keep it?
   - Recommendation: Keep old `schedule` table untouched. Create new `schedule_items` table alongside it. Phase 1.4 will migrate data and remove old table. This avoids breaking current dashboard behavior.
