# Phase 2.6: registration — Context

**Gathered:** 2026-04-11
**Status:** Ready for planning
**Source:** Direct analysis + conversation with product owner

<domain>
## Phase Boundary

This phase makes the app usable for real new families — not just adam/alim. Currently:
- Onboarding wizard (v2) is a UI prototype that logs config to console and redirects to /dashboard without writing anything to DB
- `children` table is never populated for new families (adam/alim rows exist from manual setup)
- `wallets` are never created for new children
- `family_members.child_id` is never populated during registration (just fixed for existing users)
- New families get broken experience: KidLayout can't resolve child, wallet pages crash, all data queries fail

Phase 2.6 wires up the full registration pipeline so any new family can onboard end-to-end.

</domain>

<decisions>
## Implementation Decisions

### Registration Entry Point
- Keep /register page (Google OAuth + email/password already works)
- After auth, detect if user has a family_members row → if not, redirect to /onboarding
- If user already has family_members row → redirect normally (parent → /parent/dashboard, child → /kid/day)

### Parent Onboarding Wizard (replace /onboarding/v2)
- Replace the console.log stub with real DB writes
- Steps: 1) Family name 2) Add children (1-4) 3) Coin rules (can use defaults) 4) Done
- Keep it SHORT — target 3 minutes. No calendar/schedule/notifications in wizard (those are in Settings)
- On "Done" button: execute all DB writes atomically, then redirect to /parent/dashboard

### DB writes on wizard completion
Order matters (FK constraints):
1. INSERT families (name, invite_code = random 6-char) → get family_id
2. UPDATE family_members SET family_id for parent (already exists from auth callback)
3. For each child:
   a. INSERT children (id = slugify(name + random suffix), name, family_id) → get child text id
   b. INSERT family_members (family_id, role='child', display_name=name, child_id=that text id, user_id=null)
   c. INSERT wallets (child_id, family_id, coins=0)
4. INSERT wallet_settings defaults (coins_per_grade_5=5, etc.) for this family

### Child ID format for new families
- Old format: 'adam', 'alim' (manual, no collision protection)
- New format: slug from name + 4-char random suffix → 'sofia_a1b2', 'max_c3d4'
- Max 20 chars, lowercase, alphanumeric + underscore only
- Must be unique in children table (retry with new suffix on collision)

### Invite Code flow (already partially built in /onboarding/join)
- Parent shares invite_code from family row (shown on wizard "Done" screen)
- Child goes to /onboarding/join, enters 6-digit code
- claim_child_profile RPC already exists — links user_id to the pre-created family_members row
- After claim: redirect to /kid/day (KidLayout will resolve child_id correctly now)
- Invitation page needs to handle: code not found, already claimed, success states

### Auth callback routing
- /auth/callback already redirects parent → /parent/dashboard, child → /kid
- Need to add: if new user (no family_members row) → /onboarding
- Check: SELECT COUNT(*) FROM family_members WHERE user_id = auth.uid()

### Wizard UX (keep simple)
- No tabs, single-scroll per step with Next/Back
- Inline validation (name required, max 4 children)
- Show invite code prominently on done screen with copy button
- Mobile-first (parents use phone)

### PIN login for children without email (ADDED)

Children 5-12 лет часто не имеют email. Нужен альтернативный вход через PIN.

**Механизм:**
- При создании ребёнка в wizard (шаг 2) родитель задаёт 4-значный PIN (опционально, можно пропустить и задать позже в /parent/settings)
- PIN хранится как SHA-256 хеш в `family_members.child_pin_hash` (новая колонка)
- При установке PIN: создаётся Supabase auth user с синтетическим email `child_{child_id}@internal.familycoins.app` + password = PIN (plaintext при создании, потом меняется на хеш в auth)
- Фактически: `supabase.auth.signUp({ email: synthetic, password: pin })` — один раз при установке PIN
- Вход ребёнка: `supabase.auth.signInWithPassword({ email: synthetic, password: pin })`

**Новая страница `/kid/login`:**
```
Шаг 1: Введи код семьи (6 букв/цифр)
Шаг 2: Выбери своё имя из списка
Шаг 3: Введи PIN (4 цифры, большие кнопки)
→ Войти → /kid/day
```

**DB migration нужна:**
```sql
ALTER TABLE family_members ADD COLUMN child_pin_hash TEXT;
```

**Две точки входа для детей:**
1. `/onboarding/join` — для детей с email (Google/email auth + код)
2. `/kid/login` — для детей без email (только код семьи + PIN)

**На экране "Готово" wizard:**
- Показывать для каждого ребёнка: "Есть email? → поделись кодом. Нет email? → задай PIN прямо сейчас"

**Новый план 02.6-04 нужен для:**
- DB migration (child_pin_hash колонка)
- `setChildPin(memberId, pin)` функция в onboarding-api.ts
- `/kid/login` страница (3 шага)
- Ссылка "Войти как ребёнок" на /login странице

- `family_members`: id, family_id, user_id, role, display_name, child_id (TEXT ref children.id)

After this runs, KidLayout reads child_id directly — no name matching needed.

### Auth callback current code
Located at: app/auth/callback/route.ts
Currently: redirects to /parent/dashboard or /kid based on role
Need to add: check if user has family → if not → /onboarding

### Slug generation for child IDs
```ts
function generateChildId(name: string): string {
  const slug = name.toLowerCase().replace(/[^a-z0-9]/g, '_').slice(0, 12)
  const suffix = Math.random().toString(36).slice(2, 6)
  return `${slug}_${suffix}`
}
```

### Invite code format
6 uppercase alphanumeric: 'A3F9K2'
Already generated and stored in families.invite_code
Show on wizard completion screen
