# Phase 02.6 Plan 04: PIN Login for Children Summary

**One-liner:** PIN-based child login with synthetic Supabase email auth, SHA-256 hash storage, and 3-step /kid/login page (family code + name picker + PIN entry).

## What Was Built

Children without email addresses can now log in using a family code + their name + a 4-digit PIN. The flow:

1. Parent sets PIN via `setChildPin(memberId, childId, pin)` which calls `/api/set-child-pin`
2. `/api/set-child-pin` (service-role) creates a Supabase auth user with synthetic email `child_{childId}@internal.familycoins.app` and the PIN as password, then stores a SHA-256 hex digest of the PIN in `family_members.child_pin_hash`
3. Child opens `/kid/login`, enters family code → picks their name → enters PIN
4. Login calls `supabase.auth.signInWithPassword` with the synthetic email + PIN
5. On success: `setFamilyId` + `setActiveMemberId(child_id)` set, redirect to `/kid/day`

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `getUserByEmail` does not exist on GoTrueAdminApi**
- **Found during:** Task 1
- **Issue:** Plan specified `adminClient.auth.admin.getUserByEmail(syntheticEmail)` but the installed version of `@supabase/supabase-js` has no such method — only `listUsers`, `createUser`, `updateUserById` exist on `GoTrueAdminApi`
- **Fix:** Changed to try `createUser` first; on "already been registered" / "already exists" error, fall back to `listUsers` (page=1, perPage=1000) to find the existing user by email, then call `updateUserById`
- **Files modified:** app/api/set-child-pin/route.ts
- **Commit:** 6a28901

## Key Decisions

| Decision | Choice | Reason |
|----------|--------|--------|
| Existing-user detection | Try createUser, catch "already registered", fall back to listUsers | getUserByEmail absent from installed @supabase/auth-js version |
| activeMemberId value | child_id TEXT slug (e.g. 'sofia_a1b2') | Kid pages (day, wallet, shop) pass activeMemberId to api.getChild/getWallet which expect TEXT child id |
