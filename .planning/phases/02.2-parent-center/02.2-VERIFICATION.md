## Gaps Summary

Phase 02.2 delivered the structural parent control center (all 6 /parent/* routes), navigation shell, and most management features. The critical gap is in the **coin economy wiring**: while the daily input form (DailyModal) correctly saves day/grade/exercise data to the database, it does NOT credit coins to children's wallets for those activities. The `awardCoinsForGrade`, `awardCoinsForRoom`, `awardCoinsForBehavior`, and `awardCoinsForSport` functions in `wallet.repo.ts` exist and are exported but are never called during the save flow in `components/DailyModal.tsx`. This means:

1. **Wallet balances never increase from daily activity** — the core incentive mechanic is broken
2. **The coin rules saved in /parent/settings have no effect** on grade coin awards (awardCoinsForGrade is hardcoded)
3. **Transaction history** (REQ-COIN-011) is incomplete — only manual adjustments and streak bonuses appear

Secondary gaps: REQ-DAY-008/009/010 require a child self-marking → parent confirmation workflow that was never built (the plan incorrectly claimed these via DailyModal which is a direct parent entry form); REQ-PARENT-006 settings page lacks category management and family settings sections; auto_approve for shop items needs a DB migration.

These gaps share a root cause: the plan assumed "DailyModal handles ALL coin logic internally via awardCoinsForGrade etc." but this was never true — those functions are utility helpers that require explicit calls, not automatic hooks.
