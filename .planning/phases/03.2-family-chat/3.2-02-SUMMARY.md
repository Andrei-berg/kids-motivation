## Decisions Made

- ChatThread accepts props (not context/hooks) for familyId and member info — keeps it portable for plan 3.2-03 sticker extension
- Optimistic update uses `id: optimistic-{timestamp}` prefix; Realtime dedup checks `prev.some(m => m.id === msg.id)` to skip when real message arrives with same content but different ID
- Duplicate prevention guard added in subscribeToMessages callback (not in plan) — required for correct behavior with optimistic inserts

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Added duplicate guard in Realtime subscription callback**
- **Found during:** Task 1 (ChatThread component)
- **Issue:** Without dedup, optimistic message + Realtime INSERT would show the same message twice
- **Fix:** Added `if (prev.some((m) => m.id === msg.id)) return prev` check in subscription handler
- **Files modified:** components/chat/ChatThread.tsx
- **Verification:** Logic prevents double-render for optimistic messages; TypeScript clean
- **Committed in:** 1df521e (Task 1 commit)

**2. [Rule 1 - Bug] Fixed kid chat page querying family_members by wrong column**
- **Found during:** Post-checkpoint verification
- **Issue:** Kid page queried `.eq('child_id', activeMemberId)` but activeMemberId from Zustand is family_members.id, not child_id
- **Fix:** Changed query to `.eq('id', activeMemberId)`
- **Files modified:** app/kid/chat/page.tsx
- **Verification:** Kid chat page correctly loads family context after fix
- **Committed in:** c5a851d

**3. [Rule 1 - Bug] Fixed duplicate messages in kid view after optimistic send**
- **Found during:** Task 3 human verification
- **Issue:** After sending, message appeared twice — once from optimistic state and once from Realtime event
- **Fix:** Replace optimistic message with real message from sendMessage response before Realtime fires; retain dedup guard in subscription callback
