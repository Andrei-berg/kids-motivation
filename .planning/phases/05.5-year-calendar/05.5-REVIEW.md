# Phase 5.5: Code Review Report

**Reviewed:** 2026-07-13T09:30:00Z
**Depth:** standard
**Files Reviewed:** 30

## Summary

Phase 5.5 (year-calendar) was reviewed adversarially at standard depth, with git-history
cross-checks to separate phase-introduced defects from pre-existing ones, and locale JSON /
identity-space verification across module boundaries.

**Security invariants verified as HOLDING:**
- No client-side wallet writes were reintroduced. All coin mutations still flow through
  `/api/wallet/award` (service-role, guarded by `requireFamilyMember` + `assertChildInFamily`).
- No client-side streak writes remain: `updateStreaks(admin, …)` is called only from
  `app/api/wallet/award/route.ts` and `app/actions/repair-achievements.ts`; migration
  `05.5-03` drops all write policies on `streaks`, and `scripts/verify-streaks-rls.mjs`
  verifies read-allowed / write-denied end-to-end.
- `localDateString()` is used for "today" in all reviewed app code (the raw
  `toISOString().slice(0,10)` in `verify-streaks-rls.mjs` is a service-role script, not
  app logic — acceptable, but see WR-06 for a real server/client timezone gap).
- The streak-bonus clamps (`clampStreakDays`, `clampStreakBonus`) correctly prevent
  0/negative thresholds and negative/unbounded bonuses.

**However, three Critical defects were found:** the entire new Calendar settings screen
renders raw i18n keys (wrong namespace); the D-09/D-10 "transparency" feature emits a
spurious streak-broken event (and push notification) whenever the save happens ON the
transparent day itself — the most common case; and the streak calculators can never
produce `current_count > 1`, which makes the newly ported server-side streak bonus dead
code at default thresholds (the phase's own integration test had to set
`streak_room_days: 1` to make the bonus fire).

## Critical Issues

### CR-01: CalendarSettingsManager uses wrong i18n namespace — entire new screen renders raw keys

**File:** `components/settings/CalendarSettingsManager.tsx:85,89,113,116,127,142,154,169,182,191,207,217,227,242`
**Issue:** Every translation call in this component uses the `calendarSettingsManager.*`
namespace (e.g. `t('calendarSettingsManager.title')`), but the keys live under
`settings.calendarSettingsManager.*` in both `public/locales/en.json` (line 722, nested
inside `"settings"`) and `ru.json`. There is **no** top-level `calendarSettingsManager`
object in either locale file (verified programmatically). `lib/i18n.tsx`'s
`getNestedValue` returns the key itself on a miss, so the whole Settings → Schedule →
Calendar screen renders literal strings like `calendarSettingsManager.title`,
`calendarSettingsManager.days.sun`, and validation errors show
`calendarSettingsManager.dateOrderError`. Compare `PeriodsManager.tsx`, which correctly
uses the `settings.periodsManager.*` prefix.
**Fix:**
```tsx
// Every call in CalendarSettingsManager.tsx:
t('settings.calendarSettingsManager.title')
t(`settings.calendarSettingsManager.${mode}`)
t(`settings.calendarSettingsManager.days.${WEEKDAY_I18N_KEYS[idx]}`)
// …etc. (or move the block to top level in both locale files — one or the other)
```

### CR-02: Transparency fails when "today" itself is the transparent day — spurious streak-broken events + push notifications

**File:** `lib/services/streaks.service.ts:110-135, 145-171, 179-204, 212-237` (all four calculators) and `240-263` (`updateStreak`)
**Issue:** The calculators assign `current` only during the iteration where
`checkDate === today`. When today itself is transparent (sick for room/sport/behavior;
weekend/vacation/sick for study), that iteration is `continue`d, so `current` stays at
its initial `0`. `updateStreak` then sees `existing.current_count > 0 && current === 0`
→ writes `current_count = 0` **and emits a `'broken'` event**, which
`KidDayFillForm`/`DailyModal` forward to `notifyStreakEvents` as a "streak broken" push.
Concretely: a child with an active study streak saves their day on **Saturday** (or any
configured weekend/vacation day) → the streak is zeroed in the DB and a break
notification fires — exactly the behavior D-09/D-10 was specified to prevent ("a
weekend/vacation/sick day must not increment **or reset** a streak", the module's own
comment at line 68). The unit tests (`tests/streaks.test.ts:163-225`) only cover
transparent days *between* today and past active days, never a transparent *today*, so
the gap is untested.
**Fix:** Track whether the today-anchored run is still open instead of keying on
`checkDate === today`, and when today is transparent, leave `current` at the existing
streak value (carry the run through). Sketch for `calculateStudyStreak` (same shape for
the other three):
```ts
let anchored = true // still in the run that (transparently) extends back from today
for (let i = 0; i < 30; i++) {
  const type = dayType?.(checkDate)
  if (type === 'weekend' || type === 'vacation' || type === 'sick') {
    checkDate = addDays(checkDate, -1)
    continue // transparent: does not close the anchored run
  }
  if (daysWithGrades.has(checkDate)) {
    streak++
    if (anchored) current = streak
  } else {
    if (streak > best) best = streak
    anchored = false
    streak = 0
  }
  checkDate = addDays(checkDate, -1)
}
```
(This also fixes CR-03 below — see note there.) Additionally, `updateStreak` should not
emit `'broken'` when the day being processed was transparent.

### CR-03: `current_count` can never exceed 1 — server-side streak bonus (D-12) is unreachable at default thresholds; milestones and streak_30 badge dead

**File:** `lib/services/streaks.service.ts:121-123` (and the identical pattern in all four calculators); `app/api/wallet/award/route.ts:282-303` (`computeStreakBonus`); `tests/streaks.test.ts:49-57`
**Issue:** The calculators walk **backwards** from today and assign
`current = streak` only on the first (today) iteration, when `streak` is at most 1.
`current_count` written to the DB is therefore always 0 or 1, on every recompute. The
phase's own tests codify this ("current is 1 (active) or 0 … never the full length",
`tests/streaks.test.ts:50-53`). Consequences for code shipped in this phase:
- `computeStreakBonus` compares `current_count >= streak_room_days` (default 7),
  `streak_study_days` (14), `streak_sport_days` (7) — with `current_count ≤ 1` the
  bonus **never fires** unless a family manually sets a threshold to 1. The integration
  test `tests/integration/award.test.ts:262-336` had to set `streak_room_days: 1` to get
  the bonus to pay — evidence the default-config path is dead.
- The milestone chat hook `[7, 14, 30].includes(current)` (`streaks.service.ts:293`)
  never fires.
- `checkStreak30` (`badges.service.ts:504-519`, `current_count >= 30`) can never award,
  while `getBadgeProgress` (`badges.service.ts:375,387`) shows `streak_30` progress from
  `best_count` — the progress bar can display 30/30 for a badge that cannot be earned.
- The kid Day page streak flame (`app/kid/day/page.tsx:132`, max of `current_count`)
  is capped at 1.
The `current`-assignment pattern predates this phase (verified via
`git diff c4c7265^ c4c7265`), but this phase built `computeStreakBonus`, the D-12.1
today-gate, and the streakEvents push pipeline on top of it and enshrined the broken
semantics in new tests — the shipped feature does not work.
**Fix:** Make `current` the length of the (transparency-bridged) run ending at the most
recent non-transparent day — the `anchored` sketch in CR-02 produces exactly that.
Update `tests/streaks.test.ts` assertions from `current: 1` to the full run length
(e.g. 5 in the 5-day test), and re-verify `computeStreakBonus` against default
thresholds.

## Warnings

### WR-01: `family_calendar` is writable by `role='child'` members

**File:** `supabase/migrations/05.5-01-family-calendar.sql:48-81`
**Issue:** The INSERT/UPDATE/DELETE policies check only family membership
(`family_id IN (SELECT family_id FROM family_members WHERE user_id = auth.uid())`), not
role. A PIN-child can, via DevTools/REST, rewrite `weekend_days`, `year_start/year_end`,
and `region_preset` for the whole family. Since 05.5-02, these values feed the
**server-side** streak computation (`updateStreaks` loads `family_calendar` via admin) —
a child can now tamper with inputs to the very calculation the phase just locked down
(e.g. mark all 7 days as weekend to manipulate study-streak transparency, or shift year
bounds so every day resolves to vacation). The direct coin impact is bounded (a
transparent today currently zeroes `current` — see CR-02 — so no bonus minting), but
after CR-02/CR-03 are fixed, an all-weekend calendar would freeze a qualifying study
streak indefinitely and let the child collect `streak_study_bonus` daily. Fix the RLS
now so the streak fix doesn't open a coin-minting vector.
**Fix:** Restrict write policies to parents, mirroring the wallet-settings pattern:
```sql
WITH CHECK (
  family_id IN (
    SELECT family_id FROM public.family_members
    WHERE user_id = auth.uid() AND role IN ('parent', 'extended')
  )
);
```
(Same consideration applies to the pre-existing `vacation_periods` policies, which are
also now server-side streak inputs.)

### WR-02: `repairAchievements` server action has no auth guard but creates a service-role client

**File:** `app/actions/repair-achievements.ts:23-39`
**Issue:** The action calls `createAdminClient()` and runs `updateStreaks(admin, …)`
without `requireParent()` (or even `requireFamilyMember()`). Server actions are
POST-invocable by any authenticated session — a child can trigger admin-client streak
writes and badge/XP recomputation for every child in the family. Blast radius is limited
because the `children` list is read through the RLS-bound cookie client (scoped to the
caller's family, empty for anonymous), but this breaches the project convention that
every privileged (service-role) write path sits behind a guard from
`lib/supabase/admin.ts`, and the UI only exposes it in the parent Settings screen.
**Fix:**
```ts
import { requireParent } from '@/lib/supabase/admin'
export async function repairAchievements(): Promise<RepairResult[]> {
  await requireParent()
  ...
}
```

### WR-03: `child_filter` identity mismatch — per-child vacation periods never match server-side

**File:** `components/settings/PeriodsManager.tsx:67-77` vs `lib/day-type.ts:40` and `lib/services/streaks.service.ts:83-84`
**Issue:** PeriodsManager builds the child picker from `family_members`
(`select('id, display_name')`) and stores `family_members.id` into
`vacation_periods.child_filter`. But `getDayType` matches
`p.child_filter === childId` where every caller passes `children.id`:
`updateStreaks` uses the `children.id` validated by `assertChildInFamily`
(`lib/supabase/admin.ts:102-107` queries the `children` table), and `DailyModal` /
`app/kid/day/page.tsx` pass ids used to query `children`. `vacation-api.ts:16` documents
the field as `'all' | child_id`. Result: a vacation period targeted at one child
silently matches **no one** — study-streak transparency and day-type resolution for that
child never see the period. The picker predates this phase, but 05.5 made the match
load-bearing in server-side streak logic.
**Fix:** Select `child_id` from `family_members` (or read from `children` directly) and
use it as the option value:
```ts
.select('id, child_id, display_name') … ({ id: c.child_id, name: … })
```
Plus a one-off backfill for existing `child_filter` values that hold `family_members.id`.

### WR-04: `applyPreset('replace')` is non-atomic — failed insert leaves the preset's periods deleted

**File:** `lib/vacation-presets.ts:44-75`
**Issue:** Replace mode issues a client-side `delete` (family+preset scoped) and then a
separate `insert`. If the insert fails (network drop, RLS error, constraint), the
family's previously materialized periods are gone with nothing in their place, and
`PeriodsManager.handleApplyPreset` proceeds to overwrite `region_preset` only on
success — but the delete already happened. Day-type/streak transparency for those dates
silently changes until someone re-applies.
**Fix:** Insert first, delete only the *old* rows after a successful insert (delete by
`id NOT IN (newIds)` within the preset scope), or move apply into a single server-side
RPC/transaction.

### WR-05: UI promises coins the server never credits, using hardcoded values

**File:** `components/DailyModal.tsx:560-583,565,571-574`; `lib/day-type.ts:77-101`; `components/settings/PeriodsManager.tsx:186-195`; `components/kid/KidDayFillForm.tsx:851,26,272,311`
**Issue:** `/api/wallet/award` credits exactly seven source types (room, behavior,
grade, sport, activity, book, streak). Yet the UIs promise more:
- `DailyModal` footer counts reading-threshold coins (`calcReadingCoins` +3/+5), home
  help (+3), and weekend homework (+5) into the "+N💰" save button and chips — none of
  these have a server-side award path, so the wallet never changes.
- `readingCoinHint` and PeriodsManager's rules banner ("≥15 min → +3💰", "Each extra
  lesson → +3💰", "Home help → +3💰") advertise the same never-credited coins.
- Hardcoded values contradict per-family `wallet_settings`, violating CLAUDE.md ("do not
  hardcode coin values in components"): `DailyModal` room=3 / behavior=5 /
  `GRADE_COINS`; `KidDayFillForm:851` coach-rating hints (+25/+15/+5/-3/-10) that ignore
  `coins_per_coach_*`; `KidDayFillForm` book fallback 20 and room fallback 3.
**Fix:** Either add server-side award intents for reading-thresholds / home-help /
homework (keyed to their rows, like `book`), or remove those coins from the previews,
hints, and the PeriodsManager banner. Drive all remaining previews from the loaded
`wallet_settings` (DailyModal already loads nothing — fetch `getWalletSettings()` like
KidDayFillForm does, and replace the KidDayFillForm coach hint with settings values).

### WR-06: Streak-bonus "today" gate compares client-local date against server-TZ date

**File:** `app/api/wallet/award/route.ts:242`
**Issue:** `date === localDateString()` runs on the server, where `localDateString()`
yields the **server's** local date (UTC on Vercel). Clients compute `today` in their own
timezone (`app/kid/day/page.tsx:65`). For a UTC+3 family, saves between 00:00 and 03:00
local send a `date` one day ahead of the server's date → the streak bonus is silently
skipped for that day (and a knowingly crafted POST for "yesterday" would pass the gate
instead). CLAUDE.md explicitly flags this 00:00–03:00 UTC+3 window as the known hazard
for date handling.
**Fix:** Accept a small window instead of strict equality, e.g. allow
`date === localDateString() || date === localDateString(addDays(new Date(), 1))`-style
server±1-day tolerance, or resolve "today" per-family (stored timezone). The
`(child_id, 'streak', date)` idempotency key already prevents double-mint within the
widened window.

### WR-07: `FamilyCalendar` type lies — PeriodsManager can create rows with NULL `year_start`/`year_end`/`term_mode`

**File:** `components/settings/PeriodsManager.tsx:154`; `lib/repositories/calendar.repo.ts:32-46`; `lib/models/calendar.types.ts:8-11`; `lib/day-type.ts:55`
**Issue:** `handleApplyPreset` upserts `{ region_preset }` only; when no
`family_calendar` row exists yet, this inserts a row with `year_start`, `year_end`,
`term_mode` all NULL (the migration allows it). The `FamilyCalendar` type declares them
non-nullable strings, so `getDayType`'s `date < familyCalendar.year_start` silently
becomes a `NaN` comparison (always false) — the out-of-year-bounds→vacation rule (D-07)
is skipped without any signal, and TypeScript cannot catch downstream misuse.
**Fix:** Type the fields as `string | null` and guard in `getDayType`
(`familyCalendar.year_start && familyCalendar.year_end && (date < … || date > …)`), or
make `upsertFamilyCalendar` refuse to create a row without year bounds (insert defaults).

### WR-08: `new Date(date).getDay()` UTC-parse in DailyModal — wrong schedule weekday in UTC- timezones

**File:** `components/DailyModal.tsx:194-196`
**Issue:** `new Date('YYYY-MM-DD')` parses as UTC midnight; `.getDay()` then reads it in
the local timezone, shifting the weekday back one day for any UTC-negative user. The
file itself uses the correct `new Date(date + 'T12:00:00')` idiom three lines apart
(`:587`, `:166-167` in PeriodsManager) and `lib/day-type.ts:60` documents why. Here it
selects which lesson schedule to show — wrong day's subjects for Americas users of this
multi-family product.
**Fix:**
```ts
const d = new Date(date + 'T12:00:00')
```

### IN-08: PeriodsManager legacy fallback uses a member id as `family_id`

an explicit error state.
