# Phase 5.5: year-calendar - Context

**Gathered:** 2026-07-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Day types resolve entirely from a configurable calendar: school-year start/end and
quarters/trimesters mode, regional vacation presets (bundled JSON, manually editable —
manual wins), custom periods, per-family weekend days, and a sick day that pauses
school blocks without burning streaks. `getDayType` becomes fully data-driven; the kid
day screen reflects vacation/weekend automatically. The phase also carries the full fix
for defect CR-01 (streak-bonus minting via arbitrary client dates), since it rewrites
the streak calculators for day-type awareness anyway.

</domain>

<decisions>
## Implementation Decisions

### Regional vacation presets
- **D-01:** Bundled JSON directory covers **Russia (традиционные четверти + Москва-триместры/модули 5+1), Kazakhstan, and Belarus** in v1. Fully manual entry must remain a first-class equal path (PeriodsManager already exists) — presets are a convenience, not a requirement.
- **D-02:** Applying a preset **materializes rows into the existing `vacation_periods` table**. No virtual overlay/merge layer. "Manual wins" falls out naturally: the parent edits/deletes materialized rows like their own. Rows created by a preset carry a source marker (e.g. `preset_id`/`source` column) so re-apply can distinguish them from hand-made periods.
- **D-03:** Date horizon: **current + next school year** (2026–27 and 2027–28). The directory is updated via app releases; no runtime fetching.
- **D-04:** Re-applying a preset when periods already exist asks the parent: **"Replace preset periods" (deletes only preset-marked rows, never touches manual ones) or "Add missing only"**.

### Calendar data model
- **D-05:** New table **`family_calendar`** — one row per family: school-year start/end, term mode (quarters | trimesters), weekend days (int[] of weekday numbers, default sat+sun), selected region preset. Standard family-isolation RLS with client writes allowed (this is NOT a money table — same trust level as `vacation_periods`).
- **D-06:** Quarters/trimesters in v1 = **mode + school-year boundaries only**. The mode drives which vacation preset variant applies and is stored for future use (per-quarter analytics, year screen in 5.8). No per-quarter date rows in v1.
- **D-07:** Days **outside [year_start, year_end] auto-resolve to `vacation`** with no explicit period row (summer needs no preset row). Semantics per user: "летом нет уроков, но есть дополнительные задания — чтение и т.д." — existing `isNonSchoolDay` behavior (reading / home help / extra lessons blocks) applies.
- **D-08 (backward compat, implied by D-05/D-07):** A family with no `family_calendar` row keeps current behavior (hardcoded sat+sun weekend, no school-year concept). Planner: treat absent row as legacy defaults.

### Sick day & streaks
- **D-09:** Sick day = **transparent skip** for streak counters: it neither increments nor resets any streak (streak is "frozen"). Not counted as success.
- **D-10:** Transparency extends to school streaks: **study streak counts only school days** — weekends, vacations, and sick days are all transparent for it (fixes the latent bug where the study streak burns every weekend because the walker requires grades on every calendar day). Room/sport/behavior streaks keep counting all days, with sick transparent.
- **D-11:** Sick marking stays as-is: **both parent and child** can set `is_sick` in the day form. No new approval mechanism.

### CR-01 defect — full fix in this phase (early plan)
- **D-12:** Include the **full** CR-01 fix as an early plan of 5.5:
  1. Server-side date validation in `/api/wallet/award` — streak bonus payable only for a server-validated real calendar date (today per `localDateString()` semantics), not any client-supplied string;
  2. Move `updateStreaks` server-side (it currently runs in the browser and writes `streaks` via the anon client);
  3. Lock the `streaks` table to SELECT-only for clients (same RLS pattern as money tables, migration mirroring `04.4-03`).
  Rationale: this phase rewrites the streak calculators for day-type awareness anyway; fixing the trust boundary at the same time avoids touching the same code twice.

### Claude's Discretion
- UI placement of calendar settings inside Parent Center → Settings (follow the 5.4 Coins Rules / Room settings card patterns).
- Exact JSON schema of the bundled preset directory (region → term-mode variants → periods).
- Weekend-day picker UX; how `getDayType`'s new signature loads `family_calendar` (the kid day page already loads vacation periods — extend that data load).
- Whether the preset apply/re-apply runs client-side (non-money table) or via a small route — planner's call, consistent with D-05.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `vacation_periods` table + `lib/vacation-api.ts` (CRUD) + `components/settings/PeriodsManager.tsx` — presets materialize into this existing structure (D-02); table needs a preset-source marker column.
- `lib/day-type.ts` — `getDayType(date, isSick, vacationPeriods, childId)` is the single resolver to make data-driven; `isNonSchoolDay` already routes vacation/weekend to reading/help/lessons blocks.
- `days.is_sick` (set in `KidDayFillForm` / `DailyModal`) — the sick flag already exists end-to-end.
- `lib/services/streaks.service.ts` — all four streak calculators walk 30 calendar days backward; this is where day-type transparency (D-09/D-10) lands, and the module that moves server-side for CR-01 (D-12).
- `app/api/wallet/award` + `app/api/wallet/_lib.ts` — award route to harden (D-12); settings PATCH route pattern from 5.4 for calendar settings if a route is chosen.

### Established Patterns
- Family settings tables use family-isolation RLS (`rls.sql`); money tables are SELECT-only for clients (`04.4-03`) — `streaks` moves to the second category, `family_calendar` stays in the first.
- `localDateString()` for "today" (never `toISOString().split('T')[0]`) — critical for the CR-01 date validation.
- Parent Center → Settings cards (5.2 Room, 5.4 Streaks & Bonuses) — the calendar settings UI follows this pattern.

### Integration Points
- `app/kid/day/page.tsx:96` and `components/DailyModal.tsx:167` — the two `getDayType` call sites; both must load `family_calendar` alongside vacation periods.
- `/api/wallet/award` streak-bonus section — reads `streaks.current_count`; after D-12 it reads server-computed streaks only.
- Phase 5.6 (day-blocks) depends on this phase's day-type resolution; keep `getDayType`'s output shape stable/extensible.

</code_context>
